# Learning x Safety 交叉设计

> Evidence Status: grounded
> 知识库映射: Reflection&Learning (Plane — Learning&Adaptation) x Trust&Identity (Plane — Security) x Action&Effect (Plane — Effects)

## 为什么需要这篇文档

技能结晶化（Skill Crystallization）是 Agent 从运行经验中提炼可复用能力的核心机制。但 2026 年初的 ClawHavoc 事件证明：**技能不仅是效率工具，也是攻击向量。** 约 1,200 个恶意技能渗入主流 Agent 市场，窃取凭据并操纵 Agent 行为。

更隐蔽的威胁来自记忆投毒。NeurIPS 2025 MINJA 研究在理想条件下达到 95% 注入成功率，但也发现**已有合法记忆提供天然稀释防御**。SkillsBench SoK（arXiv 2602.20867）揭示自生成技能比外部验证技能平均低 1.3pp，且技能库超过临界规模后选择准确率急剧下降。本文档建立技能结晶化过程中的安全门控、投毒防御和生命周期管理框架。

---

## 交叉点识别

| 交叉点 | Learning 侧关注 | Safety 侧关注 | 冲突/张力 | 设计要求 |
|--------|---------------|--------------|----------|---------|
| 技能生成 | 从成功 trace 中提炼技能 | 技能可能编码不安全行为 | 学习速度 vs 安全审查 | 安全门控流水线 |
| 技能激活 | 快速召回提升效率 | 恶意技能被激活即产生危害 | 激活延迟 vs 安全保障 | 多阶段激活协议 |
| 技能来源 | 外部导入扩充能力 | 供应链投毒风险 | 生态丰富度 vs 信任边界 | 来源验证 + 沙箱 |
| 技能退化 | 旧技能可能不再适用 | 过时技能可能违反更新的安全策略 | 向后兼容 vs 安全演进 | 定期重验证 |
| 技能选择 | 匹配最佳技能 | 恶意技能可能伪装成最佳匹配 | 匹配精度 vs 安全过滤 | 信任感知选择 |
| 技能规模 | 更多技能 = 更广能力 | 规模增长 → 选择退化 + 审计负担 | 规模增长 vs 治理成本 | 技能库规模管理 |

## 攻击面矩阵

### 技能生命周期各阶段的安全威胁

| 阶段 | 威胁 | 攻击方法 | 案例/证据 | 严重度 |
|------|------|---------|----------|--------|
| **外部导入** | 供应链投毒 | 恶意技能伪装为合法工具上传到公共市场 | ClawHavoc: ~1,200 个恶意技能，影响 40,000+ 实例 | Critical |
| **外部导入** | 依赖劫持 | 技能依赖的外部资源被篡改 | 类 npm/PyPI 供应链攻击模式 | High |
| **自生成** | 行为编码 | 成功 trace 中包含越权但恰好成功的路径 | Learning Plane — "越权但成功的路径"应标记为 anti-pattern | High |
| **自生成** | 记忆投毒传导 | 投毒记忆被提炼为技能，固化恶意行为 | MINJA: 95% 注入成功率（理想条件） | High |
| **激活** | 语义伪装 | 恶意技能的描述精心设计以匹配高频查询 | ClawHavoc: 命名模仿热门技能 | Medium |
| **执行** | 运行时注入 | 技能在执行时访问超出声明范围的资源 | ClawHavoc: 技能继承 Agent 全部权限 | Critical |
| **积累** | 选择退化 | 技能库过大导致匹配精度下降 | SkillsBench: 超过临界规模后性能急剧退化 | Medium |

## 技能安全生命周期

### 六阶段门控模型

技能从产生到退役，必须经过六个安全阶段。每个阶段有明确的准入条件和安全检查。

```text
candidate → quarantine → shadow → activated → monitored → retired

candidate:    新生成或新导入的技能，尚未通过任何验证
quarantine:   正在接受安全扫描和功能验证
shadow:       在生产流量旁运行，但输出不被采纳（影子模式）
activated:    通过验证，可被正式召回使用
monitored:    持续监控行为和安全指标
retired:      因过时、安全问题或冗余被停用
```

### 各阶段的安全检查

| 阶段转换 | 安全检查 | 通过条件 | 失败处理 |
|---------|---------|---------|---------|
| → candidate | 格式合规 + 来源标注 | 有明确来源和 provenance | 拒绝入库 |
| candidate → quarantine | 静态分析 + 权限声明审查 | 无已知恶意模式；权限声明合理 | 标记可疑 + 人工审查 |
| quarantine → shadow | 沙箱执行 + 行为监控 | 无异常网络/文件/凭据访问 | 隔离 + 告警 |
| shadow → activated | 影子模式对比 + 质量验证 | 输出质量 >= 基线；无安全偏差 | 回退到 quarantine |
| activated → monitored | 持续行为审计 | 持续满足安全策略 | 降级或 retire |
| monitored → retired | 定期重验证 | 不再满足当前安全策略或已冗余 | 软删除 + 保留审计记录 |

## 自生成技能的质量风险

### SkillsBench 发现

SkillsBench SoK 对 Agent 技能系统的系统性评估揭示了自生成技能的固有风险：

| 发现 | 数据 | 设计影响 |
|------|------|---------|
| 自生成技能性能偏低 | 比外部验证技能平均低 1.3pp | 自生成技能必须经过外部验证才能激活 |
| 技能库规模的相变点 | 超过临界规模后选择准确率急剧下降 | 必须控制技能库规模 + 定期清理冗余 |
| 技能选择是瓶颈 | 技能再多，选错了等于没有 | 技能索引和匹配质量比数量更重要 |

**退化根因**：技能库增长 → 语义重叠增加 → 选择器区分度下降 → 错误匹配上升 → 反而不如无技能基线。**缓解**：规模上限 + 语义去重 + 使用频率衰减淘汰 + 自生成技能强制外部 eval 验证。

## 记忆投毒与技能投毒的关系

MINJA 的发现在技能层面有直接映射：合法记忆稀释投毒记忆 → 经过验证的技能稀释恶意技能；投毒记忆通过语义匹配被召回 → 恶意技能通过描述匹配被选择。防御含义：维持高质量技能密度本身就是防御手段；技能选择必须加入信任权重；provenance 不可变。

### 投毒传导链

```text
恶意外部输入 → 被写入记忆 → 记忆被提炼为技能
                                ↓
                    投毒技能被激活并反复使用
                                ↓
                    恶意行为被固化为"标准操作"
```

**阻断点**：
1. 记忆写入阶段的安全扫描（见 memory-x-security.md）
2. 技能提炼阶段的行为验证（本文档 quarantine 阶段）
3. 技能激活阶段的影子模式对比（本文档 shadow 阶段）

## 防御架构

### 信任感知的技能选择

技能选择器不应仅基于语义匹配度排序，必须联合考虑信任因子：`skill_score = semantic_match(query, skill) x trust_weight(skill)`。trust_weight 由来源可信度（verified_publisher > self_generated > unknown）、最近验证时间、历史安全事件数和当前策略合规性决定。

### 来源追踪

每个技能必须携带不可变的 provenance 字段：`origin`（self_generated / external_marketplace / curated_library / human_authored）、`publisher_trust_level`、`code_signature`、`validation_policy_version`、`security_incidents`。自生成技能还需记录 `lineage`（parent_memory_ids + extraction_method），支持从记忆到技能的推导路径追溯。

### 定期重验证协议

技能不是"一次验证，永久有效"。安全策略会演进，技能必须定期重新验证。

| 触发条件 | 重验证范围 | 失败处理 |
|---------|----------|---------|
| 安全策略版本更新 | 全部 activated 技能 | 不通过的降级为 quarantine |
| 定期审计（建议 30 天） | 采样 20% activated 技能 | 异常技能全量扫描 |
| 安全事件关联 | 涉事来源/发布者的全部技能 | 涉事技能立即隔离 |
| 长期未使用（>90 天） | 单个技能 | 自动退役 |

## 常见错误与案例

### 错误 1: 信任自生成技能等同于信任外部验证技能

**表现**: Agent 从自身成功 trace 中提炼的技能直接进入 activated 状态
**数据**: SkillsBench 显示自生成技能性能平均低 1.3pp
**修正**: 自生成技能必须经过独立 eval 验证，不能仅凭"来源是自己"就跳过验证

### 错误 2: 技能库只增不减

**表现**: 持续积累技能，从不淘汰
**数据**: SkillsBench 发现超过临界规模后技能选择准确率急剧下降
**修正**: 设定技能库容量上限 + 冗余检测 + 使用频率衰减淘汰

### 错误 3: 外部技能导入不做沙箱验证

**表现**: 从公共市场导入的技能直接进入 Agent 运行时
**案例**: ClawHavoc——~1,200 个恶意技能通过松散的审核机制渗入，窃取凭据并分发恶意软件
**修正**: 外部技能必须经过 quarantine → shadow → activated 全流程；沙箱执行验证行为

### 错误 4: 忽视记忆到技能的投毒传导

**表现**: 记忆层做了安全扫描，但从记忆提炼技能时不再检查
**案例**: MINJA 的投毒记忆通过技能提炼被固化为持久化恶意行为
**修正**: 技能提炼是安全关键操作——提炼后的技能必须独立通过安全扫描；安全策略版本变更时触发全量重验证

## 设计启发

1. **技能是高权限的可复用代码/策略——安全标准应高于普通记忆。** 记忆影响单次会话，技能影响所有匹配场景。技能投毒的影响范围远大于记忆投毒。
2. **自生成技能需要外部验证。** 1.3pp 的性能差距说明"成功一次"不等于"可靠复用"。独立 eval 是最低要求。
3. **技能库规模不是越大越好。** 相变点之后，更多技能 = 更低性能。积极淘汰冗余技能比不断积累更重要。
4. **合法技能密度是天然防御。** MINJA 的发现表明，维持高质量、经过验证的技能密度，本身就降低了恶意技能被选中的概率。
5. **来源追踪是事后调查的基础。** 当恶意技能被发现时，provenance 链是定位注入点、评估影响范围、批量隔离相关技能的唯一手段。
6. **安全策略演进要求技能重验证。** 技能不是"一劳永逸"的资产，必须随安全策略版本持续合规。
7. **记忆 → 技能是攻击的升级路径。** 投毒记忆被提炼为技能后，从"单次影响"升级为"持久化影响"。这条路径上的每个转换点都需要安全检查。
8. **六阶段门控的成本可以通过风险分级降低。** 低风险自生成技能可以快速通过 quarantine；高风险外部技能必须完整走完全流程。
