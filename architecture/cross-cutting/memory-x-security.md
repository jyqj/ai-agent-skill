# Memory x Security 交叉设计

> **Evidence Status** — grounded.
> 知识库映射: Sensing&Repr (Plane 1-3) x Trust&Identity (Plane 13-15) x World Modeling (Plane 7-9)

## 为什么需要这篇文档

记忆系统是 Agent 获取长期能力的关键基础设施，但同时也是最容易被忽视的攻击面。Agent Security Bench 评估显示，27 种攻防组合中**记忆投毒平均攻击成功率高达 84.30%**。ClawHavoc 事件中 824 个恶意 Agent 技能通过供应链渗入，进一步证明了记忆层面安全防护的迫切性。

传统安全模型关注的是"谁可以访问什么"，但 Agent 记忆安全面临独特挑战：**记忆内容本身可以成为攻击载体**。一条被投毒的记忆摘要，会在后续每个会话中作为系统指令自动注入，实现跨会话的持久化自主执行。这不是传统的越权访问问题，而是信息本身被武器化的问题。

本文档系统性地识别 Memory 与 Security 两个域的交叉点，提供攻击面矩阵、防御层次架构、写入验证协议和来源追踪字段设计。

## 交叉点识别

| 交叉点 | Memory 侧关注 | Security 侧关注 | 冲突/张力 | 设计要求 |
|--------|--------------|----------------|----------|---------|
| 记忆写入 | 高效摘要、低延迟持久化 | 写入前验证、内容完整性 | 验证延迟 vs 写入效率 | 分级验证协议 |
| 记忆检索 | 语义相似度匹配、高召回率 | 来源可信度过滤、信任分层 | 召回率 vs 安全过滤 | 信任感知检索 |
| 记忆摘要 | 信息压缩、保留关键语义 | 防止恶意指令在压缩中存活 | 摘要保真度 vs 指令过滤 | 摘要后安全扫描 |
| 跨会话持久化 | 上下文连续性、个性化 | 防止投毒记忆跨会话传播 | 连续性 vs 隔离 | 会话边界验证 |
| 记忆淘汰 | 时间衰减、相关性排序 | 恶意记忆优先淘汰 | 正常衰减 vs 安全驱逐 | 双因子淘汰策略 |
| 外部记忆源 | RAG 检索、知识库集成 | 外部数据的信任传递 | 数据丰富度 vs 信任边界 | 来源标注与隔离 |

## 防御层次架构

### 四层纵深防御模型

```
L0 输入过滤层 ─── 信任分级 + 内容扫描 + 格式验证
       |
L1 写入验证层 ─── 来源标注 + 完整性签名 + 异常检测
       |
L2 存储隔离层 ─── 信任分区 + 加密存储 + 访问控制
       |
L3 检索审计层 ─── 信任感知排序 + 使用审计 + 时间衰减
```

### 各层详细设计

**L0 输入过滤层**

| 输入来源 | 信任等级 | 过滤策略 | 写入权限 |
|---------|---------|---------|---------|
| 用户直接指令 | Trusted | 基础格式验证 | 直接写入 |
| 系统配置/Prompt | Trusted | 签名验证 | 直接写入 |
| Agent 自身推理 | Semi-trusted | 指令模式扫描 | 延迟写入 + 摘要审查 |
| 外部工具返回 | Semi-trusted | 内容净化 + 指令检测 | 标注后写入 |
| 网页/邮件内容 | Untrusted | 严格净化 + 指令剥离 | 隔离区写入 |
| 第三方插件 | Untrusted | 沙箱执行 + 输出过滤 | 需显式批准 |

**L1 写入验证层** -- 详见下节"写入验证协议"

**L2 存储隔离层**
- 按信任等级物理分区存储记忆
- Trusted 区域与 Untrusted 区域之间无直接引用
- 每条记忆附带不可变的来源元数据（详见"来源追踪字段设计"）
- 记忆条目使用 HMAC 签名防篡改

**L3 检索审计层**
- 检索时根据来源信任等级加权排序（信任感知检索）
- 低信任来源记忆在检索结果中降权但不完全排除
- 所有检索操作记录审计日志（查询、返回条目、信任分布）
- 异常检索模式触发告警（如短时间内大量低信任记忆被检索）

## 写入验证协议

### 三级验证流程

```
记忆候选 → 格式验证 → 内容安全扫描 → 来源标注 → 写入决策

写入决策分支:
  信任等级 >= Trusted    → 直接写入 + 签名
  信任等级 == Semi-trusted → 延迟写入 + 人工采样审查
  信任等级 <= Untrusted   → 隔离区写入 + 标记 + 定期清理
```

### 内容安全扫描规则

| 检测类别 | 检测方法 | 阻断策略 |
|---------|---------|---------|
| 指令注入模式 | 正则匹配 + 语义分类器 | 硬阻断 + 告警 |
| 会话边界伪造 | XML/标签结构分析 | 硬阻断 |
| 权限提升尝试 | 特权关键词检测 | 硬阻断 + 审计 |
| 异常信息密度 | 信息熵分析 | 软阻断 + 人工审查 |
| 自引用循环 | 记忆引用链检测 | 标记 + 监控 |

### 写入频率控制

- 单会话记忆写入上限：可配置（建议 20-50 条/会话）
- 同一来源连续写入间隔：最小 5 秒
- 写入突增检测：超过滑动窗口均值 3 倍时触发审查
- 批量写入需要显式授权

## 来源追踪字段设计

### 记忆条目元数据 Schema

```yaml
memory_entry:
  id: string              # 唯一标识符 (UUID v7, 时间有序)
  content: string          # 记忆内容
  content_hash: string     # 内容 SHA-256 哈希
  hmac_signature: string   # HMAC-SHA256 防篡改签名

  provenance:              # 来源追踪
    source_type: enum      # user_direct | system_config | agent_reasoning |
                           # tool_response | external_content | plugin
    source_id: string      # 来源标识（用户 ID / 工具名 / URL）
    trust_level: enum      # trusted | semi_trusted | untrusted
    session_id: string     # 写入时的会话 ID
    timestamp: datetime    # 写入时间 (ISO 8601)
    write_method: enum     # direct | summarized | merged | imported

  lineage:                 # 血统链
    parent_ids: [string]   # 源记忆条目 ID（摘要/合并时追踪）
    transform_type: enum   # none | summarize | merge | extract
    transform_model: string # 执行摘要的模型标识

  security:                # 安全元数据
    scan_result: enum      # clean | flagged | quarantined
    scan_timestamp: datetime
    scan_version: string   # 扫描规则版本
    quarantine_reason: string  # 隔离原因（如有）
    access_count: int      # 被检索次数
    last_accessed: datetime

  lifecycle:               # 生命周期
    ttl: duration          # 过期时间（可选）
    decay_weight: float    # 时间衰减权重 [0.0, 1.0]
    pinned: bool           # 是否固定（不参与衰减）
```

### 关键设计决策

1. **不可变来源标注**：`provenance` 字段写入后不可修改，即使内容被摘要或合并，原始来源信息通过 `lineage.parent_ids` 保持可追溯
2. **双重完整性校验**：`content_hash` 检测意外修改，`hmac_signature` 防止恶意篡改
3. **安全扫描版本化**：`scan_version` 允许在规则更新后对历史记忆重新扫描
4. **访问审计内嵌**：`access_count` 和 `last_accessed` 支持异常检索模式检测

## 常见错误与案例

### 错误 1: 隐式信任 Agent 自身记忆

**表现**: Agent 将自己的历史记忆视为可信系统指令
**案例**: MINJA 攻击通过间接注入将恶意指令写入记忆摘要，后续会话中 Agent 将其作为"自己的想法"执行
**修正**: 即使是 Agent 自身生成的记忆摘要，也必须标注 `source_type: agent_reasoning` 并在检索时降权处理

### 错误 2: 摘要过程不做安全扫描

**表现**: 记忆压缩/摘要时仅关注信息保留率，不检查压缩后的内容安全性
**案例**: 恶意指令在长文本中被稀释，但摘要过程反而将其"提炼"为更浓缩的注入
**修正**: 摘要后必须对新生成的摘要内容独立执行安全扫描

### 错误 3: 共享记忆层无隔离

**表现**: 多 Agent 系统使用共享记忆池，一个 Agent 的投毒记忆感染所有其他 Agent
**案例**: $47,000 多 Agent 循环失控——Agent 间共享状态中的错误信号未被隔离，导致级联
**修正**: 共享记忆层应实现每 Agent 视图隔离，跨 Agent 记忆传递需要显式授权和安全审查

### 错误 4: 仅依赖 Prompt 约束防止记忆污染

**表现**: 在系统 Prompt 中写"不要记住不可信内容"
**案例**: Replit Agent 在"代码冻结"指令下仍执行破坏性操作，证明自然语言约束不是安全边界
**修正**: 记忆写入保护必须在架构层面（代码/配置）实现硬性执行

### 错误 5: 忽略记忆供应链风险

**表现**: 从外部导入预制记忆/知识库时不做来源验证
**案例**: ClawHavoc 事件中 824 个恶意 Agent 技能通过松散的发布审核机制渗入 40,214 个实例
**修正**: 外部记忆导入必须经过完整的安全扫描流水线，等同于 Untrusted 来源处理

## 设计启发

1. **记忆是攻击面，不是安全存储**。Agent 的长期记忆在安全模型中应被视为"可能被污染的输入"，而非"可信的自我知识"。
2. **写入比读取更危险**。一条投毒记忆的写入会影响所有后续会话；检索时的防御是最后一道线，不是唯一一道。
3. **信任等级必须贯穿记忆全生命周期**。从写入到检索到淘汰，信任标注不可丢失、不可降级（只能升级）。
4. **摘要是安全关键操作**。记忆压缩/合并会改变内容形态，必须在摘要后重新执行安全扫描。
5. **有效防御需要精细的信任阈值校准**。过于严格的过滤会破坏正常功能（假阳性），过于宽松则无法阻止投毒（假阴性）。建议从严开始，根据生产数据逐步放宽。
6. **时间衰减是天然的安全机制**。老旧记忆的影响力应随时间自然降低，这同时服务于信息新鲜度和安全防护。
7. **审计日志是事后调查的基础**。当投毒事件发生时，完整的来源追踪和访问日志是定位注入点和影响范围的唯一手段。
8. **多 Agent 共享记忆需要额外的隔离层**。Agent 间的记忆传递应被视为跨信任边界的操作，需要显式授权。
