# Paradigms x Economics 交叉设计

> **Evidence Status** — grounded.
> 知识库映射: Cognition (Plane 4-6) x Lifecycle&Economics (Plane 17-18) x Coordination (Plane 16)

## 为什么需要这篇文档

范式选择（ReAct / Plan-and-Execute / ReWOO / Hybrid）是 Agent 架构的核心设计决策，但在大多数设计讨论中，范式选择主要基于"能力匹配"，即哪种范式最适合任务类型。成本维度常被推迟到部署后才考虑。

然而，生产数据表明：
- RouteLLM 通过模型路由实现 **-85% 成本**，质量损失可控
- TALE 框架通过 Token Budget 管理实现 **-67% token 消耗**
- Budget-Aware 工具使用实现 **-40% 搜索调用 + 2x 性能提升**
- Agent 工作负载比标准聊天贵 **3-10x**（多轮循环 50x token、输出溢价 3-8x）

这意味着范式选择直接决定了运营成本的数量级。本文档建立范式与成本的交叉映射，提供成本驱动的范式选择框架和预算耗尽时的降级策略。

## 交叉点识别

| 交叉点 | Paradigms 侧关注 | Economics 侧关注 | 冲突/张力 | 设计要求 |
|--------|----------------|----------------|----------|---------|
| 推理深度 | 更深推理 = 更高准确率 | 推理 token 成本高（输出/输入比 4:1~8:1） | 准确率 vs 成本 | Budget-Aware 推理深度控制 |
| 工具调用频率 | ReAct 每步调用 LLM + 工具 | 每轮调用产生 schema/call/result token | 交互性 vs token 效率 | 批量工具调用 / ReWOO 模式 |
| 规划开销 | Plan-and-Execute 需前置规划成本 | 规划使用推理模型（高单价） | 规划质量 vs 规划成本 | 规划摊销 / 规划缓存 |
| 迭代次数 | 复杂任务需要多轮迭代 | 10 轮 ReAct 循环 = 50x 单次 token | 任务完成度 vs 预算消耗 | 硬性迭代上限 + 进度评估 |
| 模型选择 | 不同范式对模型能力要求不同 | 前沿模型 vs 小模型成本差 190x | 模型能力 vs 成本 | 层级化模型路由 |
| 上下文窗口 | 长会话需要更大上下文 | 128K vs 8K 窗口成本差 64x (n^2) | 上下文完整性 vs 成本 | Compaction + Sub-agent |

## 范式 x 成本矩阵

### 各范式的成本特征

| 范式 | 单任务 Token 消耗 | 工具调用模式 | 模型要求 | 可预测性 | 适用成本区间 |
|------|-----------------|------------|---------|---------|-------------|
| **ReAct** | 高（每步重发上下文） | 高频、逐步 | 中-高（需推理+工具能力） | 低（步数不确定） | $0.5-5/任务 |
| **Plan-and-Execute** | 中（规划一次+轻量执行） | 中频、按计划 | 规划用高端+执行用低端 | 中（计划框定范围） | $0.3-3/任务 |
| **ReWOO** | 低（一次性规划全部调用） | 批量、预规划 | 中（需强规划能力） | 高（调用数确定） | $0.1-1/任务 |
| **Hybrid** | 可变（分层动态切换） | 自适应 | 多模型协同 | 中-高（有框架） | $0.2-4/任务 |
| **Reflective** | 极高（双倍推理+验证） | 高频+自检 | 高（需自我评估能力） | 低（反思轮数不确定） | $1-10/任务 |

### 成本放大因子详解

| 因子 | 放大倍数 | 影响最大的范式 | 缓解策略 |
|------|---------|-------------|---------|
| 多轮上下文累积 | 50x (10 轮) | ReAct, Reflective | Prompt Caching（-90%）、Compaction |
| 输出 token 溢价 | 3-8x | Reflective（大量推理输出） | 控制推理 token 预算（TALE/BudgetThinker） |
| 工具 Schema 注入 | 每轮 +500-2000 token | ReAct（高频工具调用） | 动态工具发现（节省 30-50% context） |
| 规划 token | 一次性 2000-5000 token | Plan-and-Execute | 规划缓存、相似任务复用 |
| 并行扩展 | Nx（N 个并行实例） | Budget-Constrained Parallel | 多数投票减少冗余 |

## 成本驱动的范式降级策略

### Budget 分级与范式映射

```
Budget 充裕（>$5/任务）
  → Hybrid: Reflective Planner + ReAct Executor
  → 使用推理模型（Opus/o3-pro）做规划
  → 完整的自我验证循环

Budget 标准（$1-5/任务）
  → Plan-and-Execute: 推理模型规划 + 轻量模型执行
  → Planner 用 Sonnet/GPT-4o，Executor 用 Haiku/Flash
  → 分离可减少推理成本达 45%

Budget 紧缩（$0.3-1/任务）
  → ReWOO: 一次性规划全部工具调用
  → 使用中档模型 + Prompt Caching
  → 避免多轮上下文累积

Budget 极低（<$0.3/任务）
  → 单步直接回答（无 agentic 循环）
  → 使用最小模型 + 语义缓存
  → 仅处理 cache hit 或极简任务
```

### 运行时 Budget 耗尽的降级行为

| 剩余 Budget | 触发行为 | 降级策略 |
|-------------|---------|---------|
| >50% | 正常执行 | 当前范式全速运行 |
| 30-50% | 黄色告警 | 切换到更低成本模型；减少推理深度（effort: low） |
| 10-30% | 橙色告警 | 范式降级（ReAct→ReWOO、Reflective→Plan-and-Execute）；停止自我验证循环 |
| 5-10% | 红色告警 | 仅允许关键工具调用；输出当前最佳结果 + 置信度标注 |
| <5% | 硬停止 | 立即输出中间结果 + 未完成标注；记录断点供人工接续 |
| 0% | 熔断 | 拒绝执行；返回结构化错误（包含已完成步骤摘要） |

### Budget 耗尽的反模式

**反模式 1: 标准 Agent 的 Budget 盲区**
- 数据: 标准 ReAct baseline 在 budget=100 工具调用时饱和，无法利用更多资源
- 问题: Agent 缺乏内在 budget 意识，"即使有额外资源也经常执行浅层搜索"
- 修正: Budget Tracker 显示剩余/已用资源 → 40.4% 更少搜索调用，31.3% 更低成本

**反模式 2: 过度紧缩反而增加消耗**
- 数据: TALE 发现过度紧缩的 budget 反而增加实际 token 消耗（"放弃合规"效应）
- 问题: 模型在极低预算下放弃遵循格式要求，产生更多无效输出
- 修正: 设定最低 budget 下限（建议标准 CoT 的 40% 作为底线）

**反模式 3: $47,000 循环失控**
- 数据: 4 个 Agent 进入无限 ping-pong 循环 264 小时，团队将成本上升归因于"用户增长"
- 问题: 无 per-agent token/成本上限；无循环检测；无实时成本监控
- 修正: 预算告警 != 预算执行；必须有硬性熔断器

## 设计决策矩阵

### 按任务场景选择范式 + 成本策略

| 场景 | 推荐范式 | 模型配置 | 成本策略 | 预期成本 |
|------|---------|---------|---------|---------|
| 时敏 QA / 故障排查 | ReAct | Sonnet/GPT-4o | Prompt Caching + 迭代上限 20 步 | $0.5-2 |
| 低风险探索性任务 | ReAct | Haiku/Flash | 语义缓存 + 轻量模型 | $0.1-0.5 |
| 多步监管工作流（金融/法律） | Plan-and-Execute | Opus 规划 + Haiku 执行 | 规划缓存 + 执行用轻量模型 | $1-3 |
| 大型复杂项目 | Hybrid | 三层模型路由 | 全栈优化（缓存+路由+压缩） | $2-5 |
| 高可靠性代码审查 | Reflective | Opus + Think Tool | 自检循环限制 2 轮 | $2-8 |
| 批量数据处理 | ReWOO | 中档模型 + Batch API | Batch API -50% + 预规划 | $0.05-0.3 |
| 实时对话助手 | 单步/浅 ReAct | Flash/mini | 语义缓存 + 最小模型 | $0.01-0.1 |

### 成本优化组合效果

| 优化策略 | 单独效果 | 组合生效 |
|---------|---------|---------|
| Prompt Caching | -90%（缓存 token） | 与 ReAct 搭配效果最好（高频重发上下文） |
| Model Routing | -87%（前沿 vs 小模型差 190x） | 与 Hybrid/Plan-and-Execute 搭配（分角色选模型） |
| 语义缓存 | -31% 查询消除 | 与所有范式搭配（通用层） |
| Prompt 压缩 | 最高 -20x 输入 token | 与长上下文范式搭配（ReAct/Reflective） |
| Batch API | -50% 折扣 | 仅与非实时范式搭配（ReWOO/批量处理） |
| Context Compaction | 2-3x 更长有效会话 | 与长会话范式搭配（Hybrid/Reflective） |

**组合效果**: 压缩 + 路由 + 缓存 = **60-80% 总成本削减**

## 常见错误与案例

### 错误 1: 在所有任务上使用同一范式

**表现**: 团队选择 ReAct 或 Plan-and-Execute 后，对所有任务统一使用
**数据**: 简单 QA 任务用 Reflective 范式，成本可达直接回答的 50-100x
**修正**: 建立任务复杂度评估器（Adaptive Reasoning Router），按复杂度路由到不同范式

### 错误 2: 过早升级到多 Agent

**表现**: "多数团队在单 Agent 达到质量天花板前就过度工程化为多 Agent 拓扑"
**数据**: 多 Agent 引入 2-5x 协调开销
**修正**: 遵循升级纪律：构建 ReAct 基线 → 测量 → 重复失败超过 ~5% 才升级

### 错误 3: 仅报告 Token 成本

**表现**: 仅计算 LLM token 费用，忽略工具费、基础设施、人工审核
**数据**: 仅报告 token 成本通常低估真实费用 40-70%
**修正**: 全载成本模型 = Token Cost + (Tool Calls x Provider Pricing) + 基础设施 + 人工监督

### 错误 4: $4,200 烧钱 63 小时

**表现**: Agent 进入重复工具调用循环，以相同参数反复调用同一函数
**根因**: 缺乏内部保护机制来识别"进度停滞"；传统监控提供事后分析而非实时干预
**修正**: 采用 "Agentic Brakes" 模式：预算硬上限 + 重复检测阈值 + 停滞步数检测

## 设计启发

1. **范式选择就是成本选择**。ReAct/Reflective 的高交互性意味着高 token 消耗；ReWOO/批量处理的低交互性意味着低成本但低灵活性。
2. **Planner-Executor 分离是最具杠杆的成本架构**。推理模型做规划 + 轻量模型做执行，可减少推理成本达 45%，且规划结果可缓存复用。
3. **Budget 意识必须是 Agent 的一等公民**。标准 Agent 在 budget=100 时饱和。显式 budget 信号可实现 40% 成本降低和 2x 性能提升。
4. **成本优化的最高杠杆是模型路由**。前沿模型 vs 小模型成本差 190x，单一路由策略可削减 87% 成本。
5. **过度紧缩比宽松更危险**。TALE 发现过度紧缩的 budget 反而增加消耗（"放弃合规"效应）。设定合理的最低 budget 下限。
6. **预算告警不等于预算执行**。$47,000 案例证明：团队会将异常成本归因于"业务增长"。必须有不依赖人类判断的硬性熔断器。
7. **步骤上限是最简单有效的成本控制**。默认 20-30 步；硬上限 50 步（超过 50 步 prefix-cache 失效成本飙升 5-10x）。
8. **组合优化比单一优化效果好 3-5x**。Prompt Caching + Model Routing + 语义缓存 = 60-80% 总成本削减。
