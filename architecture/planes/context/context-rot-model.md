# Context Rot 模型：上下文退化的机制、检测与缓解

> **所属域**：2. Cognition & Continuity — 上下文退化与注意力衰减
>
> **Evidence Status** — grounded. Chroma 2025 "Mastering LLMs: Context Window" 研究（18 个前沿模型实测）、arXiv 2601.15300 智能退化实验、企业级 Agent 失败统计。

**Principle Refs**: BR-02 — 上下文信息随时间退化。

---

## 1. 定义

**Context Rot**（上下文腐化）指 Agent 在输入 token 数增加的过程中，即使面对简单任务也出现性能退化的现象。其机制是注意力权重被稀释：信息仍在窗口中，但模型实际上无法有效利用。

与传统系统中的数据腐化不同，Context Rot 是 Transformer 注意力机制的固有特性：

```text
注意力矩阵复杂度 = O(n²)

当 n 翻倍时：
  - 每个 token 的平均注意力权重减半
  - 特定 token 被"注意到"的概率下降
  - 中间位置的信息尤其容易被忽略（Lost in the Middle）
```

---

## 2. Chroma 研究：18 个前沿模型全部退化

Chroma 在 2025 年的系统性研究中测试了 18 个前沿模型（包括 GPT-4o、Claude 3.5、Gemini 1.5 Pro 等），核心发现：

### 2.1 所有模型都退化

**没有例外**。无论模型声称支持多大的上下文窗口（128K、200K、1M），在实际信息提取任务中，性能随上下文长度单调递减。

### 2.2 中间位置最严重

```text
信息提取准确率（标准化）

  位置：  [前10%]  [10-30%]  [30-70%]  [70-90%]  [后10%]
  准确率：  0.92     0.78      0.61      0.75      0.89
                              ▲
                         最差区间
```

中间 40% 的位置区域（30%-70%）是注意力的"死区"，与 Liu et al. 的 Lost in the Middle 发现一致。

### 2.3 临界阈值：40-50% 窗口利用率

| 利用率区间 | F1 表现 | 退化程度 |
|---|---|---|
| 0-20% | 基线水平 | 无显著退化 |
| 20-40% | 轻微下降 | -5% 到 -15% |
| **40-50%** | **急剧下降** | **F1 下降 45.5%** |
| 50-80% | 持续恶化 | -50% 到 -70% |
| 80-100% | 严重退化 | 不可依赖 |

**40-50% 利用率是关键拐点**：在此之前退化可控，之后急剧恶化。这意味着 128K 窗口的有效可用容量实际上约为 50-60K token。

## 3. 模型间的退化差异

不同模型在上下文压力下的失败模式不同：

| 模型族 | 失败模式 | 表现 | 影响 |
|---|---|---|---|
| Claude 系列 | **倾向弃权** | 回答"我无法确定"、增加免责声明 | 安全但任务失败率上升 |
| GPT 系列 | **倾向幻觉** | 生成看似合理但与上下文不符的答案 | 危险——错误答案被当作正确 |
| Gemini 系列 | **混合模式** | 部分弃权、部分幻觉 | 不可预测 |

**架构启示**：

- Claude 的弃权倾向在安全关键场景更可靠，至少不会给出错误行动
- GPT 的幻觉倾向需要更强的验证层（参见 `../../../ARCHITECTURE.md` 的 Verify 阶段）
- 无论哪种模型，都需要 Context Rot 检测机制

## 4. arXiv 2601.15300：智能退化的量化

arXiv 2601.15300 对长上下文场景下的"智能退化"进行了系统量化，核心发现：

### 4.1 退化不是线性的

```text
推理准确率
  100% ─────╮
             ╲
   80% ──────╲───
              ╲
   60% ────────╲──────
                ╲
   40% ──────────╲────── ← 40-50% 窗口利用率
                  ╲
   20% ────────────╲────
                    ╲
    0% ──────────────╲──
        0%   25%   50%   75%   100%  ← 窗口利用率
```

退化曲线呈现 S 形：初期平坦，中段急剧，后段趋于底部。这意味着存在一个相对安全的操作区间（< 40%），和一个必须避免的危险区间（> 50%）。

### 4.2 任务类型敏感性

| 任务类型 | 退化敏感度 | 说明 |
|---|---|---|
| 精确信息提取 | 极高 | "找到文档中的特定数字" |
| 多步推理 | 高 | 推理链中的错误会复合累积 |
| 摘要生成 | 中等 | 统计性质的任务相对鲁棒 |
| 情感/风格判断 | 低 | 全局特征，不依赖特定位置 |

## 5. 企业影响：复合失败

### 5.1 单步可靠性的复合效应

一个 20 步的 Agent 工作流，假设每步 95% 的成功率：

```text
复合成功率 = 0.95^20 = 0.358 ≈ 36%
```

这意味着即使每个单步看起来"几乎总是成功"，整个工作流仍有 **64% 的概率失败**。当 Context Rot 将单步可靠性从 95% 降到 90%：

```text
复合成功率 = 0.90^20 = 0.122 ≈ 12%
```

Context Rot 导致的 5% 单步退化，在 20 步流程中放大为 24% 的整体成功率差异。

### 5.2 企业失败归因

行业统计数据表明，企业级 AI Agent 的失败案例中：

| 失败原因 | 占比 | 与 Context Rot 的关系 |
|---|---|---|
| 上下文漂移 / 记忆丢失 | **65%** | 直接相关 |
| 工具调用错误 | 20% | 部分由上下文退化导致工具选择错误 |
| 模型幻觉 | 10% | 上下文过长时幻觉率上升 |
| 其他（权限、超时等） | 5% | 间接相关 |

## 6. 检测信号

Context Rot 不易直接观测，但以下信号可用于早期检测：

### 6.1 行为信号

| 信号 | 描述 | 严重程度 |
|---|---|---|
| 重复工具调用 | Agent 重复调用已经成功执行过的同一工具 | 高——表明 Agent "忘记"了之前的结果 |
| 回答具体性下降 | 从具体数字/路径退化为模糊描述 | 中——注意力稀释的早期信号 |
| 对冲语言增加 | "可能"、"大概"、"不确定"等表达频率上升 | 中——模型对自身上下文的信心下降 |
| 指令遵循退化 | 开始忽略 system prompt 中的格式/风格要求 | 高——system prompt 的注意力已被稀释 |
| 目标漂移 | Agent 开始偏离原始任务目标 | 严重——核心任务信息已被淹没 |

### 6.2 量化指标

| 指标 | 计算方式 | 阈值 |
|---|---|---|
| 上下文利用率 | current_tokens / max_tokens | > 40% 预警，> 70% 触发压缩 |
| 工具重复率 | 连续 N 轮中重复调用次数 / 总调用次数 | > 20% 为异常 |
| 输出熵变化 | 当前输出的信息熵 vs. 历史基线 | 突然上升可能表示幻觉增多 |
| 指令遵循分 | 输出是否符合格式/风格/语言约束 | 连续 2 轮不遵循为异常 |

## 7. 缓解策略

### 7.1 预防性策略

| 策略 | 机制 | 效果 | 详见 |
|---|---|---|---|
| 预算监控 | 实时追踪利用率，40% 预警 | 在退化加速前介入 | `overview.md` |
| 70% 触发压缩 | 利用率达 70% 时自动触发 compaction | 避免进入 > 80% 的危险区间 | `compression-strategies.md` |
| Cache-Aware 布局 | 关键信息放首尾 | 降低 Lost-in-Middle 影响 | `context-assembly-algorithm.md` |
| todo.md 召回 | 在末尾复述当前目标 | 对抗目标漂移 | `context-assembly-algorithm.md` |
| Sub-Agent 隔离 | 子任务使用独立上下文 | 每个子 Agent 从零开始，无退化累积 | `../../../concepts/context-engineering.md` |

### 7.2 反应性策略

| 策略 | 触发条件 | 机制 |
|---|---|---|
| 紧急压缩 | 利用率 > 80% 或检测到重复工具调用 | 立即触发 LLM 摘要压缩 |
| 上下文重置 | 连续 3 轮检测到目标漂移 | 保留 system prompt + 最新摘要，清空历史 |
| 关键信息重注入 | 指令遵循分下降 | 将 system prompt 关键段落重新注入到最新位置 |
| 降级到简单任务 | 复合成功率预估 < 50% | 将复杂计划拆分为更短的子任务 |

## 8. 与认知科学的平行

Context Rot 与 Baddeley 工作记忆模型中的多个现象高度对应（详见 `../../../cognitive-architecture/working-memory-dynamics.md`）：

| Baddeley 模型概念 | Context Rot 对应 | 共享机制 |
|---|---|---|
| 容量限制 | Token 上限 | 信息容器的硬约束 |
| 被动衰减 | 早期 token 注意力权重下降 | 不被主动维护的信息逐渐消退 |
| 干扰效应 | 新工具输出覆盖旧推理 | 新信息淹没旧信息 |
| 主动维护（rehearsal） | Pinning / todo.md 召回 | 通过复述保持信息活性 |
| 中央执行器资源竞争 | 注意力矩阵的零和博弈 | 总注意力有限，分配给 A 就减少 B |

这个平行关系提供了设计启发：认知科学中对抗遗忘的策略（间隔重复、组块化、外部辅助记忆），都可以映射到 Agent 上下文管理的具体技术。

## 9. 设计决策清单

在设计 Agent 系统时，以下问题必须回答：

- [ ] 你的最长预期工作流有多少步？复合可靠性是否可接受？
- [ ] 上下文利用率是否有监控？告警阈值设在多少？
- [ ] 压缩触发点设在什么利用率？是否有分层压缩策略？
- [ ] 关键信息（目标、约束）放在什么位置？是否会被 Lost-in-Middle 影响？
- [ ] 是否有 Context Rot 检测信号的采集和报警？
- [ ] 模型的失败模式（弃权 vs. 幻觉）是否与任务风险匹配？
