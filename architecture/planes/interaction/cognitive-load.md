# 认知负荷与 Agent 交互设计

> **Evidence Status** — synthesized. 基于 Sweller 认知负荷理论、Miller 工作记忆容量研究和渐进披露原则，映射到 Agent 输出设计和人机交互决策。

**Principle Refs**: MC-01 — 不确定性显式传达；Interaction Plane 原则 — Minimal Sufficient Interruption, Progressive Disclosure

## 1. 核心问题

Agent 的输出何时对用户来说"太多了"？用户在与 Agent 交互时，其认知资源是有限的——阅读、理解、判断、决策都消耗认知带宽。当 Agent 的输出超出用户的认知处理能力时，用户要么忽略关键信息，要么做出低质量决策，要么直接放弃交互。

这是一个**交互有效性**问题：如果用户无法有效处理 Agent 的输出，那么再正确的输出也没有价值。

## 2. 理论基础

### 2.1 Sweller 认知负荷理论

John Sweller 的认知负荷理论（Cognitive Load Theory, CLT）将工作记忆中的负荷分为三种类型：

| 类型 | 定义 | Agent 交互中的对应 |
|---|---|---|
| 内在负荷（Intrinsic Load） | 由材料本身的复杂度决定 | 任务本身的复杂度——修一个简单 bug vs 做架构决策 |
| 外在负荷（Extraneous Load） | 由信息呈现方式导致的额外认知开销 | Agent 输出的冗余信息、混乱结构、术语不匹配 |
| 相关负荷（Germane Load） | 用于构建心智模型的有效认知投入 | 用户理解 Agent 推理过程、建立对 Agent 行为的预期 |

**设计目标**：最小化外在负荷，管理内在负荷（通过分块和排序），保留空间给相关负荷。

### 2.2 Miller 的 7 +/- 2 法则

George Miller (1956) 的经典发现：人的工作记忆一次只能保持约 7 +/- 2 个信息单元。虽然后续研究将这个数字修正为更接近 4 +/- 1（Cowan, 2001），核心启示不变：

- 一次给用户的**独立信息项**不应超过 4-7 个
- 信息可以通过**分块（chunking）**来压缩——把多个细节打包成一个概念
- **选项数量**同样受此限制——给用户 8 个方案选择几乎等于没给

### 2.3 渐进披露原则

渐进披露（Progressive Disclosure）的核心是**按需分层呈现**：

```text
第一层：结论 + 推荐行动（所有用户都需要）
第二层：关键证据和推理（想理解"为什么"的用户）
第三层：完整 trace、原始数据、边界情况（需要深入审查的用户）
```

这与 Sweller 的框架直接对应——第一层最小化外在负荷，第二层和第三层为愿意投入的用户提供相关负荷的空间。

## 3. Agent 交互中的认知负荷来源

### 3.1 输出长度与信息密度

Agent 的输出长度本身不构成问题，**信息密度不匹配**才会造成认知过载。一段 500 字的解释如果结构清晰、层次分明，比 100 字的信息轰炸更容易处理。

| 失败模式 | 认知负荷类型 | 示例 |
|---|---|---|
| 信息倾泻 | 外在负荷过高 | 一次输出 50 个文件的 diff，没有摘要 |
| 过度压缩 | 内在负荷未被管理 | "改了 3 处，已验证"——用户无法判断是否正确 |
| 结构混乱 | 外在负荷过高 | 重要信息和次要信息混在一起，没有视觉层次 |
| 术语错位 | 外在负荷过高 | 对非开发者说"需要 rebase 到 upstream" |

**设计原则**：输出长度应与**用户决策所需的最小信息量**匹配，而非与 Agent 内部处理的复杂度匹配。

### 3.2 审批请求的复杂度

当 Agent 请求用户审批时，用户需要理解足够多的信息才能做出**知情决策**。但"知情"和"过载"之间的界限取决于决策的风险等级：

```text
低风险审批（如格式选择）：
  → 用户只需要知道"做什么"和"推荐哪个"
  → 认知预算：1-2 个信息项

中风险审批（如代码修改）：
  → 用户需要知道"改了什么"、"为什么改"、"影响范围"
  → 认知预算：3-5 个信息项

高风险审批（如生产部署、数据删除）：
  → 用户需要完整的影响分析、回退方案、验证证据
  → 认知预算：无硬性上限，但必须结构化
```

审批请求的信息量不应一视同仁。低风险审批如果携带过多信息，用户会形成"审批疲劳"，开始不加审查地批准一切，包括高风险操作。这就是 automation complacency 的入口。

### 3.3 进度报告的频率与详细度

进度报告存在两个极端：

| 极端 | 问题 | 用户体验 |
|---|---|---|
| 太少 | 用户不知道发生了什么，焦虑感上升 | "它到底在干嘛？是不是卡住了？" |
| 太多 | 每一步都报告，信息过载 | "别告诉我你在 import 什么库" |

进度报告的最佳频率取决于：

- **任务时长**：10 秒的任务不需要进度报告，10 分钟的任务需要
- **用户参与度**：主动等待的用户 vs 后台运行时的用户
- **风险等级**：高风险任务应更频繁报告，让用户有机会介入
- **阶段变化**：跨越 milestone 时报告，而非按时间间隔报告

```text
推荐策略：
  1. 任务启动时：报告理解的目标和计划（对齐）
  2. milestone 完成时：报告完成了什么 + 下一步（进度感）
  3. 遇到异常时：报告问题 + 建议方案（风险告知）
  4. 任务完成时：报告结果 + 验证证据（闭环）
  5. 长时间无 milestone 时：每 N 分钟简短心跳（存在感）
```

### 3.4 错误解释的层次化

当 Agent 遇到错误或失败时，用户需要理解发生了什么。但不同用户在不同场景下需要不同深度的解释：

```text
层次 1 — 结论（所有用户）：
  "部署失败：数据库连接超时。"

层次 2 — 原因（想理解的用户）：
  "数据库服务器 db-prod-03 在连接池耗尽后拒绝新连接。
   当前活跃连接数 150，上限 100。"

层次 3 — 上下文（需要排查的用户）：
  "连接池耗尽的原因是迁移脚本 migration_042 持有了长事务。
   该事务从 14:32 开始运行，持续了 12 分钟，占用了 60 个连接。"

层次 4 — trace（需要深度调试的用户）：
  完整的错误堆栈、SQL 日志、连接池状态快照
```

**设计原则**：默认给出层次 1，允许用户逐层展开。不要把层次 4 的 trace 直接倒给用户——这是外在负荷的典型来源。

### 3.5 多步任务中的认知锚点

长任务中，用户的心智模型需要"抓手"来跟踪整体进度。这就是**认知锚点（Cognitive Anchor）**的作用：

```text
锚点类型：
  - Milestone 锚点："第 3 步（共 5 步）：已完成数据迁移"
  - 进度比例锚点："已处理 340/500 个文件"
  - 状态摘要锚点："当前状态：测试通过 42/50，失败 8 个"
  - 决策点锚点："需要你在这里做一个选择，然后继续"
```

认知锚点的作用：

1. **降低内在负荷**：用户不需要记住所有细节，只需要知道"我在哪里"
2. **建立预期**：用户知道接下来会发生什么
3. **支持中断恢复**：用户离开后回来，可以通过锚点快速重建上下文

## 4. 认知负荷评估框架

### 4.1 输出复杂度评估

在生成输出前，Agent（或 Harness）可以评估输出的认知负荷：

```yaml
CognitiveLoadAssessment:
  information_items: int        # 独立信息项数量
  decision_points: int          # 需要用户做决策的点
  context_switches: int         # 需要用户在不同上下文间跳转
  technical_depth: low | medium | high
  user_expertise_match: boolean # 技术深度是否匹配用户水平
  recommended_strategy: summary_first | layered | full_detail
```

### 4.2 信息密度控制矩阵

| 场景 | 用户角色 | 默认密度 | 展开选项 |
|---|---|---|---|
| 快速问答 | 所有 | 低 | 无 |
| 代码修改审批 | 开发者 | 中 | diff 详情 |
| 代码修改审批 | 管理者 | 低 | 影响摘要 |
| 错误报告 | 开发者 | 中高 | trace + 日志 |
| 错误报告 | 终端用户 | 低 | 原因解释 |
| 长任务进度 | 主动等待 | 中 | 详细步骤 |
| 长任务进度 | 后台运行 | 低 | milestone 列表 |
| 研究报告 | 所有 | 低（摘要）| 证据 + 来源 |

## 5. 与 ux-patterns.md 的关系

`ux-patterns.md` 定义了具体的交互模式（Progress Card、Approval Block、Evidence Drawer 等），本文档为这些模式提供**理论依据**：

| UX Pattern | 认知负荷理论依据 |
|---|---|
| Progress Card | 认知锚点 + 进度报告频率控制，降低长任务中的焦虑和信息过载 |
| Approval Block | 审批信息按风险分级，管理内在负荷 |
| Evidence Drawer | 渐进披露的直接实现，将外在负荷推迟到用户主动请求时 |
| Correction Receipt | 通过确认回显降低用户的不确定性（"我的纠正被收到了吗？"） |
| Partial Completion Banner | 管理用户预期，避免完成状态的认知失调 |

本文档是"为什么这样设计"，ux-patterns.md 是"具体怎么做"。

## 7. 设计检查清单

```text
输出设计：
  [ ] 输出的第一句话是否传达了核心信息？
  [ ] 独立信息项是否控制在 4-7 个以内？
  [ ] 信息是否按重要性排序（而非按发现顺序）？
  [ ] 技术深度是否匹配目标用户的知识水平？
  [ ] 是否提供了渐进披露的展开路径？

审批设计：
  [ ] 审批信息量是否与风险等级匹配？
  [ ] 低风险审批是否足够轻量（避免审批疲劳）？
  [ ] 高风险审批是否提供了充分的影响分析？
  [ ] 是否有明确的推荐选项（而非只列选项）？

进度报告：
  [ ] 报告频率是否与任务时长和风险匹配？
  [ ] 是否在 milestone 处提供了认知锚点？
  [ ] 是否避免了无信息增量的状态更新？

错误解释：
  [ ] 是否默认给出结论层（层次 1）？
  [ ] 是否支持逐层展开到 trace？
  [ ] 是否避免将原始 trace 直接暴露给非技术用户？

多步任务：
  [ ] 是否提供了整体进度的认知锚点？
  [ ] 用户中断后回来是否能快速重建上下文？
  [ ] 决策点是否与执行步骤分离（不在执行中突然要求决策）？
```

## 8. 常见反模式

| 反模式 | 表现 | 认知负荷问题 | 修复 |
|---|---|---|---|
| 信息倾泻 | 一次输出全部 diff、日志、分析 | 外在负荷爆炸 | 摘要优先 + Evidence Drawer |
| 虚假简洁 | "已完成，无问题" | 内在负荷被隐藏，用户无法校准信任 | 提供最小验证证据 |
| 选择瘫痪 | "以下 8 个方案各有优劣" | 选择过载 | 推荐方案 + 理由 + 替代项折叠 |
| 中断风暴 | 每一步都问用户 | 持续的上下文切换 | 按风险等级控制中断频率 |
| 术语墙 | 用专业术语解释错误 | 技术壁垒型外在负荷 | 用户角色感知 + 术语适配 |
| 锚点缺失 | 长任务中不报告整体进度 | 用户丢失心智模型 | milestone 锚点 + 进度比例 |
