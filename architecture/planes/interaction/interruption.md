# Interruption Strategy

> **Evidence Status** — synthesized. 长任务、审批、澄清和高风险动作中的常见交互需求。

## 1. 中断触发器

| 触发器 | 是否必须中断 | 说明 |
|---|---:|---|
| 不可逆外部动作 | 是 | 删除、付款、发送、部署、提交等 |
| 关键对象识别不确定 | 是 | 用户 A 还是客户 A，文件名相近等 |
| 权限不足 | 是 | 不能伪装完成 |
| 成本或时间超预算 | 是 | 让用户选择降级、继续或停止 |
| 低风险信息不足 | 否 | 可先读取或做保守假设并标注 |
| milestone 完成 | 可选 | 长任务需要进度，但不应频繁刷屏 |
| 用户偏好可学习 | 可选 | 只在稳定偏好出现时询问是否记住 |

## 2. 信息密度

| 场景 | 信息密度 | 格式 |
|---|---|---|
| 快速澄清 | 低 | 一个问题 + 2-3 个选项 |
| 高风险审批 | 中 | 动作、影响、目标对象、可撤销性、默认选项 |
| 事故/失败 | 高 | 已做什么、证据、风险、下一步选择 |
| 完成交付 | 中 | 结果、证据、剩余不确定性 |

## 3. 中断反模式

- Every-step Approval：每一步都问用户，破坏自治。
- Silent Risk：明知风险却继续执行。
- Dense Wall：把 trace 全量发给用户，让用户自己判断。
- Ambiguous Approval：只问“是否继续”，不说明继续会发生什么。
