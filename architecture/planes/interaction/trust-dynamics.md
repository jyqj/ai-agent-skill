# 信任动力学——从 0 到 1 的信任演化

> **Evidence Status** — synthesized. 基于 Mayer-Davis-Schoorman 组织信任模型和 Lee & See 自动化信任框架，映射到 Agent 信任建设、破坏与修复的全生命周期。

**与 trust-escalation.md 的区别**：trust-escalation.md 是信任的**工程机制**——阶段划分、升降级条件、授权范围。本文档是信任的**理论框架**——信任从何而来、如何演化、为何崩塌、怎样修复。前者是"怎么做"，后者是"为什么这样做"。

## 1. 核心问题

用户对 Agent 的信任如何从零开始建立？信任曲线是线性的还是存在关键拐点？不同品类的 Agent 依赖不同的信任维度吗？过度信任（automation complacency）是否比不信任更危险？

这些问题不是学术假设——它们直接决定了 Agent 产品的留存率和安全性。用户的初次体验如果不能建立足够信任，产品就会被弃用；但如果信任建立过快且缺乏校准，用户就会停止审查 Agent 的输出，在 Agent 犯错时承受严重后果。

## 2. 理论基础

### 2.1 Mayer-Davis-Schoorman 信任模型

Mayer, Davis & Schoorman (1995) 提出了组织信任的三维度模型，最初用于描述人际信任，但其框架对人-Agent 信任具有高度适用性：

| 维度 | 定义 | Agent 中的对应 |
|---|---|---|
| 能力（Ability） | 被信任方在特定领域的技能和胜任力 | Agent 能否正确完成任务？输出质量如何？ |
| 善意（Benevolence） | 被信任方是否关心信任方的利益 | Agent 是否为用户的目标服务，而非其他隐含目标？ |
| 诚信（Integrity） | 被信任方是否遵循信任方可接受的原则 | Agent 是否透明、一致、诚实地报告不确定性？ |

**关键洞察**：三个维度独立贡献信任，但**诚信的破坏对信任的损害最大**。一个能力不足但诚实的 Agent（"我不确定，让我验证一下"）比一个能力强但不透明的 Agent（直接给出看似自信但错误的答案）更容易被信任。

### 2.2 Lee & See 自动化信任框架

Lee & See (2004) 专门研究了人对自动化系统的信任，提出了几个 Agent 设计中直接可用的概念：

**信任的三个基础**：

| 基础 | 含义 | Agent 映射 |
|---|---|---|
| 性能（Performance） | 系统做了什么，做得多好 | Agent 的输出质量和任务完成率 |
| 过程（Process） | 系统如何运作的 | Agent 的推理过程是否可理解、可预测 |
| 目的（Purpose） | 系统的设计意图是什么 | Agent 是为谁的利益设计的 |

**适当信任（Appropriate Trust）**：信任程度应与系统的实际能力匹配。信任过高（complacency）和信任过低（disuse）都是校准失败。

**信任的动态性**：信任不是静态的——它随每次交互经验不断调整。一次严重失败可以摧毁长期积累的信任。

### 2.3 信任不对称原则

心理学研究反复发现：**信任的建立是缓慢的，信任的破坏是瞬间的**。Slovic (1993) 将此称为"信任不对称原则"（Asymmetry Principle of Trust）：

- 建立信任需要多次正面体验的累积
- 破坏信任只需要一次严重的负面事件
- 修复信任比初次建立更困难，因为负面事件会改变用户的归因框架

对 Agent 设计的启示：**防止信任破坏比加速信任建设更重要。**

## 3. 信任演化路径

### 3.1 四阶段模型

用户对 Agent 的信任通常经历四个阶段：

```text
阶段 1：初始信任（Initial Trust）
  来源：品牌声誉、产品承诺、他人推荐、第一印象
  特征：脆弱、基于预期而非经验
  持续：第一次到前几次交互

阶段 2：校准信任（Calibration Trust）
  来源：前几次交互的直接经验
  特征：快速调整、对错误高度敏感
  持续：前 5-20 次交互
  关键：这个阶段的失败体验权重极高

阶段 3：稳定信任（Stable Trust）
  来源：持续的正确交付和一致的行为模式
  特征：相对稳定、对小错误有一定容忍度
  持续：持续使用期间
  风险：可能演变为过度信任

阶段 4：弹性信任（Resilient Trust）
  来源：经历过失败和成功修复的完整循环
  特征：信任有弹性、用户理解 Agent 的能力边界
  持续：长期关系
  前提：Agent 在失败时表现出诚信（承认错误、解释原因、修复问题）
```

### 3.2 阶段转换条件

| 转换 | 促进因素 | 阻碍因素 |
|---|---|---|
| 初始 → 校准 | 第一次交互成功且透明 | 第一次交互就犯严重错误 |
| 校准 → 稳定 | 连续多次正确交付 | 校准期内出现不可解释的失败 |
| 稳定 → 弹性 | 失败后的透明处理和有效修复 | 失败后掩盖或推卸 |
| 任何阶段 → 信任崩塌 | — | 严重失败 + 不透明 + 无修复 |

### 3.3 信任曲线的非线性特征

信任不是线性增长的。几个关键的非线性特征：

```text
1. 初始跳跃或初始坠落
   第一次交互的结果会让信任大幅跳到一个初始锚点。
   好的第一次体验 → 信任快速上升到"愿意继续尝试"
   差的第一次体验 → 信任直接跌到"可能不再使用"

2. 边际递减
   从"不信任"到"基本信任"需要 5 次成功
   从"基本信任"到"高度信任"需要 50 次成功
   信任越高，每次成功的边际信任增量越小

3. 断崖式下跌
   严重失败（特别是涉及诚信的失败）导致信任瞬间跌落
   跌幅远大于任何单次成功的涨幅
   这就是信任不对称原则的表现

4. 修复折扣
   失败后即使完美修复，信任也不会恢复到失败前的水平
   用户会保留一个"它曾经失败过"的负面锚点
   多次成功修复可以逐渐缩小这个折扣，但很难完全消除
```

## 4. 信任破坏与修复

### 4.1 高破坏力行为

不是所有错误都同等地破坏信任。按破坏力排序：

| 行为 | 破坏力 | 涉及的信任维度 | 示例 |
|---|---|---|---|
| 掩盖错误 | 极高 | 诚信 | Agent 知道输出有问题但不告知用户 |
| 自信地犯错 | 高 | 诚信 + 能力 | 高置信度的错误答案，用户基于此做了决策 |
| 无法解释的失败 | 高 | 过程透明性 | Agent 失败了但不解释原因 |
| 超出授权执行 | 高 | 诚信 | 未经审批执行了高风险操作 |
| 不一致的行为 | 中高 | 能力 + 过程 | 同样的任务有时成功有时失败，无规律 |
| 能力不足 | 中 | 能力 | 任务完成质量持续不达标 |
| 过度中断 | 低中 | 能力 | 每一步都问用户，显示出缺乏独立判断 |

**关键观察**：诚信维度的破坏（掩盖、欺骗、越权）比能力维度的破坏严重得多。用户可以原谅"做不好"，但很难原谅"不诚实"。

### 4.2 信任修复策略

信任破坏后的修复需要针对被破坏的维度：

```text
能力破坏的修复：
  1. 承认特定能力的不足
  2. 在该领域增加验证步骤或降低自治等级
  3. 通过后续的正确交付逐步恢复
  时间线：中期（10-20 次成功交互）

诚信破坏的修复：
  1. 完全透明地解释发生了什么以及为什么
  2. 展示已采取的防止重复的措施
  3. 主动增加透明度（过度报告而非隐瞒）
  4. 长期的一致性行为
  时间线：长期（需要大量交互重建）

过程透明性破坏的修复：
  1. 增加推理过程的可见性
  2. 主动提供证据和来源
  3. 在不确定时明确标记
  时间线：中短期（用户看到透明行为后较快恢复）
```

### 4.3 信任修复中的 Agent 行为

```text
失败发生时：
  1. 立即承认："这个结果有错误。" （不掩盖）
  2. 解释原因："错误来自 X，因为我没有验证 Y。" （不推卸）
  3. 量化影响："这影响了 Z，没有影响 W。" （不夸大也不缩小）
  4. 提供修复："我可以通过 A 来修复，需要你的确认。" （不被动等待）
  5. 预防措施："后续我会在这类操作前增加验证步骤。" （展示学习）
```

## 5. 品类信任差异

不同品类的 Agent 依赖不同的信任维度，信任建设路径也不同：

### 5.1 Coding Agent

```text
主要信任维度：能力（代码质量）+ 诚信（验证透明性）
信任建设锚点：代码通过测试、diff 可审查、改动范围可控
信任校准机制：用户可以直接验证输出（跑测试、读代码）
信任特征：
  - 校准速度快（输出可验证）
  - 对能力不足的容忍度中等（写错了可以改）
  - 对越权执行的容忍度低（未经确认就修改了不该改的文件）
```

### 5.2 Research Agent

```text
主要信任维度：能力（信息准确性）+ 过程（来源可追溯）
信任建设锚点：引用可验证、推理有逻辑、不确定性被标记
信任校准机制：用户抽检来源和引用
信任特征：
  - 校准速度慢（用户难以完全验证研究结果）
  - 对幻觉的容忍度极低（一次虚假引用可能摧毁信任）
  - 对承认不确定性的期望高（"我不确定"比错误的确定更好）
```

### 5.3 自动化执行 Agent（CI/CD、运维）

```text
主要信任维度：能力（可靠性）+ 诚信（权限边界）
信任建设锚点：执行结果可验证、回退机制可用、权限不越界
信任校准机制：日志审计、效果验证
信任特征：
  - 需要高初始信任才会被使用（生产环境不容试错）
  - 对不一致性的容忍度极低（同一操作必须产生相同结果）
  - 信任一旦建立较稳定（行为高度可预测）
  - 信任一旦崩塌极难修复（生产事故的代价太高）
```

### 5.4 Companion / 对话 Agent

```text
主要信任维度：善意（为用户着想）+ 诚信（情感真实性）
信任建设锚点：情感共鸣、记忆连续性、个性化响应
信任校准机制：主观感受（难以客观度量）
信任特征：
  - 信任基于情感连接而非任务表现
  - 对"机械感"的容忍度低（打破沉浸感会损害信任）
  - 隐私泄露是最大的信任破坏事件
  - 信任修复困难（情感信任一旦失去难以通过逻辑重建）
```

## 6. 过度信任与校准

### 6.1 Automation Complacency

过度信任（automation complacency）是用户停止审查 Agent 输出的状态。这比不信任更危险，因为：

- 不信任时用户会主动验证或不使用——最差情况是效率低
- 过度信任时用户会盲目接受——最差情况是严重错误被执行

过度信任的信号：

```text
- 用户对审批请求的响应时间越来越短
- 用户停止展开 Evidence Drawer
- 用户对所有审批都回复 "确认" 而不查看详情
- 用户不再检查 Agent 的输出（如不运行测试、不读代码）
- 用户在 Agent 犯错时感到"惊讶"而非"预期之中"
```

### 6.2 信任校准策略

目标是让用户的信任程度与 Agent 的实际能力匹配——不多也不少。

| 策略 | 机制 | 适用场景 |
|---|---|---|
| 不确定性显式标记 | 在输出中标记置信度和不确定性 | 所有场景 |
| 能力边界声明 | 主动告知用户 Agent 在哪些方面不擅长 | 初始交互、新任务类型 |
| 验证证据展示 | 提供可独立验证的证据（测试结果、来源链接） | 校准阶段 |
| 定期失败提醒 | 在长期使用中偶尔提醒 Agent 的局限性 | 稳定阶段（防止 complacency） |
| 审批摩擦保持 | 对高风险操作保持适度的审批摩擦（不追求"零摩擦"） | 所有阶段 |
| 抽样验证建议 | 建议用户定期抽检 Agent 输出 | 稳定阶段 |

### 6.3 校准检测

```yaml
TrustCalibrationSignals:
  under_trust:
    - 用户反复要求已验证结果的证据
    - 用户手动重做 Agent 已完成的工作
    - 用户拒绝授予已证明安全的操作权限
  over_trust:
    - 审批响应时间持续缩短
    - Evidence Drawer 展开率趋近于零
    - 用户不再运行 Agent 建议的验证步骤
  well_calibrated:
    - 用户在高风险操作前查看证据
    - 用户在低风险操作时快速确认
    - 用户能准确预测 Agent 何时可能出错
```

## 7. 与意向立场的连接

`concepts/foundations/intentional-stance.md` 指出，Agent 的"目标"是功能性的（由 Harness 注入），而非内禀性的。这对信任设计有深刻影响：

**用户的信任对象是谁？** 用户信任的不是 LLM 本身，而是整个 Agent 系统（LLM + Harness + 工具 + 验证机制）。当 Agent 失败时，故障可能来自任何组件——但用户只看到一个统一的"Agent"。

**善意维度的特殊性**：LLM 没有善意或恶意，但 Agent 系统的设计可以体现善意（为用户利益优化）或违背善意（为平台利益优化、收集不必要的数据）。用户感知到的"善意"实际上是对系统设计意图的判断。

**诚信维度的工程化**：意向立场告诉我们，"Agent 是诚实的"实际上意味着"Harness 被设计为在输出中标记不确定性、不掩盖错误、不夸大能力"。诚信不是模型的性格特征，而是 Harness 的设计属性。

这个视角的实践价值：当信任出问题时，不要问"模型为什么不诚实"，而要问"Harness 的哪个设计导致了不透明的输出"。

## 8. 与 trust-escalation.md 的互补

本文档与 `trust-escalation.md` 构成理论-工程的互补关系：

| 维度 | trust-dynamics.md（本文） | trust-escalation.md |
|---|---|---|
| 关注点 | 信任为什么变化 | 信任等级如何运作 |
| 抽象层次 | 心理学模型和信任理论 | 工程阶段和升降级条件 |
| 输出 | 设计原则和校准策略 | 具体的阶段定义和授权范围 |
| 回答的问题 | "用户为什么不信任这个 Agent？" | "Agent 现在应该处于哪个信任等级？" |

trust-escalation.md 中的五个阶段（Observe → Prepare → Execute with Approval → Execute within Policy → Continuous Autonomy）可以用本文的四阶段信任模型来解释为什么阶段顺序是这样的：每个阶段的升级条件本质上是信任从校准到稳定的证据积累。

## 9. 设计检查清单

```text
信任建设：
  [ ] 初始交互是否展示了 Agent 的核心能力？（能力维度）
  [ ] Agent 是否在不确定时主动标记？（诚信维度）
  [ ] Agent 的行为是否可预测、一致？（过程维度）
  [ ] 输出是否提供了可独立验证的证据？（校准支持）

信任保护：
  [ ] 高风险操作是否保持了适度的审批摩擦？
  [ ] 是否有机制检测用户的过度信任信号？
  [ ] Agent 是否定期提醒其能力边界？
  [ ] 失败时是否立即承认并解释？

信任修复：
  [ ] 失败后是否透明地解释了原因？
  [ ] 是否展示了防止复发的措施？
  [ ] 是否在受影响的领域增加了验证步骤？
  [ ] 修复行为是否一致且持续？

品类适配：
  [ ] 信任建设策略是否匹配该品类的主要信任维度？
  [ ] 信任校准机制是否利用了该品类可验证的特征？
  [ ] 过度信任的防护是否针对该品类的特定风险？
```
