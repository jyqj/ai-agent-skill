# Financial Trading Agent Design Decisions

> **Evidence Status** — synthesized. Step Finance $4,000 万损失事件、传统量化交易系统架构、MiFID II / Dodd-Frank 合规要求、交易所风控机制观察。

## 延迟容忍度

| 策略类型 | 延迟要求 | 行情 TTL | 决策模式 |
|---|---|---|---|
| 高频做市 | < 1ms（Agent 不适合） | < 1ms | 硬件加速，非 LLM |
| 日内趋势 | < 100ms | < 500ms | 信号驱动 + 规则化执行 |
| 波段/日级 | < 1s | < 5s | 分析驱动 + 人工确认 |
| 组合再平衡 | < 10s | < 30s | 优化计算 + 批量执行 |
| 合规报告 | 分钟到小时级 | N/A | 批处理 |

默认建议：LLM-based Financial Agent 最适合波段/日级策略和组合管理。高频场景应由确定性引擎处理，Agent 仅参与策略调参和异常处理。

## 执行场所选择

| 策略 | 适合 | 风险 | 默认建议 |
|---|---|---|---|
| 单交易所直连 | 低频、单一市场 | 交易所故障即全面停止 | 小规模起步 |
| 多交易所聚合 | 最佳执行义务 | 对账复杂度、延迟增加 | 大规模必备 |
| 暗池路由 | 大单减少市场冲击 | 信息不对称风险 | 仅限有经验团队 |
| DeFi 链上执行 | 去中心化资产 | 不可逆 + MEV 攻击 + 智能合约风险 | 强化审批门控 |

默认：主交易所 + 备用交易所。DeFi 场景必须参考 Step Finance 教训，增加额度阈值和时间延迟。

## 风险限额架构

三层风险检查是 Financial Agent 的强制组件：

### Pre-Trade Risk Check（交易前）

```text
在订单提交到交易所之前，必须通过以下检查：
[ ] 单笔金额 < single_order_limit
[ ] 该标的累计头寸 < position_limit
[ ] 账户总敞口 < gross_exposure_limit
[ ] 保证金充足（margin_available > required_margin）
[ ] 日内累计亏损 < daily_loss_limit
[ ] fat-finger 检查（价格偏离 < N%，数量 < 历史 P99 的 M 倍）
[ ] 合规检查（无交易限制、无禁止名单命中）
任一失败 → 阻断订单 → 记录原因 → 通知
```

### Real-Time Risk Monitor（实时）

```text
持续运行的风险监控，不依赖交易触发：
- VaR 实时计算（每分钟或更频繁）
- 最大回撤监控
- 敞口集中度监控
- 相关性突变检测
- 市场波动率跳变检测
超过阈值 → 告警 → 如果超过硬限额 → 自动降仓或冻结
```

### Post-Trade Risk Check（交易后）

```text
每笔成交后触发：
- 头寸对账
- 风险度量重算
- 滑点记录
- 合规状态更新
每日结束时：
- 全量对账
- P&L 核对
- 合规报告生成
- 风控参数审查
```

## 合规引擎集成

| 决策 | 默认建议 |
|---|---|
| 集成方式 | 内嵌于交易路径（inline），非旁路（sidecar）。违规必须阻断订单，不是事后告警 |
| 规则更新 | 合规规则由合规官维护，Agent 启动时加载 + 定期检查更新 |
| 违规处理 | 违规 → 阻断 → 记录 → 通知合规官 → 人工决定是否例外放行 |
| 审计要求 | 每个交易决策的完整链路（信号 → 风检 → 合规 → 执行 → 确认）必须可追溯 |
| 报告义务 | 大额交易报告、可疑活动报告按监管要求自动生成 |

## 市场数据可靠性

```text
主数据源（Primary Feed）
  ↓ 正常时使用
  ↓ 同时将数据发给"一致性检查器"
备数据源（Backup Feed）
  ↓ 主源故障时自动切换
  ↓ 正常时作为交叉验证

一致性检查器：
  主备价差 < threshold → 正常
  主备价差 > threshold → 标记 suspect → 使用更保守的价格
  主源无数据 → 切换到备源 → 告警
  双源无数据 → 紧急停止所有依赖行情的交易
```

## 头寸对账频率

| 对账类型 | 频率 | 触发条件 |
|---|---|---|
| 增量对账 | 每笔成交后 | 成交回报触发 |
| 定时对账 | 每 15 分钟 | 定时任务 |
| 全量对账 | 每日收盘后 | 定时任务 |
| 紧急对账 | 事件触发 | 连接恢复 / 差异告警 / 人工请求 |

对账不一致时：以交易所为准修正本地，记录差异，如果金额 > threshold 则升级调查。

## 人工覆盖机制

| 覆盖类型 | 触发方式 | 效果 | 恢复条件 |
|---|---|---|---|
| 紧急冻结 | 一键操作 / API | 立即停止所有新订单 + 撤销所有挂单 | 人工明确解除 |
| 策略暂停 | 指定策略 | 暂停特定策略的信号和执行 | 人工重启 + 参数确认 |
| 限额覆盖 | 需要多因素审批 | 临时调整风险限额 | 自动恢复（TTL）或人工恢复 |
| 强制平仓 | 人工指令 | 按指定逻辑清仓 | N/A |
| Agent 降级 | 运维操作 | Agent 仅分析不执行 | 人工升级 |

核心原则：**人工覆盖权限始终高于 Agent 自治权限**。任何时候人类都可以冻结 Agent、撤销全部挂单、接管交易控制权。

## Step Finance 教训的设计映射

> 参考：`../../design-space/lessons-learned/step-finance-40m-loss.md`

| Step Finance 根因 | 本架构的对应设计 |
|---|---|
| 共享 API 密钥 | 每个 Agent 独立凭据 + 最小权限 + 定期轮换 |
| 无交易额度阈值 | 三层限额：单笔 / 累计 / 日限额，超限人工审批 |
| 无异常检测 | 多维异常检测（金额/频率/目标/时间）内置于运行时 |
| 零信任缺失 | 多层独立防线：设备认证 → 凭据隔离 → 操作授权 → 交易审批 → 异常检测 |
| 不可逆交易无门控 | 不可逆操作（链上转账）强制人工审批 + 时间延迟 |

## Operations 策略

Financial Trading Agent 进入生产环境后，至少需要：

```text
[ ] ConfigFingerprint: 策略参数 / 风控限额 / 合规规则 / 交易所配置 / 权限矩阵
[ ] TraceEvent: 每笔订单的完整生命周期（信号 → 风检 → 提交 → 成交 → 对账）
[ ] Regression suite: 回测验证 + 历史异常场景回放 + 风控边界测试
[ ] Shadow mode: 策略生成信号但不实际下单，对比信号质量和执行模拟
[ ] Paper trading: 模拟盘验证策略和风控逻辑的完整性
[ ] Canary: 小规模真实资金验证，逐步放大
[ ] Incident response: 异常交易 / 连接中断 / 对账差异 / 合规违规的处置流程
[ ] Kill switch: 全局紧急停止按钮，一键冻结所有交易活动
```

## 升级触发器

| 观察 | 升级 |
|---|---|
| 滑点持续超预期 | 优化执行策略 / 引入更好的执行场所 |
| 对账差异频繁 | 增强实时对账 + 调查连接可靠性 |
| 风控限额频繁触达 | 审查策略风险参数 / 考虑策略调整 |
| 合规规则变更频繁 | 引入规则版本管理 + 热加载 |
| 市场结构变化 | 策略回测 + 参数重校准 |
| 多策略交互效应 | 引入策略间协调层 + 全局风险视图 |
| 跨市场/跨链扩展 | 增强对账 + 统一风险度量 + 合规覆盖 |
