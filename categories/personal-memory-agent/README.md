# Personal Memory Agent Architecture

> **Evidence Status** — synthesized. 基于 Mem0、MemMachine（LoCoMo 基准 0.92）、Memoria、MemoryOS、Zep 以及认知科学记忆分类模型归纳。

## Core Job

Personal Memory Agent 在用户可控前提下，判断什么值得记、准确存储、高效检索、接受修正，提供个性化连续服务：

```text
观察对话 → 判断记忆价值 → 提取并分类 → 存储(带来源和时间戳) → 按需检索 → 接受修正/删除 → 衰减/更新
```

核心挑战："记忆是可追溯的主张（claim），不是现实（truth）。用户曾说的不等于现在还是这样。"每条记忆都需要来源标注、时效意识和冲突处理能力。

## 与 Companion Agent 的边界

| 维度 | Personal Memory Agent | Companion Agent |
|---|---|---|
| 核心职责 | 信息的存储、检索和管理 | 关系维护和人格表达 |
| 情感维度 | 不提供人格和情感回应 | 有情绪感知和情感回应 |
| 记忆判断 | 基于信息价值（未来有用吗） | 基于关系意义（对关系重要吗） |
| 用户感知 | 工具/服务 | 伙伴/朋友 |
| 独立性 | 可独立服务任何上游 Agent | 可能调用 Memory Agent 作为下游 |

Memory Agent 是基础设施层，Companion Agent 是应用层。Memory Agent 不关心"我是谁"，只关心"用户说过什么、需要什么"。

## 用户模型

| 用户 | 心智模型 | 信任建立方式 | 默认交互 |
|---|---|---|---|
| 日常用户 | 可靠的笔记本/助手 | 记得准、找得到、改得了 | 隐式记忆（对话中自动提取） |
| 隐私敏感用户 | 受控的信息保险箱 | 透明、可审计、可删除 | 显式确认后才记忆 |
| 多 Agent 场景 | 共享记忆层 | API 稳定、权限清晰 | 其他 Agent 通过 API 读写 |
| 长期用户 | 了解我的系统 | 长期一致性、不遗忘重要事项 | 主动应用记忆到新对话 |

## 记忆类型分类

> **Evidence Status** — synthesized. 认知科学将人类记忆分为声明性记忆（Declarative Memory）和非声明性记忆（Non-declarative Memory）。Memory Agent 的分类借鉴此框架但适配 AI 场景。

| 类型 | 定义 | 示例 | 存储特征 |
|---|---|---|---|
| 事实记忆（Factual） | 关于用户的客观信息 | 名字、生日、工作单位、家庭成员 | 高持久性，可被 supersede |
| 偏好记忆（Preference） | 用户的喜好和习惯 | 喜欢深色主题、不吃辣、偏好简洁回答 | 可变，需要跟踪变化 |
| 情景记忆（Episodic） | 具体事件和经历 | 上周去了日本旅行、昨天面试了某公司 | 时间敏感，可能过时 |
| 程序性记忆（Procedural） | 用户的工作流和操作习惯 | 代码风格偏好、文档格式要求、常用工具链 | 相对稳定，但随技能演化 |

每条记忆应标注其类型，因为不同类型的写入、更新、衰减策略不同。

## 写入策略

### 什么值得记

| 准则 | 说明 | 示例 |
|---|---|---|
| 持久性 | 信息在未来对话中可能有用 | 用户的职业、技术栈 |
| 明确性 | 用户明确陈述，而非模糊暗示 | "我叫张三" vs "有人可能觉得..." |
| 可验证性 | 信息可被回溯到具体对话 | 带时间戳和对话 ID 的来源标注 |
| 非冗余性 | 不重复已有记忆 | 写入前检查是否已存在相同或矛盾信息 |

### 什么不该记

- **临时情绪**：用户说"今天好烦"不应作为长期事实存储
- **未验证假设**：用户说"我可能要换工作"是假设，不是事实
- **敏感信息（除非必要）**：密码、财务细节、医疗信息，遵循数据最小化原则
- **对话机制信息**：用户说"帮我重新回答"是指令，不是记忆
- **他人信息**：用户提到的第三方隐私信息应谨慎处理

## 检索策略

> **Evidence Status** — synthesized. MemMachine 在 LoCoMo 基准上达到 0.92 的检索准确率，核心在于多维度检索的组合。

单一检索方式不够。有效的记忆检索需要三种策略组合：

| 策略 | 机制 | 适用场景 |
|---|---|---|
| 语义搜索（Semantic Search） | 向量相似度匹配 | 用户问"我之前说过关于旅行的事" |
| 时间感知（Time-aware） | 按时间权重排序，近期优先 | 用户问"最近有什么变化" |
| 上下文触发（Context-triggered） | 当前对话主题自动触发相关记忆 | 用户提到"面试"时自动召回之前的面试准备记忆 |

检索还应考虑：
- **记忆类型过滤**：根据当前需求只检索特定类型（事实 vs 偏好 vs 情景）
- **置信度排序**：高置信度、近期验证过的记忆优先
- **关联扩展**：检索到一条记忆后，拉取相关联的记忆（同一事件的多个方面）

## 更新与冲突处理

记忆不是 append-only 的日志，也不是 last-write-wins 的数据库。需要一套冲突处理策略：

| 场景 | 处理方式 | 示例 |
|---|---|---|
| 事实变更 | 旧记忆标记 superseded，新记忆关联旧记忆 | 用户改名：旧名标记为"曾用名"，新名设为当前值 |
| 偏好变化 | 记录偏好演变轨迹，当前偏好标记为 active | 从"喜欢 Python"变为"转向 Rust" |
| 矛盾信息 | 标记冲突，必要时向用户确认 | 用户先说在北京工作，后说在上海工作 |
| 时效过期 | 情景记忆按 TTL 降低置信度，不自动删除 | "下周要面试"在面试日期后标记为历史事件 |
| 精度提升 | 用更精确的信息替换模糊信息 | "在科技公司工作" → "在字节跳动做后端开发" |

关键原则：**不覆盖，标记 superseded**。旧记忆保留用于审计和理解演变，但检索时返回最新版本。

## 存储架构

> **Evidence Status** — synthesized. MemMachine 的三层存储架构（ground truth + 结构化提取 + 向量索引）在准确率和检索效率之间取得了有效平衡。

```text
Layer 1: Ground Truth（原始对话片段）
  ↓ 提取
Layer 2: Structured Memory（结构化记忆条目，含类型/来源/时间/置信度）
  ↓ 索引
Layer 3: Vector Index（向量化索引，支持语义检索）
```

| 层 | 用途 | 保留策略 |
|---|---|---|
| Ground Truth | 可追溯性、争议仲裁 | 按用户数据保留策略 |
| Structured Memory | 日常检索和应用 | 活跃记忆长期保留；superseded 记忆按策略清理 |
| Vector Index | 语义搜索加速 | 随 Structured Memory 同步更新 |

## 隐私设计

| 原则 | 实现 |
|---|---|
| 可查看 | 用户可浏览所有记忆条目及其来源 |
| 可编辑 | 用户可修正不准确的记忆 |
| 可删除 | 真实删除，级联到所有层（ground truth + 结构化 + 向量索引） |
| 可导出 | 用户可导出全部记忆数据（结构化格式） |
| 数据最小化 | 只存储通过写入准则检验的信息 |
| 来源标注 | 每条记忆标注来源对话和时间戳 |

## 评估方法

| 指标 | 定义 | 目标 |
|---|---|---|
| 记忆准确率（Precision） | 存储的记忆中，与事实一致的比例 | > 95% |
| 检索召回率（Recall） | 相关记忆被成功检索的比例 | > 90%（参考 MemMachine LoCoMo 0.92） |
| 冲突检测率 | 矛盾信息被正确标记的比例 | > 85% |
| 时效感知准确率 | 过时信息被正确降级的比例 | > 80% |
| 用户满意度 | 用户对记忆应用效果的主观评价 | 定期调研 |
| 隐私合规率 | 删除/导出请求的完整执行率 | 100% |

## 闭环模型

```text
Observe：接收对话内容 + 上游 Agent 的记忆请求
  → Represent：提取候选记忆 + 分类 + 来源标注
  → Decide：写入判断（值得记吗）+ 冲突检测（与已有记忆矛盾吗）
  → Act：写入/更新/标记 superseded + 索引同步
  → Verify：写入后回读验证 + 冲突解决确认
  → Update：衰减过期记忆 + 刷新置信度 + 响应删除请求
```

Stop gate：

```text
[ ] 新记忆通过写入准则检验
[ ] 来源和时间戳已标注
[ ] 冲突信息已检测并处理
[ ] 隐私约束未被违反（不记录不该记的）
[ ] 删除操作已级联到所有层
```

## 品类设计决策

| 决策 | 默认建议 |
|---|---|
| 存储 | Ground truth(原始) + 结构化提取 + 向量索引三层架构 |
| 更新 | 不覆盖；标记 superseded，保留演变历史 |
| 删除 | 真实删除；cascade 到所有索引和衍生数据 |
| 时效 | 事实类设 TTL 并降低置信度；偏好类跟踪变化轨迹 |
| 来源 | 每条记忆标注来源对话和时间戳 |
| 隐私 | 可查看/编辑/删除/导出；数据最小化 |
| 冲突 | 标记冲突，必要时向用户确认，不静默覆盖 |

## 验证与评估扩展

记忆的存储和检索准确性可通过客观 postcondition 验证（recall/precision）。但记忆价值判断（什么值得记）和用户满意度涉及主观维度：

- 客观验证：记忆检索 recall/precision、矛盾检测准确率、过期记忆失效率
- 主观维度：写入决策的合理性、用户感知的"了解我"程度
- 扩展验证：`../../concepts/beyond-verification.md` — 超越 postcondition 的验证语义
- 评估框架：`../../evaluation/subjective-eval.md` — 长期用户满意度评估
