# 注意力与显著性——Agent 如何在信息洪流中分配权重

> **Evidence Status** — mixed. 注意力理论来自认知心理学（Broadbent、Treisman、Kahneman），LLM Agent 中的映射为工程实践推导，部分策略已在 Context Engine 和 Compaction 实现中验证。

## 1. 核心问题

Agent 在每一步都面临大量信息：工具输出、对话历史、检索到的记忆、世界状态快照、系统指令。Context Window 的容量是有限的，即使在窗口内，注意力权重也不是均匀分布的。核心问题是：

```text
"在所有可用信息中，什么应该进入当前窗口？进入后，什么值得更多注意力？"
```

这不是一个存储问题（Memory Plane 解决存储），而是一个选择问题。选择错误的后果是：关键信息被淹没、无关信息占据注意力、推理质量下降。

## 2. 经典认知科学中的注意力模型

### 2.1 Broadbent 过滤模型（1958）

早期选择理论。信息在进入意识加工之前就被过滤——只有被选中的通道才会被深度处理，其余信息在感知层面就被丢弃。

```text
所有输入 → [物理特征过滤器] → 选中通道 → 深度加工
                              → 未选中通道 → 丢弃
```

对 Agent 的启发：不是所有工具输出都需要进入 Context Window。在信息进入窗口之前，就应该有一个基于结构特征的过滤层——文件大小、输出行数、与当前目标的表面相关性。

### 2.2 Treisman 衰减模型（1964）

修正了 Broadbent 的"全有全无"过滤。未被选中的信息不是被完全丢弃，而是被衰减（attenuated）——如果衰减后的信号仍然足够显著（如自己的名字），仍然可以被意识到。

```text
所有输入 → [衰减器] → 选中通道（全强度）→ 深度加工
                    → 未选中通道（衰减）→ 如果超过显著性阈值 → 仍可进入加工
```

对 Agent 的启发：被压缩或摘要化的信息不是完全不可用。如果压缩后的摘要中出现了异常信号（错误码、unexpected 关键词），应该能触发"回查原文"的操作。这就是 Tool Output Offloading 中保留 raw ref 的认知基础。

### 2.3 Kahneman 容量模型（1973）

注意力不是过滤器，而是有限的资源池。关键不是"过滤什么"，而是"分配多少认知资源"。分配策略受三个因素影响：

1. **唤醒水平**（arousal）：整体可用的注意力总量
2. **持久倾向**（enduring dispositions）：长期偏好（如对自己名字的自动注意）
3. **即时意图**（momentary intentions）：当前目标驱动的分配

```text
可用注意力资源（有限池）
  ├── 持久倾向分配：始终监控的信号（错误、安全警告）
  ├── 即时意图分配：与当前目标相关的信息
  └── 剩余资源：用于发现意外但重要的信号
```

对 Agent 的启发：Context Window 的 token 预算就是注意力资源池。System Prompt 是持久倾向（始终存在），当前任务目标是即时意图，而保留一定的"空闲预算"用于处理意外发现是重要的。

### 2.4 Posner 的注意力网络理论

Posner 区分了三种注意力网络：

| 网络 | 功能 | Agent 对应 |
|---|---|---|
| 警觉网络（Alerting） | 维持对外部信号的准备状态 | 错误检测、异常监控 |
| 定向网络（Orienting） | 将注意力转向特定位置或特征 | 工具输出的焦点提取 |
| 执行网络（Executive） | 冲突解决、目标管理 | 多信息源冲突时的决策 |

## 3. LLM Agent 的注意力机制

### 3.1 Context 组装时的信息选择

Context Pack 组装是 Agent 最核心的注意力决策。决定什么进入窗口、什么被排除，等价于决定 Agent "看到"什么。

| 选择维度 | 策略 | 认知基础 |
|---|---|---|
| 与当前目标的相关性 | 按 goal alignment 排序 | Kahneman 的即时意图 |
| 信息的新鲜度 | 最近的观察优先 | 工作记忆的时间衰减 |
| 信息的显著性 | 异常信号优先 | Treisman 的显著性阈值 |
| 信息的结构优先级 | System Prompt > Goal > Recent > Background | Broadbent 的通道选择 |
| 信息的可靠性 | 高 trust 来源优先 | Kahneman 的持久倾向 |

```text
Context Pack 组装流程（注意力视角）：

1. 固定分配（pin）：System Prompt、Goal、Constraints
   → 认知等价：持久倾向 + 工作记忆的主动维护

2. 目标驱动分配：与当前 milestone 相关的 observations、memory
   → 认知等价：即时意图驱动的资源分配

3. 显著性驱动分配：异常信号、错误、冲突
   → 认知等价：衰减信号超过阈值时的自动注意

4. 预算分配：剩余 token 空间用于背景信息
   → 认知等价：剩余认知资源的分配
```

### 3.2 工具输出的显著性评估

工具调用返回的结果可能从几行到几千行。Agent 需要从中提取"值得注意"的部分。

**显著性信号**：

| 信号类型 | 示例 | 处理策略 |
|---|---|---|
| 错误和异常 | exit code != 0、stack trace、error 关键词 | 自动提升优先级 |
| 与预期不符 | 搜索预期有结果但为空；文件内容与假设矛盾 | 触发反思（→ metacognitive-control） |
| 关键数据点 | API 返回中的 status field、测试通过/失败数 | 提取到摘要 |
| 重复模式 | 多个文件中出现相同错误 | 聚合为模式而非逐条列出 |
| 安全信号 | 权限错误、认证失败、敏感信息 | 强制注意 + 告警 |

**非显著信号的处理**：

```text
大量正常输出（如 200 个测试全部通过的详细日志）
  → 压缩为摘要："200 tests passed, 0 failed"
  → 保留 raw ref 供需要时回查
  → 认知等价：Broadbent 过滤，不进入深度加工
```

### 3.3 任务相关性过滤

信息的重要性不是绝对的，而是相对于当前目标的。同一条信息在不同任务阶段的显著性可能完全不同。

```text
场景：Agent 在调试一个网络请求超时问题

高显著性（与当前目标直接相关）：
  - 网络配置文件
  - 超时相关的错误日志
  - HTTP client 的重试策略代码

低显著性（与当前目标无关）：
  - 数据库 schema 定义
  - UI 组件样式文件
  - 构建配置（除非构建配置导致了超时问题）

关键点：如果低显著性信息中出现了异常信号（如数据库连接也超时了），
它的显著性应该被提升——这可能揭示了根因不是网络层而是基础设施层。
```

### 3.4 异常检测作为显著性信号

异常（unexpected observations）是最重要的显著性来源之一。认知科学中，新异刺激（novelty）会自动捕获注意力，不受当前任务意图的控制。

Agent 中的异常信号：

| 异常类型 | 表现 | 为什么重要 |
|---|---|---|
| 预测误差 | 预期文件存在但不存在 | 世界模型需要更新 |
| 矛盾信号 | 两个工具返回冲突信息 | 可能存在版本不一致或数据竞争 |
| 模式打破 | 前 9 个测试通过、第 10 个以完全不同的方式失败 | 可能暴露了边界条件 |
| 无声失败 | 命令成功但输出为空 | 可能是权限问题或路径错误 |

**设计原则**：异常信号的注意力优先级应该高于"正常的任务相关信息"。正常信息确认假设，异常信息可能推翻假设——推翻假设的信息价值更高。

## 4. 与 Working-Memory-Dynamics 的关系

注意力和工作记忆是紧密耦合的：

```text
注意力决定什么进入工作记忆
工作记忆的容量限制注意力能处理的信息量
注意力的分配策略决定工作记忆的组块结构
```

| 工作记忆概念 | 注意力视角 | 共同影响 |
|---|---|---|
| 容量限制 | 注意力资源池大小 | Context Window 的 token 预算 |
| 组块化 | 注意力的单位 | 信息应按 chunk 而非原子 token 组织 |
| 主动维护（pin） | 持续注意力分配 | 关键信息需要持续占用注意力资源 |
| 被动衰减 | 注意力撤回 | 不再关注的信息自然衰减 |
| 中央执行器 | 注意力控制器 | Context Engine 的核心决策逻辑 |

## 5. 与 Compaction 策略的关系

Compaction 是注意力管理的工程实现之一：当注意力资源（token 预算）耗尽时，通过压缩来释放资源。

| Compaction 策略 | 注意力模型 | 说明 |
|---|---|---|
| Snip（裁剪工具输出） | Broadbent 过滤 | 直接丢弃低显著性的原始输出 |
| Micro-summary（微摘要） | Treisman 衰减 | 保留关键信号，衰减细节 |
| Collapse（折叠历史） | 容量模型的资源回收 | 释放被旧信息占用的注意力资源 |
| Auto-compaction（全自动） | 注意力网络的自适应 | 根据当前目标动态调整保留策略 |

Compaction 应该遵循注意力原则：

1. **高显著性信息最后压缩**：错误信息、异常观察、关键决策点
2. **低显著性信息优先压缩**：成功的常规操作、已确认的事实
3. **压缩不等于遗忘**：保留 raw ref，允许注意力在需要时"回看"
4. **压缩应保留异常信号**：即使整段历史被压缩，其中的异常仍应在摘要中保留

## 6. Plane Mapping

| Plane | 注意力影响的参数 | 设计影响 |
|---|---|---|
| Context Plane | Context Pack 的组装优先级、压缩策略、offloading 阈值 | 组装逻辑应显式实现注意力分配策略，而非简单 FIFO |
| Representation Plane | Observation 的显著性标注、摘要粒度 | 表示层应为每个 observation 标注显著性等级，支持 Context Engine 做选择 |
| Prompting Plane | System Prompt 的结构分区、指令优先级 | 高优先级指令应放在注意力最强的位置（开头和结尾），避免 Lost in the Middle |
| Memory Plane | 记忆检索的相关性排序 | 检索结果应按与当前目标的相关性排序，而非仅按语义相似度 |
| World State Plane | 状态快照的选择性加载 | 不是所有 world state 都需要加载——只加载与当前任务相关的维度 |

## 7. 设计启发

### 7.1 信息进入窗口前应有显著性评估

不要把所有工具输出原样塞入 Context Window。在信息进入窗口前，应该有一个评估层：

```text
工具输出 → [显著性评估]
  → 高显著性：原样保留或结构化提取
  → 中显著性：摘要 + raw ref
  → 低显著性：仅保留 raw ref，不进入窗口
```

### 7.2 异常信号应自动提升优先级

无论当前任务焦点是什么，以下信号应自动获得注意力：

- 错误和异常退出
- 与预期矛盾的观察
- 安全相关的警告
- 资源异常（内存、时间、token 消耗突增）

### 7.3 注意力分配应随任务阶段变化

不同阶段关注的信息类型不同：

| 阶段 | 注意力焦点 | 降低关注的信息 |
|---|---|---|
| 理解需求 | 用户意图、约束、上下文 | 具体实现细节 |
| 搜索/探索 | 代码结构、文件组织、模式 | 已确认的事实 |
| 实现 | 具体代码、API 签名、测试结果 | 高层需求描述 |
| 验证 | 测试输出、diff、边界条件 | 实现过程的中间步骤 |
| 交付 | 变更摘要、影响范围 | 探索过程的细节 |

### 7.4 保留"注意力余量"用于意外发现

不要把 100% 的 token 预算都分配给已知需要的信息。保留 10-15% 的余量用于：

- 处理工具输出中的意外发现
- 容纳检索返回的额外相关信息
- 应对任务方向调整带来的新信息需求

### 7.5 检查清单

```text
Context Pack 组装是否有显式的优先级策略（不是简单追加）？
工具输出是否经过显著性评估再决定保留策略？
异常信号是否有自动提升优先级的机制？
注意力分配是否随任务阶段动态调整？
是否保留了处理意外发现的 token 余量？
Compaction 是否优先压缩低显著性信息？
压缩后是否保留了异常信号的摘要？
System Prompt 中的关键指令是否放在注意力最强的位置？
信息选择是否同时考虑了目标相关性和异常显著性？
```

### 7.6 常见反模式

| 反模式 | 症状 | 修正 |
|---|---|---|
| 注意力均匀分布 | 所有信息同等对待，关键信号被淹没 | 引入显著性评估和优先级分配 |
| 纯目标驱动 | 只关注与当前目标相关的信息，忽略异常 | 为异常信号保留自动注意力通道 |
| 注意力僵化 | 任务阶段变了但关注的信息类型没变 | 阶段转换时重新评估注意力分配 |
| 预算耗尽 | 所有 token 都被占满，无法处理新信息 | 保留注意力余量 + 主动压缩 |
| 摘要丢失异常 | 压缩时把异常信号一起压掉了 | 压缩策略应显式保留异常信号 |
