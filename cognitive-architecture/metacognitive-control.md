# 元认知控制——Agent 何时该反思、何时该行动

> **Evidence Status** — mixed. 元认知理论来自认知心理学（Flavell、Nelson & Narens），LLM Agent 中的映射为工程实践推导。

## 1. 核心问题

Agent 在执行过程中需要不断做一个元决策：

```text
"我应该继续行动，还是停下来反思？"
```

这个决策本身不是关于任务的——它是关于"我的认知过程是否在正轨"的。这就是元认知（metacognition）：对自己认知过程的监控和调节。

## 2. 行动 vs 反思的 Tradeoff

### 2.1 过度行动

Agent 一直调用工具但不检查方向是否正确。

```text
症状：
  Step 1: 读文件 A → 没找到想要的内容
  Step 2: 读文件 B → 没找到想要的内容
  Step 3: 读文件 C → 没找到想要的内容
  ...
  Step 15: 读文件 O → 还是没找到

问题：Agent 在第 3 步就应该停下来想"我的搜索策略是否正确"
```

这就是 ReAct 漫游——Agent 在行动空间中随机游走，每一步都"合理"但整体没有方向。

### 2.2 过度反思

Agent 一直在"想"但不动手验证。

```text
症状：
  Thought: 可能是 A 的问题
  Thought: 但也可能是 B
  Thought: 如果是 A，那么 C 也会受影响
  Thought: 但 C 受影响的前提是 D 为真
  Thought: D 是否为真取决于...
  ...
  [20 轮 thought，0 次工具调用]

问题：Agent 在第 2 步就应该调用工具去验证 A 或 B
```

这就是无限自省幻觉——Agent 在推理空间中打转，每一步都"有道理"但没有新证据进入。

### 2.3 平衡点

```text
行动应该产生新信息（observation）
反思应该利用已有信息调整方向（strategy update）

如果行动不产生新信息 → 停下来反思
如果反思不改变行动策略 → 停止反思去行动
```

## 3. 触发反思的信号

Agent 不应该在每一步都反思（成本太高），也不应该永远不反思（方向会偏）。以下信号提示"该反思了"：

| 信号 | 典型表现 | 对应知识库 |
|---|---|---|
| 进展停滞 | 连续 N 步 TaskState 无变化；ToolCall 返回相似空结果 | `loop-detection.md` |
| 置信度下降 | 之前的假设被推翻；新观察与预期矛盾 | -- |
| 意外结果 | 文件不存在；测试不该通过却通过了；API 返回意外格式 | -- |
| 计划偏离 | 原计划 5 步现在第 10 步；修改文件数远超预期 | -- |
| 用户反馈 | "不是这个意思"；用户给出新方向 | -- |
| 资源超支 | Token 消耗过半但任务完成不足 25% | `depth-budgeting.md` |

关键判断：意外结果不一定是错误，但它意味着 Agent 的世界模型与现实不一致，需要暂停更新认知再继续行动。

## 4. 反思后的行动选择

反思不是目的，反思之后的**策略调整**才是。

### 4.1 修正方向

改变策略或工具选择，继续推进。

```text
反思前：用 grep 逐文件搜索关键词
反思后：改用 AST 分析工具精确定位符号引用
```

这是最常见的反思结果——方法不对，换一个方法。

### 4.2 回退

回到上一个已知正确的状态（checkpoint），从那里重新出发。

```text
反思前：基于错误假设修改了 3 个文件
反思后：撤销修改，回到修改前的状态，基于新假设重新开始
```

对应知识库：`design-space/patterns/checkpoint-hydration.md`

### 4.3 升级

请求人工介入。

```text
触发条件：
  - Agent 多次反思后仍无法找到有效策略
  - 问题超出 Agent 的能力边界
  - 决策需要业务判断而非技术判断
```

升级不是失败。过晚升级（继续浪费资源）比过早升级（打断用户）代价更大。

### 4.4 放弃当前子目标

切换到可行的替代路径。

```text
反思前：尝试通过修改源码修复问题
反思后：发现源码修改影响面太大，改为在配置层面 workaround
```

对应知识库：`cognitive-architecture/goal-hierarchy.md` 中的目标冲突解决。

### 4.5 范式切换

从当前推理范式切换到更适合的范式。

```text
反思前：使用 ReAct 逐步探索
反思后：发现问题可以分解为独立子任务，切换到 Plan-Execute
```

对应知识库：`paradigms/paradigm-routing.md`

## 5. 元认知控制的实现层次

| 层次 | 控制方式 | 成本 | 适用场景 |
|---|---|---|---|
| 硬编码 | 固定规则（如"超过 10 步必须反思"） | 最低 | 简单场景、已知的常见问题 |
| 启发式 | 基于信号的条件触发 | 中等 | 多数生产场景 |
| 自适应 | Agent 自己决定何时反思 | 最高 | 高度不确定的探索任务 |

推荐：以启发式为主，硬编码兜底，在有限范围内允许自适应。

```text
硬编码兜底：
  - 连续 N 步无进展 → 强制反思
  - 资源消耗超预算 → 强制反思
  - 用户反馈负面 → 强制反思

启发式触发：
  - 置信度变化 → 建议反思
  - 意外结果 → 建议反思
  - 计划偏离 → 建议反思

自适应空间：
  - Agent 可以选择在启发式触发后立即反思还是再观察一步
  - Agent 可以选择反思的深度（快速检查 vs 全面回顾）
```

## 7. 设计启发

### 7.1 反思应该有证据

反思不是"再想想"，而是"基于已有证据重新评估策略"。

```text
差的反思：
  "让我重新想想... 也许应该试试别的方法"

好的反思：
  "最近 5 步都在搜索文件但没找到目标。
   已有证据：grep 'functionName' 在 src/ 下没结果。
   新假设：函数可能被重命名了，或在其他目录。
   新策略：搜索函数的调用者，从调用链反向定位。"
```

### 7.2 反思应该有时间预算

不要让反思变成另一种形式的漫游。

```text
反思预算：
  快速检查（1 步）：检查目标对齐、进展评估
  标准反思（2-3 步）：回顾证据、修正假设、调整策略
  深度回顾（5+ 步）：仅在重大卡点时使用，且需要新证据输入
```

### 7.3 检查清单

```text
Agent 是否有明确的反思触发机制？
触发反思后是否有结构化的反思流程（不是自由联想）？
反思是否基于证据而不是猜测？
反思是否有时间/步骤预算？
反思后是否有明确的行动选择（修正/回退/升级/放弃/切换）？
是否有硬编码兜底防止永远不反思？
是否有机制防止过度反思（反思循环）？
```
