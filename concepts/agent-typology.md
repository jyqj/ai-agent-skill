# Agent 类型学

> **Evidence Status** — synthesized. 跨项目观察归纳，多品类 Agent 运行模式对比。


## 为什么需要类型学

Agent 不只有一种运行模式。按执行目标和验证方式的差异，可以区分三类基本形态，每类对应不同的核心循环和架构重心。

三类的划分依据是两个正交问题：（1）是否存在可客观验证的完成条件？（2）行为主体是单个 Agent 还是多 Agent 交互？任务交付型有明确完成条件且以单 Agent 为主；探索陪伴型没有明确完成条件，同样以单 Agent 为主；涌现协作型的关键行为产生在 Agent 之间而非 Agent 内部，不论单个 Agent 是否有明确任务。

当前知识库以 ORDA-VU 闭环为主线，覆盖的是任务交付型 Agent。但现实中大量 Agent 系统并不以"完成一个可验证的任务"为目标——长期陪伴、创意探索、多主体涌现协作，都需要不同的循环结构和评估标准。把三类形态显式区分出来，才能避免用一套框架硬套所有场景。


## 1. 任务交付型（Task-Delivery）

这是最成熟、边界最清晰的 Agent 形态。用户给出目标，Agent 自主规划、执行、验证，直到目标达成或明确失败。

核心循环是 ORDA-VU：Observe → Represent → Decide → Act → Verify → Update。每一步都有明确的输入输出契约——观察产出表示，决策产出行动意图，行动产出效果，验证判定效果是否匹配目标。整个循环围绕一个可检验的完成条件运转，执行深度可以量化为目标持久性、工具链可靠性、效果验证覆盖率等指标。

架构重心落在 Effects Plane（行动是否真正改变了外部世界）、Tools Plane（工具链的可靠性和组合能力）和 Recovery Plane（失败后能否从断点恢复）。典型的失败模式是 Ghost Success——工具返回成功但现实未改变，Agent 却以为任务已完成。

Coding Agent、Workflow Agent、Ops Agent 属于这一类。当前知识库的主要覆盖范围就是这个形态。


## 2. 探索陪伴型（Exploratory-Companion）

陪伴型 Agent 没有一个会"完成"的任务。用户不会在某个时刻说"好了，陪伴结束，验收通过"。这意味着 ORDA-VU 中的 Verify 阶段失去了锚点——没有可量化的完成条件可供验证。

这类 Agent 的循环更接近 OREU：Observe → Resonate → Express → Update。各阶段的含义如下：**Observe** 感知用户当前输入及情感信号；**Resonate** 在用户状态、对话历史、长期记忆与当前语境之间寻找共鸣点，而非在候选行动中做最优选择；**Express** 以语言、情感回应或创意内容作用于用户内在状态，而非调用工具改变外部系统；**Update** 将本轮交互中获得的用户偏好、关系进展和记忆线索写回长期存储，维持跨会话一致性。

Resonate 与 Decide 不同：Decide 在候选行动中选择最优解，Resonate 在用户的情感状态、对话历史、长期记忆和当前语境之间寻找共鸣点。一个学习辅导 Agent 需要判断的是"这个学生现在是困惑、沮丧还是即将突破"，而非"用哪个工具"。Express 与 Act 也不同：Act 的输出是工具调用和外部效果，Express 的输出是语言、情感回应、创意内容，作用于用户的内在状态而非外部系统。

架构重心从 Effects Plane 转移到 Memory Plane 和 Interaction Plane。Memory Plane 承担跨会话的人格一致性、用户偏好记忆、关系演进追踪，远超单纯的上下文管理。Interaction Plane 处理对话节奏、情感基调、表达风格的动态调整。评估指标是用户满意度、对话深度、记忆一致性，而非任务完成率。这些指标天然带有主观性和滞后性，评估体系的设计难度远高于任务交付型。

典型失败模式有两种：记忆漂移（Agent 忘记了之前建立的关系上下文，用户感到"它不记得我了"）和人设崩坏（Agent 在某次回复中表现出与长期人格不一致的行为）。

Companion Agent、创意助手、学习辅导 Agent 属于这一类。

> OREU 循环目前作为陪伴型 Agent 的推荐参考模型，尚未在独立范式文件中定义。


## 3. 涌现协作型（Emergent-Collaborative）

前两种形态讨论的都是单个 Agent 的内部循环。涌现协作型关注的是另一个层次：多个 Agent 各自运行独立循环，系统级行为从它们的交互中涌现出来。

这与当前知识库 T6 Multi-Agent 中的 Coordinator-Worker 模式有本质区别。Coordinator-Worker 有一个中央调度者负责任务分解和结果聚合，Worker 的自主性被限定在子任务范围内。涌现协作型没有中央调度者——每个 Agent 根据局部信息和共享信号自主决策，全局秩序从局部交互中自组织而成。

三种主要子类型：

- **Market-based**：Agent 通过竞标、定价、拍卖机制分配任务和资源。每个 Agent 有自己的效用函数，通过市场价格信号协调行为。适用于资源分配、负载均衡、去中心化服务编排。
- **Debate-based**：Agent 通过论证、质疑、反驳来收敛到结论。不同 Agent 持有不同立场或信息源，最终结论是多方博弈的均衡点，而非某个 Agent 单独推理的产物。适用于复杂推理验证、多源事实核查、对抗性评估。
- **Stigmergy**：Agent 通过修改共享环境间接通信，类似蚁群通过信息素协调觅食路径。Agent 之间不直接对话，而是在共享的 World State 上留下痕迹，其他 Agent 根据这些痕迹调整自己的行为。适用于大规模并行探索、分布式优化。

架构重心在 Orchestration Plane 的去中心化协议和 World State 的共享信号设计。核心挑战是交互协议能否让群体收敛到有用的结果、收敛速度是否可接受、是否会陷入振荡或死锁。

这些子类型共享一个特征：系统级行为无法通过分析单个 Agent 来预测。因此，典型失败模式也发生在群体层面而非个体层面：协调失败（Agent 各自优化局部目标导致全局次优）、振荡（系统在几个状态之间反复跳转无法收敛）、信号淹没（共享环境中的信息过多，有效信号被噪声掩盖）。

> 具体协作协议（Market-Based、Debate、Stigmergy、Ensemble）的定义见 [collaboration-paradigms.md](../paradigms/collaboration-paradigms.md#去中心化协作形态)。


## 为什么区分三类

区分这三类形态的根本原因是：它们面对的**设计决策空间**不同。任务交付型的核心设计问题是效果验证——行动是否真的改变了外部世界，以及如何在验证不可达时安全退化。涌现协作型的核心设计问题是共识收敛——多个自主体的局部决策如何在没有中央调度的情况下产生有用的全局结果。探索陪伴型的核心设计问题是关系持续性——如何在没有明确终止条件的交互中维持人格一致、记忆连贯和情感适配。三类形态的验证方式、失败模式和架构重心都由各自的核心设计问题决定，试图用一套统一框架覆盖所有形态，必然导致某些形态的关键问题被忽略或扭曲。

三类之间的边界不是绝对的。同一个产品可能在不同场景下展现不同形态：陪伴型 Agent 接到具体任务请求时切换为任务交付模式，Research Agent 的多源交叉验证可能采用 debate 机制，大型协作系统中每个节点可能是完整的任务交付型 Agent。但架构选择应锚定主要形态，按其核心设计问题分配最多的设计资源。试图构建"通用 Agent"——一个在三种形态上都做到最优的系统——通常意味着三种形态都没有被充分设计。


## 形态不是互斥的

实际系统常常混合多种形态——上一节已给出典型例子。这里强调的工程含义是：混合形态的系统需要在模块边界上做显式切换，而不是在一个循环内隐式兼容。例如，陪伴型 Agent 切换到任务交付模式时，应显式切换到 ORDA-VU 循环和效果验证逻辑，而非试图让 OREU 循环"兼容"任务完成条件。


## 对知识库的影响

| 对比项 | 任务交付型 | 探索陪伴型 | 涌现协作型 |
|---|---|---|---|
| 核心循环 | ORDA-VU | OREU | 多循环涌现 |
| 验证方式 | Effect Verification | 用户反馈 + 记忆一致性 | 群体收敛指标 |
| 关键 Plane | Effects, Tools, Recovery | Memory, Interaction | Orchestration, World State |
| 失败模式 | Ghost Success | 记忆漂移、人设崩坏 | 协调失败、振荡 |
| 评估方法 | 任务完成率 + 效果验证 | 用户满意度 + 对话质量 | 收敛速度 + 解质量 |

当前知识库以任务交付型为主线，探索陪伴型和涌现协作型的 Plane 定义、范式选择和评估体系尚未展开，是后续扩展的方向。
