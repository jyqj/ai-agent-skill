# 有界理性与 Harness 必然性

> **Evidence Status** — theoretical

## 核心论点

Agent——无论是人还是 AI——都在有限的算力、有限的信息和有限的时间下做决策。这是智能系统的结构性约束，无法通过"更大的模型"消除。正因如此，Harness 是 Agent 系统的必然组成部分，而非可选的辅助设施。

## 1. Simon 的有界理性

Herbert Simon 在 1955 年提出 bounded rationality：人类（以及任何决策者）不具备完全理性所需的三个前提：

- **完备信息**：决策者不可能知道所有相关事实
- **无限算力**：决策者不可能穷举所有方案并计算最优解
- **稳定偏好**：决策者的目标和评价标准会随情境变化

这三条对 LLM Agent 完全适用：

| 完全理性的要求 | LLM Agent 的现实 |
|---|---|
| 完备信息 | 只能看到进入上下文窗口的内容；窗口外的世界等于不存在 |
| 无限算力 | 受 token 预算、延迟要求、成本约束限制 |
| 稳定偏好 | "目标"是外部注入的 prompt，可能模糊、冲突或不完整 |

## 2. 满意解 vs 最优解

Simon 提出的替代方案是 satisficing（满意化）：不追求最优解，而是设定一个"足够好"的阈值，找到第一个满足阈值的方案就停下来。

这个思想在 Agent 架构中无处不在：

- **MVA 阶梯（Minimum Viable Agent）**：不要从最复杂的架构开始，而是找到能完成当前任务的最小方案，验证有效后再递增复杂度。这就是满意解思维的工程化。
- **Depth Budget**：给每次推理分配有限的深度预算，到了预算就必须输出当前最好的结果，而不是无限展开搜索。
- **Compaction**：上下文快满时，保留"足够做出下一步决策"的信息，放弃完整性幻觉。

满意解思维的实践含义：**停止条件和资源上限是设计决策，而非妥协**。没有明确停止条件的 Agent 更危险，因为它会无限消耗资源却不保证更优结果。

## 3. 为什么 Harness 必然存在

如果 Agent 具备完全理性，它只需要一个模型就够了：给它任务，它计算出最优解，输出结果。但有界理性意味着模型单独无法完成闭环：

| 有界理性的约束 | 导致的缺口 | Harness 的补偿机制 |
|---|---|---|
| 信息不完备 | 模型不知道自己不知道什么 | 工具系统（获取信息）、World State（追踪状态） |
| 信息会过期 | 模型无法感知时间流逝 | Freshness 管理、TTL、refresh-before-act |
| 算力/上下文有限 | 推理深度受限、历史会被遗忘 | Compaction、Memory 系统、Checkpoint |
| 目标可能模糊 | 模型可能误解任务 | TaskEnvelope、PromptContract、Success Criteria |
| 行动效果不可见 | 模型无法直接观察现实 | Effect Verification、Read-after-write |
| 决策可能错误 | 没有纠错机制就会一路错下去 | Recovery、Rollback、Human-in-the-loop |

**Harness 的作用是弥补有界理性的结构性缺口。** 这就是 `what-is-agent.md` 将 Agent 定义为"模型 + Harness + 接口 + 验证"而非"更好的模型"的原因。

## 4. Context Rot 是有界理性的直接后果

有界理性的信息维度有两个推论：

1. Agent 能处理的信息总量有上限（上下文窗口）
2. 信息的可靠性随时间衰减（世界在变，表示不变）

这两者结合就产生了 Context Rot：

- **物理层面**：上下文越来越满，早期信息被压缩或丢弃
- **语义层面**：保留在上下文中的信息可能已经过期，但模型无法区分"当前事实"和"历史快照"
- **决策层面**：基于过期表示做出的决策，即使推理过程完美，结论也是错的

Context Rot 是有界理性在时间维度上的必然表现，而非 bug。扩大窗口只是推迟问题，有效的应对是显式管理信息生命周期：标注 freshness、设置 TTL、在关键决策前 refresh。

## 5. 实践启发

### 不要追求完美的单次推理

有界理性意味着任何单次推理都可能出错，因为信息不完备、算力有限、目标可能被误解。提升推理精度有帮助，但更有效的应对是**设计可容错的多步闭环**：

- 每步都产出可验证的中间结果
- 每步都有回退路径
- 错误被检测后可以被纠正，而不是被累积

这就是 ORDA-VU 的设计动机：不依赖一次推理的正确性，而是通过 Observe → Represent → Decide → Act → Verify → Update 的循环来逐步逼近正确结果。

### 资源约束应该被显式管理

有界理性的"有限算力"应当作为约束暴露给决策层，而非作为实现细节隐藏：

- Depth Budget 让 Agent 知道自己还能探索多深
- Token Budget 让 Agent 知道自己还能说多少
- Cost Gate 让 Agent 知道自己还能花多少

这些约束不会降低 Agent 的能力，反而会提高决策质量，因为它迫使 Agent 区分"值得深入"和"可以跳过"的问题。

### 满意解思维应贯穿设计

在 Agent 设计中，满意解思维意味着：

- **选架构**：不要上来就搞 Multi-Agent + RAG + Long-term Memory，先问"最简单的什么方案能满足核心需求"
- **选范式**：选"对当前任务够用"的范式，而非"最强"的范式（这就是 paradigm-routing 的逻辑）
- **选验证**：优先验证最高风险的失败路径，而非穷举所有可能
- **选停止**：以用户的 success criteria 为停止线，而非追求完美

## 6. 与经典 AI 的连接

有界理性不只是 LLM Agent 的问题。经典 AI 中的搜索算法（A*、Alpha-Beta 剪枝）、启发式方法、分层规划、anytime algorithms 都是有界理性的工程响应。LLM Agent 的 Harness 工程继承了这个传统，只是把"有限算力"扩展到了"有限上下文 + 有限工具 + 有限验证能力"。

| 经典 AI 的有界理性策略 | LLM Agent 中的对应 |
|---|---|
| 启发式搜索（A*）：用不完美但高效的评估函数替代穷举 | paradigm-routing：用任务特征快速选择范式，而非尝试所有范式 |
| Alpha-Beta 剪枝：提前排除不可能的分支 | Depth Budget：限制探索深度，聚焦最可能的方向 |
| 分层规划（HTN）：把复杂任务分解为可管理的子任务 | Plan-Execute 范式：先规划再逐步执行 |
| Anytime Algorithms：随时可以中断并返回当前最好的结果 | Checkpoint + Compaction：任何时刻都有可恢复的中间状态 |
| 有限视野规划：只规划未来 N 步 | 单次循环只决定下一步行动，而非完整计划 |

这种连续性说明，有界理性是智能系统设计的永恒约束，早于 LLM 时代。理解这个传统，有助于从经典 AI 中借鉴已被证明有效的应对策略。

## 7. 常见误区

### "模型足够大就不需要 Harness"

有界理性是结构性约束，不是规模问题。更大的模型有更大的上下文窗口和更好的推理能力，但它仍然：
- 无法感知时间流逝（信息过期问题不会消失）
- 无法直接操作外部世界（需要工具）
- 无法自我验证（需要外部反馈）
- 无法跨会话记忆（需要持久化存储）

模型规模的增长扩展的是有界理性的"界"，但不消除"有界"本身。

### "有界理性 = Agent 很弱"

恰恰相反。承认有界理性是设计强大系统的前提。不承认有界理性的系统更脆弱：它假设自己总是对的，因此没有纠错机制、没有验证步骤、没有回退路径。显式承认自身有限性的 Agent，比假装无所不能的 Agent 更可靠。

### "满意解 = 低质量"

满意解不是"将就"。它是在有限资源下最理性的策略。追求最优解的系统在资源耗尽前可能什么都没产出；满意解系统在资源耗尽时至少有一个可用的结果。在实际应用中，"足够好且可靠"几乎总是优于"理论最优但不确定能否达到"。

## 8. 有界理性的设计检查清单

在设计或评审一个 Agent 系统时，可以用以下清单检查是否正确应对了有界理性的约束：

```text
信息维度：
[ ] Agent 是否知道自己看不到什么？（Out-of-scope、Unobserved 是否被标记）
[ ] 关键信息是否有 freshness 管理？（TTL、refresh-before-act）
[ ] 信息冲突是否被保留而非静默合并？（ConflictRecord）

算力维度：
[ ] 是否有明确的 Depth Budget？
[ ] 是否有 Token/Cost Budget？
[ ] 上下文满时是否有 Compaction 策略？
[ ] 是否有 Checkpoint 机制防止从头开始？

目标维度：
[ ] success criteria 是否明确且可验证？
[ ] 停止条件是否合理（不过早、不过晚）？
[ ] 目标模糊时是否有澄清机制？

容错维度：
[ ] 每步是否有可验证的中间结果？
[ ] 关键行动是否可回退？
[ ] 连续失败时是否有策略调整机制？
```
