# 具身认知与情境性 -- Agent 不是悬浮在真空中

> **Evidence Status** — theoretical

## 核心论点

传统 AI 把智能看作纯粹的信息处理——一个大脑悬浮在真空中，接收输入、计算输出。具身认知和情境认知的研究告诉我们这是错的：认知不仅发生在"大脑"里，还受制于身体和环境。对 Agent 架构来说，这意味着工具不是"附件"，而是认知能力的延伸；Agent 的能力不是固定的属性，而是由当前环境动态决定的。

## 1. 具身认知的核心主张

具身认知（Embodied Cognition）是认知科学中的一个研究范式，核心主张是：

> 认知过程不仅由"大脑"完成，还深刻依赖于身体的结构、感知能力和行动能力。

经典例子：
- 人类的空间推理能力部分依赖于手的操作——不让你用手比划，你的空间推理就会变差
- 情绪不只是"大脑的状态"，还受身体姿态影响——身体紧绷会影响判断
- 外部工具会改变认知：纸笔让你能处理超出工作记忆的复杂计算

### 对 Agent 的翻译

LLM Agent 没有身体，但它有功能上等价的东西：

| 人类的"身体" | Agent 的"身体" | 认知功能 |
|---|---|---|
| 眼睛、耳朵 | Interface Plane（输入接口） | 感知世界 |
| 手、嘴 | Tool Runtime、Execution Host | 作用于世界 |
| 工作记忆 | 上下文窗口 | 短期信息处理 |
| 长期记忆 | Memory 系统、文件系统 | 知识积累与检索 |
| 运动系统 | API 调用、代码执行 | 精确操作 |
| 本体感觉 | 内部状态监控 | 知道自己的状态 |

**关键洞察**：当我们给 Agent 添加或移除一个工具时，我们改变的不是"Agent 旁边有什么附件"，而是 Agent 的认知能力本身。

## 2. 工具作为认知延伸

### Clark 和 Chalmers 的延展心智

Andy Clark 和 David Chalmers 在 1998 年提出了"延展心智"（Extended Mind）假说：如果一个外部工具在功能上替代了一个内部认知过程，那么这个工具应该被视为认知系统的一部分。

经典思想实验：Otto 把信息记在笔记本上，需要时查阅。Inga 把同样的信息记在大脑里。如果我们承认 Inga "知道"这些信息，那 Otto 也"知道"——只是他的知识部分存储在外部。

### 对 Agent 设计的直接推论

**工具不是附件，是认知器官**

一个没有代码编辑器的 Coding Agent 不是"少了一个工具"，而是"失去了编辑代码的认知能力"。一个没有搜索工具的 Research Agent 不是"功能不全"，而是"视野被截断到训练数据"。

这改变了我们对工具设计的思考方式：

| 附件思维 | 认知延伸思维 |
|---|---|
| "给 Agent 加个搜索工具" | "扩展 Agent 的信息获取边界" |
| "工具失败了，重试一次" | "Agent 的认知能力暂时受损，需要评估影响" |
| "减少工具数量以简化系统" | "缩减认知能力必须评估哪些能力被截断" |
| "工具只是 API 包装" | "工具的设计直接影响 Agent 的思维质量" |

**工具的可用性影响推理质量**

如果工具设计得不好（返回值不清晰、错误信息模糊、延迟高），Agent 的推理质量会下降——不是因为模型变笨了，而是因为它的"感知器官"和"行动器官"变迟钝了。

工具设计不应只考虑"能不能用"，还要考虑"用起来是否支持高质量推理"：

- 返回值是否结构化？非结构化的返回需要额外的解析推理
- 错误信息是否可操作？"Error 500"比"数据库连接超时，请检查连接配置"需要更多推理来定位问题
- 响应是否及时？长延迟打断推理链的连贯性

## 3. 情境认知：能力是动态的

### 什么是情境认知

情境认知（Situated Cognition）的核心主张是：认知不是脱离情境的抽象过程，而是在特定环境中、为了特定目的、使用特定资源的具体活动。

简单来说：**同一个人在不同环境中展现出的"智能"是不同的**。一个数学家在讲台上能推导复杂公式，在厨房里可能做不好一道菜——不是因为他的"智能"变了，而是情境变了。

### 对 Agent 的推论

Agent 的能力不是一个固定的属性，而是由当前情境动态决定的：

| 情境因素 | 对 Agent 能力的影响 |
|---|---|
| 可用工具 | 直接决定 Agent 能感知什么、能做什么 |
| 权限等级 | 限制 Agent 能操作的范围 |
| 上下文内容 | 决定 Agent "知道什么" |
| 用户状态 | 影响什么样的交互方式是合适的 |
| 执行环境 | 决定行动的安全性和可逆性（沙箱 vs 生产环境） |
| 网络状况 | 影响工具调用的成功率和延迟 |
| 时间预算 | 限制探索深度和验证范围 |

### 不要设计"通用智能体"

情境认知的推论是：不要试图设计一个"在所有情境下都有效"的通用 Agent，而要设计一个"在特定情境中高效工作"的系统，并让它能感知和适应情境变化。

实践含义：

1. **情境感知**：Agent 应该在任务开始时评估当前情境——有什么工具可用、有什么权限、上下文里有什么信息、用户期望什么——然后据此调整策略

2. **能力降级**：当情境受限时（工具不可用、权限不足、信息缺失），Agent 应该能优雅降级而不是静默失败。比如搜索工具不可用时，明确告知"无法搜索最新信息，以下回答基于训练数据"

3. **情境迁移**：当 Agent 从一个情境切换到另一个情境（不同的项目、不同的用户、不同的执行环境），它的策略和假设需要相应更新

## 4. 多模态作为具身边界的扩展

多模态不只是"多几种输入格式"，从具身认知的角度看，它是 Agent 具身边界的本质性扩展：

### 文本时代

Agent 只能"看到"文本 → 只能处理可文本化的信息 → UI 状态、物理环境、表情动作都不可见

### 视觉扩展

Agent 能"看到"图像和截图 → 能处理 UI 状态、图表、代码截图 → 具身边界从"文本世界"扩展到"视觉世界"

### 音频扩展

Agent 能"听到"声音 → 能处理会议录音、语音指令、环境声音 → 具身边界扩展到"听觉世界"

### 物理具身

Embodied Robot Agent 有真正的物理身体 → 能直接感知和操作物理世界 → 具身边界扩展到"物理世界"

每一次具身边界的扩展都带来新的表示边界问题（怎么把新模态的信息转换为可处理的表示）和新的效果边界问题（怎么通过新模态验证效果）。这就是为什么 `representation-and-effects.md` 的三道边界框架对所有模态都适用。

## 5. 环境中的交互循环

具身认知的另一个关键概念是"感知-行动循环"（perception-action loop）：Agent 不是先感知完再行动，而是通过行动来获得新的感知，通过感知来引导行动。

这与 ORDA-VU 的循环结构高度一致：

```text
感知-行动循环（具身认知）：
  感知环境 → 行动 → 感知变化 → 调整行动 → ...

ORDA-VU：
  Observe → Represent → Decide → Act → Verify → Update → Observe → ...
```

关键区别在于 ORDA-VU 在循环中加入了 Represent（显式处理感知到表示的转换）和 Verify（显式检查行动效果），这是对具身认知理论的工程化增强——承认 Agent 的"身体"（工具和接口）是不完美的，需要在每一步都检查"感知是否准确"和"行动是否有效"。

## 6. 实践启发

### 工具设计应以认知支持为目标

设计工具时，不要只考虑"功能是否实现"，还要考虑"是否支持 Agent 的高质量推理"：

- 返回值是否包含足够的上下文信息（不只是结果，还有来源、时效、可靠性）
- 错误信息是否指向可操作的修复路径
- 工具之间是否有清晰的组合关系（Agent 需要理解"先用 A 再用 B"的逻辑）

### 情境评估应该是循环的第一步

在 ORDA-VU 的 Observe 阶段，除了观察任务相关的外部状态，还应该观察 Agent 自身的情境：

- 当前有哪些工具可用？
- 权限是否足够完成任务？
- 上下文中是否有足够的信息？
- 用户是否在等待（时间敏感）？

### 能力边界应该被显式声明

Agent 应该能在任务开始时声明自己的能力边界——"在当前情境下，我能做 X、Y、Z，但不能做 A、B、C"。这比让 Agent 默默尝试然后失败要好得多。

这连接到 `agent-epistemics.md` 中的 Out-of-scope 类型：不仅是信息不可达，更是能力不可达。
