# 意向立场与 Agent 目标归因

> **Evidence Status** — theoretical

## 核心论点

当我们说"Agent 想要完成任务"时，这个"想要"是一种解释工具，不是对模型内部状态的描述。LLM 没有内在目标，目标是 Harness 通过 TaskEnvelope 和 PromptContract 注入的。混淆解释立场和本体论事实，会导致错误的设计决策和错误的故障归因。

## 1. Dennett 的三种立场

Daniel Dennett 提出，我们可以用三种不同的立场来预测和解释一个系统的行为：

### 物理立场（Physical Stance）

从物理规律出发预测行为。比如预测一块石头掉下来会砸到脚——你用的是重力和运动学。

对 Agent：从 GPU 计算、token 采样概率、网络延迟出发解释模型输出。准确但太底层，几乎没有设计指导价值。

### 设计立场（Design Stance）

从系统的设计意图出发预测行为。比如预测闹钟到了设定时间会响——你不需要理解电路，只需要知道它被设计成"到点就响"。

对 Agent：从 Harness 的设计出发预测 Agent 行为。"它会先搜索再回答，因为 prompt 要求它这样做"。这是 Agent 架构师最常用的立场。

### 意向立场（Intentional Stance）

把系统当作"有信念、有欲望、有意图"的主体来预测行为。比如预测一个棋手会走某步——你假设"它想赢"，然后推导"想赢的话应该走这步"。

对 Agent："它选择先查看测试结果，因为它想确保代码是正确的"。这种说法在日常交流中方便且有效，但它是一种**解释策略**，不是对模型心智状态的描述。

## 2. 意向立场作为解释工具

意向立场之所以有效，是因为一个被良好设计的系统的行为**恰好像是**有目标的行为。但"像是"和"是"之间有本质区别：

| 维度 | 意向立场的说法 | 实际发生的事情 |
|---|---|---|
| 目标 | "它想完成用户的任务" | TaskEnvelope 定义了 success criteria，Harness 驱动循环直到满足 |
| 信念 | "它认为文件已经修改了" | World State 中的快照显示文件状态为 modified |
| 推理 | "它决定先运行测试" | 模型在当前上下文下，对"运行测试"这个 token 序列赋予了最高概率 |
| 坚持 | "它不放弃，一直在重试" | Harness 的 retry 逻辑在循环中重新发起请求 |

意向立场不是"错误"的——它是高效的交流和预测工具。问题出在把它当成本体论事实，从而做出错误推论。

## 3. 拟人化陷阱

当团队在意向立场和本体论事实之间滑动时，会产生系统性的设计错误：

### 错误归因

| 拟人化说法 | 隐含假设 | 正确归因 |
|---|---|---|
| "模型误解了任务" | 模型有理解能力但理解错了 | Prompt 描述不够明确，或上下文缺少关键信息 |
| "模型不愿意执行" | 模型有意愿但选择拒绝 | 安全训练的边界判断触发了拒绝，或 prompt 触发了保守策略 |
| "模型忘记了之前的指令" | 模型有记忆但遗忘了 | 上下文窗口溢出导致早期信息被截断 |
| "模型在偷懒" | 模型有动机但缺乏努力 | Depth Budget 不足、停止条件设置过于宽松、或 compaction 丢失了关键指令 |
| "模型学会了" | 模型积累了经验 | 上下文中注入了之前的反馈，模型条件概率发生了变化 |

### 设计偏差

拟人化思维导致的设计偏差：

- **过度信任**："模型很聪明，它能自己想到需要验证"——实际上如果 Harness 没有验证步骤，模型不会主动验证
- **过度解读**："模型给出了高置信度，说明它很确定"——置信度是模型在当前上下文下的概率输出，不是对结论可靠性的自我评估
- **责任错位**："模型犯了错"——如果错误可以通过 Harness 防止（更好的 prompt、验证步骤、回退机制），那是 Harness 设计的问题，不是模型的问题

## 4. 目标的工程化注入

如果模型没有内在目标，那 Agent 的"目标"从哪里来？答案是 Harness 的目标注入机制：

### TaskEnvelope

TaskEnvelope 是目标的载体。它把用户的意图结构化为机器可处理的规格：

- **输入**：用户的原始请求
- **目标**：success criteria -- 什么算完成
- **约束**：权限边界、资源预算、时间限制
- **验证方式**：怎么确认目标已达成

没有 TaskEnvelope，模型只是在"生成最可能的下一个 token"。有了 TaskEnvelope，模型的生成被导向一个具体的完成标准。

### PromptContract

PromptContract 是行为的规范。它定义了模型在特定角色下"应该怎么做"：

- 什么时候该行动、什么时候该请求确认
- 什么格式输出、什么信息必须包含
- 遇到不确定性时的默认策略

### Autonomy Level

Autonomy Level 是意向性的递进授权。从 L0（只回答）到 L5（持续自治），每一级都是 Harness 对 Agent 的"目标授权"的扩大：

- L0-L1：Harness 只注入"理解任务"的目标，不注入"执行任务"的目标
- L2-L3：Harness 注入"执行"目标，但保留审批权
- L4-L5：Harness 注入完整的任务目标，Agent 在规则边界内自主推进

## 5. 实践启发

### 使用设计立场做架构决策

当你设计 Agent 时，用设计立场思考："给定这个 Harness 设计，Agent 会表现出什么行为？"而不是"Agent 想做什么？"

- 不说"Agent 想先搜索"，说"Prompt 指示 Agent 在回答前先搜索"
- 不说"Agent 决定跳过验证"，说"当前配置没有验证步骤"
- 不说"Agent 学会了处理这种情况"，说"Memory 中注入了之前的处理模式"

### 用意向立场做日常交流

在团队讨论和文档中，意向立场是可以用的——说"Agent 会先查看测试结果"比"Harness 驱动的循环会在下一步调度 test-runner 工具"更高效。但要时刻意识到这是一种**方便的简写**，在出现问题时能切换回设计立场做根因分析。

### 故障排查时切换到设计立场

当 Agent 行为不符合预期时，立即从意向立场切换到设计立场：

1. 不问"它为什么误解了"，问"上下文中缺了什么信息"
2. 不问"它为什么不执行"，问"Harness 的哪个条件阻止了执行"
3. 不问"它为什么幻觉了"，问"哪些表示的置信度被错误地升级为事实"

## 6. 功能性目标 vs 内禀性目标

需要澄清一个常见的困惑：说"LLM 没有内在目标"不是说 Agent 系统没有目标——Agent 系统当然有目标，但这些目标是**功能性的**（由 Harness 注入和维护的），不是**内禀性的**（从模型内部涌现的）。

| 目标类型 | 来源 | 持久性 | 可观测性 | 可修改性 |
|---|---|---|---|---|
| 内禀性目标 | 系统内部涌现 | 持久且难以改变 | 难以直接观测 | 难以从外部修改 |
| 功能性目标 | 外部注入（TaskEnvelope） | 随任务生命周期存在 | 可直接检查 | 可随时调整 |

功能性目标的优势在于**可检查性**和**可修改性**：你可以审查 TaskEnvelope 中的 success criteria 来判断目标是否合理，可以在运行时修改目标来适应变化。内禀性目标（如果存在的话）则难以审查和修改——这恰恰是对齐问题的根源。

从这个角度看，当前 LLM Agent "没有内在目标"反而是优势：它让目标管理变成了一个工程问题而非哲学问题。

## 7. 团队沟通中的意向立场管理

在实际团队中，完全避免意向立场的说法是不现实的，也没有必要。关键是建立团队共识：意向立场是**交流的快捷方式**，不是**分析的基础**。

### 日常交流（意向立场可接受）

- "Agent 选择了先搜索再回答"
- "Agent 不确定应该用哪个工具"
- "Agent 在这种情况下会犹豫"

### 设计讨论（切换到设计立场）

- "当前 prompt 指示 Agent 在回答前先搜索"
- "工具选择的评分函数在这两个工具上得分接近"
- "安全训练在这类输入上触发了保守策略"

### 故障分析（必须使用设计立场）

- 不说"Agent 误解了用户意图"，说"TaskEnvelope 中的 success criteria 与用户实际期望不匹配"
- 不说"Agent 幻觉了"，说"上下文中缺少关键事实，模型在训练分布中找到了一个看似合理但错误的补全"
- 不说"Agent 放弃了"，说"循环在第 N 步触发了停止条件，但任务实际上未完成"

建立这种分层的语言习惯，能让团队在高效沟通和精确分析之间取得平衡。

## 8. 与其他基础概念的连接

意向立场与有界理性互补：有界理性解释了为什么 Agent 需要 Harness（能力有限），意向立场解释了为什么 Agent 的目标需要外部注入（没有内在目标）。两者共同支撑了知识库的核心架构原则：**Agent 是被设计的系统，而非自主涌现的智能体。**

意向立场也连接到 `metacognition.md`：当我们说"Agent 知道自己不知道什么"时，这同样是意向立场的说法。实际发生的是 UnknownRecord 被填充、Confidence 字段被标记。元认知能力由 Harness 通过显式检查点和状态追踪实现，而非模型"自发的自我反思"。
