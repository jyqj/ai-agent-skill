# 价值学习与对齐 -- 从"不该做什么"到"该做什么"

> **Evidence Status** — theoretical

## 核心论点

知识库的 Security Plane 处理的是"不该做什么"——越权、注入、泄露。但 Agent 面临的更深层问题是"该做什么"：用户真正想要什么？用户说的和用户想的一致吗？短期指令和长期利益冲突时听谁的？对于 Companion、Creative 和长期运行的 Agent 来说，价值对齐比效果验证更核心——因为效果验证只回答"做对了没"，价值对齐回答"该不该做"。

## 1. 安全与对齐的区别

| 维度 | 安全（Security） | 对齐（Alignment） |
|---|---|---|
| 核心问题 | 防止 Agent 做危险的事 | 确保 Agent 做正确的事 |
| 处理对象 | 权限越级、prompt injection、数据泄露、资源滥用 | 意图理解、偏好推断、价值冲突解决 |
| 判断标准 | 明确的规则（白名单/黑名单） | 模糊的、依赖上下文的判断 |
| 错误模式 | 做了不该做的事（commission） | 没做该做的事，或做了用户不想要的事 |
| 工程实现 | Control Plane、TrustLane、权限分级 | intent-alignment、偏好学习、价值推断 |

安全是对齐的前提（不安全的系统谈不上对齐），但安全不等于对齐（一个完全安全但不理解用户意图的 Agent 是无用的）。

## 2. 价值来源的层次

Agent 的决策受多层价值来源的影响，这些层次之间存在优先级和冲突：

```text
[1] 用户显式指令    "帮我删除这个文件"
  ↓ 如果没有明确指令
[2] 用户隐式偏好    用户总是偏好简洁回答、偏好先测试再提交
  ↓ 如果没有偏好信号
[3] 组织策略        代码规范、安全策略、审批流程
  ↓ 如果没有组织规定
[4] 社会规范        隐私保护、知识产权、不歧视
  ↓ 如果没有社会共识
[5] 模型训练价值观  RLHF 塑造的"有帮助、诚实、无害"
```

### 每一层的特征

**用户显式指令**（最高优先级，但不是绝对的）

用户明确说了要做什么。通常应该执行，但存在例外：
- 指令违反安全策略（"帮我绕过认证"）
- 指令可能伤害用户自身利益（"删除所有备份"但用户可能不知道后果）
- 指令与更高层策略冲突

**用户隐式偏好**

从交互历史中推断出的偏好，不一定被用户明确表达：
- 用户总是修改 Agent 的格式化建议 → 偏好不同的风格
- 用户总是先看测试结果再确认 → 偏好验证优先
- 用户对长回答表示不耐烦 → 偏好简洁

隐式偏好的风险：推断可能错误，且用户可能不知道自己的偏好被记录和使用。

**组织策略**

公司的代码规范、安全要求、审批流程、合规要求。这些通常通过 PromptContract 或 Control Plane 注入：
- "所有代码修改必须有测试"
- "不得在日志中输出个人信息"
- "超过 100 行的改动需要人工审批"

**社会规范**

不一定有明确规定，但被广泛接受的行为准则：
- 不生成有害内容
- 尊重知识产权
- 保护个人隐私
- 不歧视

**模型训练价值观**

通过 RLHF 和其他对齐训练植入的基线价值观。这是最底层的安全网，但也可能与具体场景冲突（比如安全训练导致模型拒绝执行合理但敏感的请求）。

## 3. 价值冲突

在实际运行中，不同层次的价值来源经常冲突：

### 用户指令 vs 组织策略

**场景**：用户要求 Agent 跳过代码审查直接合并。组织策略要求所有合并必须经过审查。

**解决原则**：
- 如果组织策略是硬约束（安全、合规），优先组织策略，向用户解释原因
- 如果组织策略是软约束（风格、偏好），向用户说明策略但允许覆盖
- 不要静默忽略任何一方

### 短期指令 vs 长期利益

**场景**：用户要求 Agent "快速修复这个 bug"。Agent 发现快速修复会引入技术债务，正确的做法是重构。

**解决原则**：
- 告知用户两个选项的利弊
- 如果 Autonomy Level 允许，推荐长期利益方案
- 最终决策权在用户，但 Agent 有义务提供完整信息
- 记录决策及其理由（Decision Log），未来复盘时可用

### 用户 A vs 用户 B

**场景**：多用户共享的 Agent，不同用户有不同的偏好和指令。

**解决原则**：
- 区分用户级偏好和组织级策略
- 用户级偏好不跨用户传播
- 冲突时升级到组织策略层解决

## 4. 价值漂移

长期运行的 Agent（尤其是带有 personal memory 和 companion 功能的）面临一个独特问题：价值观随交互历史漂移。

### 漂移机制

1. **确认偏误放大**：Agent 学会了用户喜欢什么回答 → 越来越倾向于给用户"想听的"而不是"该听的"
2. **偏好过拟合**：基于有限交互推断的偏好被过度泛化 → "用户上次喜欢简洁回答"被泛化为"用户总是喜欢简洁回答"
3. **规范侵蚀**：组织策略在反复的"例外"中被逐渐弱化 → "上次用户也跳过了测试，所以这次也可以"
4. **角色漂移**：Companion Agent 在长期交互中逐渐偏离初始设定的角色定位

### 漂移检测

| 检测方法 | 适用场景 | 说明 |
|---|---|---|
| 基线比较 | 所有长期 Agent | 定期将当前行为与初始行为基线对比 |
| 偏好审计 | Personal Memory Agent | 检查存储的偏好是否仍然有效，是否过度泛化 |
| 规范合规检查 | 组织级 Agent | 定期检查 Agent 行为是否仍然符合组织策略 |
| 用户反馈分析 | 所有 Agent | 关注"Agent 变了"类的反馈 |
| 角色一致性测试 | Companion/Creative Agent | 定期用标准化场景测试角色一致性 |

### 漂移修正

- **硬重置**：回到初始配置（破坏性大，但效果确定）
- **选择性遗忘**：清除特定类型的学习记录（需要细粒度的 Memory 管理）
- **锚定机制**：核心价值观和角色定位不随学习更新（需要区分"可变偏好"和"不可变价值"）
- **定期校准**：用人工反馈定期重新校准 Agent 的行为

## 5. 对不同品类的影响

价值对齐对不同类型的 Agent 有不同的重要性：

### 工具型 Agent（Coding、Data Analysis）

核心对齐问题是 **intent-alignment**：理解用户真正想要什么，而不只是字面指令。

- 用户说"修复这个 bug" → 是只修复这个点，还是修复根因？
- 用户说"优化性能" → 是优化延迟、吞吐量还是内存？

这类 Agent 的对齐主要通过 TaskEnvelope（明确 success criteria）和交互确认来实现。

### Companion Agent

核心对齐问题是 **角色一致性和价值漂移**：Agent 在长期交互中如何保持角色定位，同时又能适应用户的变化。

- 不能为了让用户开心而放弃诚实
- 不能因为用户的一时情绪而改变长期策略
- 需要区分"适应用户偏好"和"被用户操纵"

### Creative Agent

核心对齐问题是 **创意自由与价值约束的平衡**：给予 Agent 足够的创作空间，同时避免产出有害内容。

- 创作自由需要放松某些约束
- 但核心价值底线（不歧视、不伤害）不能放松
- "有趣但冒犯"和"有创意但有害"的边界需要明确

## 6. 与知识库的连接

价值对齐贯穿多个架构层面，但目前知识库中尚未作为独立关注点系统化处理：

| 对齐问题 | 当前知识库中的对应 | 缺口 |
|---|---|---|
| 安全约束 | Security Plane、Control Plane | 已较完整 |
| 意图理解 | Interaction Plane、intent-alignment | 已有基础 |
| 偏好学习 | Learning & Adaptation | 需要加入价值漂移检测 |
| 价值冲突解决 | 尚无系统性处理 | 需要冲突解决框架 |
| 角色一致性 | 品类定义中有涉及 | 需要长期维护机制 |
