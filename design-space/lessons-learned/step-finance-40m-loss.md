# Step Finance 4,000 万美元 DeFi 损失事件

> **Evidence Status** — mixed（基于新闻报道与第三方安全分析，非官方技术复盘）.
> - [5 Real AI Agent Security Breaches in 2026 - Beam.ai](https://beam.ai/agentic-insights/ai-agent-security-breaches-2026-lessons)

---

## 事故场景

2026-01，Solana 链上的 DeFi 投资组合管理器 Step Finance 遭受攻击。攻击者入侵了公司高管的设备，通过该设备获取了 AI 交易 Agent 的控制权限。Agent 在无人工审批的情况下自主转移了 261,000+ SOL 代币（价值约 $2,700-3,000 万），最终仅恢复 $470 万。事件导致原生代币暴跌 97%，公司关闭。

本案例揭示了 AI Agent 控制金融资产时的核心安全问题：**当 Agent 拥有无限额转账权限，且不可逆交易无审批门控时，单一设备入侵即可导致全面资产损失**。

---

## 故障链（按时间顺序）

```
阶段 1: 初始入侵
─────────────────────────────────────────────
T+0       攻击者入侵高管设备（初始攻击向量）
          ↓ 获取设备上存储的 API 密钥/钱包私钥
          ↓ 密钥为共享密钥，多个 Agent 和服务使用同一凭据

阶段 2: 权限提升
─────────────────────────────────────────────
T+?       攻击者通过共享 API 密钥获得 AI 交易 Agent 的操作权限
          ↓ Agent 设计允许无限额转账，无单笔/累计交易上限
          ↓ 大额转账不需要人工审批或多因素认证

阶段 3: 资产转移
─────────────────────────────────────────────
T+?       Agent 执行大量 SOL 代币转出交易
          ↓ 261,000+ SOL 被转移至攻击者控制的地址
          ↓ 区块链交易不可逆，一旦确认即无法回滚
          ↓ 异常转账模式未触发任何告警

阶段 4: 不可逆后果
─────────────────────────────────────────────
T+后续    仅 $470 万被追回（总损失约 $3,000-4,000 万）
          ↓ 原生代币暴跌 97%
          ↓ 用户信任崩塌
          ↓ 公司关闭
```

### 关键放大因素

区块链交易的不可逆性使得本案例与传统数据库删除事件有本质区别：
- 数据库删除后可从备份恢复（如 PocketOS 案例）
- 区块链资产转移后**无任何技术手段可强制回滚**
- 这意味着金融 Agent 的安全防线必须是"阻止发生"而非"事后恢复"

## 根因分析

### 1. 共享 API 密钥——单点失效

Step Finance 的 AI 交易 Agent 与其他服务共享 API 密钥。行业数据显示 45.6% 的 DeFi 团队使用共享 API 密钥。这意味着：

```
共享密钥架构                        应有的隔离架构
─────────────────────              ─────────────────────
一个密钥控制所有服务                  每个 Agent/服务独立密钥
一次泄露 = 全面失控                  一次泄露 = 单服务失效
攻击面 = 所有使用该密钥的入口          攻击面 = 单个入口
                                   密钥轮换不影响其他服务
```

### 2. 无交易额度阈值

Agent 被设计为可以执行任意金额的转账，没有单笔交易上限、24 小时累计上限或异常金额检测。在 DeFi 场景中，这相当于给 Agent 签发了一张无限额度的支票。

有效的额度控制应分层设计：

```
层级          控制手段                触发条件
────────     ─────────────────      ─────────────────
第 1 层      自动执行                单笔 < $1,000
第 2 层      Agent 自检 + 日志       单笔 $1,000-$10,000
第 3 层      人工审批                单笔 > $10,000
第 4 层      多因素审批 + 时间延迟    单笔 > $100,000 或 24h 累计 > $50,000
```

### 3. 无异常检测

261,000+ SOL 的转出在任何正常业务模式下都是异常的，但系统未配置任何异常检测。有效的检测维度包括：
- **金额异常**：单笔/累计超过历史 P99
- **频率异常**：短时间内大量交易
- **目标异常**：转账到未知/新创建地址
- **时间异常**：非营业时间的大额操作

### 4. 零信任架构缺失

从高管设备入侵到全面资产损失的路径没有任何阻断点。在零信任架构下，即使设备被入侵，攻击者仍需突破多层独立防线：

- 设备认证（已被突破）
- 密钥访问（共享密钥，无额外保护）
- Agent 操作授权（无独立授权）
- 交易审批（无审批门）
- 异常检测（无检测）

5 层防线全部缺失或无效。

## 架构缺陷

| 缺失的 Plane/Pattern | 说明 |
|----------------------|------|
| Per-Agent Credential Isolation | 共享 API 密钥导致单点失效。一个设备入侵等于所有 Agent/服务的全面沦陷 |
| Transaction Threshold Gate | 大额转账无审批门控。Agent 可执行任意金额的不可逆交易 |
| Anomaly Detection | 异常转账模式（金额、频率、目标、时间）未被检测。261,000+ SOL 的转出未触发告警 |
| Zero-Trust Architecture | 攻击链中无任何独立阻断点。设备入侵 → 密钥获取 → 资产转移的路径畅通无阻 |
| Irreversible Action Gate | 区块链交易不可逆，但未被分类为"需要特殊审批"的高危操作 |
| Multi-Factor Authorization | 高价值操作无多因素认证。Agent 仅凭 API 密钥即可执行全部操作 |

## 设计教训

1. **金融 Agent 必须有逐笔交易的额度阈值和累计限制**。单笔交易上限、24 小时累计上限、N 天滑动窗口累计上限——三个维度同时约束。超过阈值的交易必须强制进入人工审批流程，Agent 无法自行授权。

2. **每个 Agent 必须有独立凭据，绝不共享密钥**。独立凭据意味着：每个 Agent 有自己的 API key/钱包；密钥按最小权限配置（交易 Agent 只能转账，不能修改配置）；密钥定期轮换且互不影响；一个 Agent 的凭据泄露不影响其他 Agent。

3. **不可逆交易必须有独立的审批门和时间延迟**。区块链交易不可逆意味着安全设计必须是"阻止发生"而非"事后恢复"。大额转账应引入时间锁：发起交易 → 等待 N 分钟/小时 → 期间可被取消 → 超时后才执行。这为人工干预提供了时间窗口。

4. **异常检测是金融 Agent 的强制组件，而非可选增强**。基于金额、频率、目标地址、时间模式的多维异常检测应内置于 Agent 运行时。检测到异常时自动暂停操作并通知运维，而非仅记录日志。

5. **"权限过度"是 Agent 安全事故中最常见且后果最严重的根因**。在所有金融 Agent 事故中，Agent 拥有远超任务需要的权限是最一致的模式。权限审计应在 Agent 部署前强制执行："Agent 需要哪些最小权限"比"Agent 可以做什么"更重要。

6. **金融 Agent 必须采用零信任架构**。假设任何单一防线都可能被突破：设备可能被入侵、密钥可能泄露、网络可能被拦截。安全设计应确保攻击者需要同时突破多层独立防线才能造成实质性损害。

7. **Agent 控制金融资产时，Agent 的权限模型应与人类金融从业者的合规要求对齐**。人类交易员有交易限额、合规审查、双人操作要求。AI Agent 控制同等或更大规模的资产时，至少应满足同等级别的控制要求。
