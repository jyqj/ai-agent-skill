# Behavior Correction Layer

> **Evidence Status** — production-validated. 来自 GenericAgent 的 `do_no_tool` 生产实现，在多项目 agent 循环中稳定运行。

## 问题

LLM 在 agent 循环中不总是按预期行为执行。常见偏差包括：连续多轮返回空内容、输出代码块却不调用工具（"伪执行"）、虚假地声明任务已完成、以及流异常导致的截断响应。如果没有系统性检测和纠正机制，这些偏差会导致 agent 静默卡死、产出幻觉结果、或永久循环。

传统做法是在循环外层加 `max_iterations` 硬上限，但这只是兜底。在触发上限之前，agent 可能已经浪费了大量 token 和时间做无意义的迭代。

## 核心机制

**`no_tool` 是 agent 循环的行为纠正层，负责对模型行为偏差进行实时检测、分类、纠正或强制退出。**

### 检测维度与响应策略

```
模型输出
  ├── 空回复 ──→ empty_counter++ → 连续 3 次 → 强制退出
  ├── 含代码块但无 tool_use ──→ 注入纠正提示："请调用工具而非输出代码"
  ├── 含"任务完成"声明但无验证证据 ──→ 注入质疑提示："请提供验证结果"
  └── max_tokens 截断 / 流异常 ──→ 自动重试当前 turn
```

### 空回复连续退出

维护一个 `empty_response_counter`。每次模型返回无文本、无工具调用的空响应时递增。一旦连续达到阈值（通常为 3），判定模型已陷入不可恢复状态，强制终止循环并返回诊断信息。任何有效输出（文本或工具调用）都会将计数器归零。

### 伪执行拦截

检测模型输出中包含代码块（` ``` `）但未生成任何 `tool_use` block 的情况。这通常意味着模型在"描述"工具调用而非实际调用。拦截后注入纠正提示，要求模型使用工具 API 而非文本输出来执行操作。

### 虚假完成检测

当模型输出包含任务完成相关语义（如"已完成"、"任务结束"）但缺少验证性证据（如测试通过、文件变更确认）时，注入质疑提示，要求模型提供具体验证结果。这防止模型在未真正完成工作的情况下过早退出循环。

### 流异常自动重试

当 API 响应被 `max_tokens` 截断或流连接中断时，不将截断结果视为正常输出，而是自动重试当前 turn。重试时可选择性地压缩上下文以避免再次触发 token 上限。

## 项目对照

| 项目 | 实现方式 |
|---|---|
| GenericAgent | `do_no_tool` 方法：空回复计数 + 代码块检测 + 完成声明拦截 + 流异常重试，完整实现上述四个维度 |
| Claude Code | 类似机制通过 `ToolUseGuard` 检测无效工具调用并注入纠正提示 |
| Codex | 在 agent 循环中对 `finish_reason: length` 做自动 continue |

## 适用条件

**适用：**
- 任何包含 agent 循环的系统
- 模型行为不可控的场景（尤其是较弱模型或长上下文后期）
- 需要无人值守长时间运行的 agent

**不适用：**
- 单轮问答系统——无循环，无需纠正
- 模型输出被严格结构化约束（如 JSON mode + schema validation）——格式偏差已被 API 层处理

## 反模式

| 反模式 | 表现 | 修复 |
|---|---|---|
| 仅靠 max_iterations | 模型在前 N-1 轮做无意义输出，直到硬上限才停止 | 引入行为检测，尽早纠正或退出 |
| 过度纠正 | 对每个非工具输出都注入纠正提示，导致模型被反复打断 | 设置纠正频率上限，允许合理的文本输出 |
| 纠正提示堆积 | 连续注入多条纠正提示占满上下文 | 纠正提示使用固定槽位，新纠正替换旧纠正 |
| 忽略截断信号 | 将 max_tokens 截断视为正常完成 | 检查 finish_reason，截断时自动重试 |
