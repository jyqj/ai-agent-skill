# Checkpoint Hydration

> **Evidence Status** — grounded. Augment（AggregateCheckpointManager + Sharding）和 OpenCode（Session 持久化 + 压缩历史）的生产实现。

长时间运行的 Agent 随时可能被中断：用户关闭终端、进程崩溃、网络断开。如果没有检查点（Checkpoint），Agent 只能从头开始，浪费已完成的工作和用户的时间。Checkpoint Hydration 的思路很直接：定期保存状态快照，启动时从最近的快照恢复（Hydrate）。

## 检查点包含什么

一个有效的检查点需要捕获 Agent 的完整可恢复状态：

- **对话历史**：消息列表，包括系统提示和工具调用结果
- **进行中的工作**：未完成的工具调用、暂存的文件变更
- **任务进度**：当前处于多步任务的哪一步
- **记忆快照**：Agent 积累的上下文知识

关键是这些状态必须作为一个原子单元保存，否则恢复后可能出现对话历史指向了不存在的工具调用等不一致问题。

## 何时保存

自动保存策略需要平衡频率和开销。实践中常见的触发时机：

- **工具调用完成后**：这是最自然的一致性边界，工具已执行完毕，状态已稳定
- **用户回合结束时**：用户发完消息、Agent 回复完毕
- **时间间隔兜底**：超过 30 秒没有保存时强制触发，防止长时间工具调用期间完全没有检查点

三者取"或"：任一条件满足即保存，但用最小间隔防止过于频繁。

## 设计要点

**增量保存**：每次只序列化变化的部分，而非全量快照。对于动辄几十轮对话的 Agent，全量保存的 I/O 开销不可忽视。

**冲突处理**：恢复时外部世界可能已经变化（文件被编辑、API 状态已更新）。恢复逻辑需要检测这类冲突并提示用户，而非盲目重放。

**清理机制**：检查点会持续累积。合理的策略是保留最近 N 个，旧的定期压缩或删除。Augment 的做法是通过 Sharding 分散存储压力。

**通知用户**：保存和恢复动作应在 UI 上可见，让用户知道"可以安全中断"以及"已从检查点恢复"。

## 权衡

检查点的主要成本是存储和序列化。对于简单的对话 Agent，消息列表本身就是天然的检查点，额外机制可能是过度设计。但对于涉及文件编辑、多步工具调用的 Coding Agent，没有检查点意味着一次崩溃就可能丢失数十分钟的工作，这时候存储开销完全值得。

序列化的复杂度取决于状态中是否包含不可序列化的对象（如文件句柄、网络连接）。实践中的做法是只保存可重建的引用，恢复时重新建立连接。
