# Compaction

> **Evidence Status** — grounded. 来自 Claude Code（五层压缩）和 OpenCode（三层溢出管理）的生产实现，辅以通用上下文管理经验。

Agent 执行长任务时，上下文窗口终究会填满。粗暴截断会丢失关键决策历史，而完全不处理则导致模型推理质量下降。Compaction 的核心思想是：**分层、分成本地压缩历史，低成本手段先用尽，再动用 LLM 摘要**。

## 核心机制

上下文压缩是一组按成本递增排列的策略，而非单次操作。以 Claude Code 的五层设计为例：

```
1. Tool Result Budget   → 限制单个工具输出大小（裁剪，无成本）
2. Snip Compact        → 历史裁剪到最近 N 轮（裁剪，无成本）
3. Microcompact        → 工具输出智能压缩（规则，低成本）
4. Context Collapse    → 相关消息折叠（LLM，中成本）
5. Auto Compact        → 完整历史摘要（LLM，高成本）
```

逐层执行，只有前一层无法释放足够空间时才触发下一层。这比一次性全量摘要既省钱又可控。

### 阈值设计

```
有效窗口 = 总窗口 - 摘要输出预留(20K)
触发阈值 = 有效窗口 - 缓冲区(13K)
```

预留空间确保压缩本身的输出不会再次溢出。缓冲区给正在进行的工具调用留有余量。

### 熔断器

压缩失败（比如 LLM 返回的摘要仍然太长）时，连续失败 N 次后停止重试，避免无效 API 调用陷入死循环。

## 压缩策略选择

决定"压缩什么、保留什么"有几种互补的思路：

- **时间衰减**：越早的消息越可能被压缩，符合"最近消息更相关"的直觉。
- **重要性标记**：关键决策点标记为"不可压缩"，确保核心上下文不被丢弃。
- **类型过滤**：工具的原始输出最先压缩，用户输入次之，Agent 的推理过程最后。
- **增量摘要**：每 N 轮生成一次摘要而非最后一次性处理，避免压缩操作本身成为瓶颈。

## OpenCode 的三层溢出管理

OpenCode 采用了不同的分层策略，更强调可恢复性：

**第一层：Pruning（修剪）** -- 将旧的工具输出标记为"已压缩"并记录时间戳，但不删除原始数据。保护最近 2 个用户回合不被触碰，`skill` 类工具的输出永不压缩（因为它们承载上下文知识）。

**第二层：Compaction（压缩）** -- 用 LLM 按结构化模板（Goal / Instructions / Discoveries / Accomplished / Relevant files）生成摘要，然后自动回放。压缩失败时会重放用户消息（媒体内容除外）作为兜底。

**第三层：Truncation（截断）** -- 对单个工具输出施加 50KB / 2000 行的硬上限，超出部分写入文件并留下引用。这是最后的安全网。

## 权衡

压缩的根本矛盾是**信息保留 vs. 空间释放**。在实践中，这个矛盾在以下场景尤其尖锐：

- 当任务涉及多轮调试时，早期的错误信息和尝试路径可能在后续再次相关，过早压缩会导致 Agent 重复犯错。
- 摘要本身消耗 LLM 调用，在高频触发场景下（如自动化流水线中的大量工具调用），压缩成本可能占总成本的 10-20%。
- "什么是重要的"这个判断本身不可靠：规则策略会漏掉隐含重要性，LLM 策略则引入不确定性。

对于大多数场景，Claude Code 的"低成本优先"分层方案是合理的默认选择。只有在任务特别需要可恢复性（如 IDE 集成）时，才值得像 OpenCode 那样增加标记和回放机制的复杂度。
