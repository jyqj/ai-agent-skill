# Layered Memory

> **Evidence Status** — grounded. 来自 GenericAgent 的 L0-L4 五层架构的参考实现观察。

Agent 的记忆系统面临一个四难困境：记太多则检索变慢、上下文爆炸；记太少则经验无法复用；记错了则后续决策全被污染；记了但找不到则等于没记。Layered Memory 的回答是：**分层存储、行动验证、索引导航**。

## 五层架构

```
L0  Meta Rules     不可变公理，定义 Agent 的基本行为边界
L1  Insight Index  ≤30 行的索引，每次对话注入，是下层内容的导航入口
L2  Global Facts   环境事实（如项目结构、API 端点），按需读取
L3  Task Skills    SOP、脚本、操作模板，按需加载
L4  Session Archive  历史会话的压缩存档，仅在回溯时搜索
```

层级越高越稳定、越抽象、越小。L0 和 L1 始终在上下文中（合计不超过 1K tokens），L2-L4 按需加载。这种设计让 Agent 在绝大多数回合里只消耗极少的记忆 token，需要深入时再精确定位。

## 写入规则：No Execution, No Memory

只有经过行动验证的信息才能写入记忆。Agent 不能仅凭推理就往记忆里写"这个 API 的超时是 30 秒"——必须实际调用过、观察到结果后才能记录。这条规则是防止记忆污染（Memory Pollution）的核心屏障。

## L1 同步机制

L1 是整个系统的枢纽。当 L2 或 L3 新增内容时，需要决定是否同步到 L1：

- **高频场景**：直接在 L1 中写入 `key -> value` 形式的定位信息，下次可一步到达
- **低频场景**：仅在 L1 列出关键词，提示 Agent "这个主题在 L2/L3 中有详细信息"
- **通用避坑规律**：压缩为一句话写入 L1 的 `[RULES]` 区域

L1 有膨胀风险。如果不定期审视和精简，它会从"索引"退化为"又一个全量注入"，失去分层的意义。

## 权衡

分层记忆的收益随 Agent 的生命周期增长而显现。对于单次会话、任务类型单一的 Agent，五层架构是过度设计——一个简单的系统提示就够了。但对于需要跨会话积累经验、处理多种任务类型的长期运行 Agent，分层带来的精确召回和成本控制几乎不可替代。

主要的维护成本在层级同步：L2/L3 的变更必须正确反映到 L1，否则信息虽然存在却无法被发现。这需要 Agent 在写入时承担额外的"更新索引"步骤，增加了每次写入的复杂度。
