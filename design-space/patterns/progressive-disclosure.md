# Progressive Disclosure

> **Evidence Status** — synthesized. Skill 系统、工具目录、多产品 Agent 平台等场景中的常见实践，尚无单一权威实现。

当 Agent 可用的工具、技能、文档和规则多到一定程度时，全量注入上下文既浪费 token 又引入噪声：模型在几十个工具描述中找到对的那个，比在三五个中选择要困难得多。Progressive Disclosure（渐进披露）的思路是：**按需分层，只在需要时才展开细节**。

## 信息分层

| 层 | 内容 | 何时加载 | 典型大小 |
|---|---|---|---|
| L0 Index | 名称 + 一句话描述 | 始终在上下文中 | 几百 tokens |
| L1 Guide | 使用说明、约束、示例 | Agent 选中某项后加载 | 1-2K tokens |
| L2 Reference | 完整文档、代码、案例 | 明确需要深入时 | 数千 tokens |
| L3 Raw Data | 原始日志、全文、数据库 | 调试或验证时 | 不限 |

L0 始终可见，让 Agent 知道"有哪些能力可用"。当 Agent 判断某项能力与当前任务相关时，加载 L1 获取使用指引。只有在需要精确参考时才深入到 L2 和 L3。

## 触发机制

从 L0 到更深层级的触发可以是显式的（Agent 主动调用"读取文档"工具），也可以是隐式的（系统根据 Agent 选择的工具自动注入该工具的详细说明）。隐式触发对 Agent 更友好，但需要预定义好触发规则；显式触发更灵活，但依赖 Agent 自身的判断力。

## 适用场景

这个模式在以下场景收益最大：Skill 系统（几十个技能，每个都有详细 SOP）、工具目录（工具数量超过模型高效处理的阈值）、大型项目文档（Agent 不可能一次读完所有文档）、企业 SOP（流程多且相互独立）。

对于工具和知识总量较少的 Agent，分层带来的间接寻址成本可能超过收益，直接全量注入反而更简单可靠。
