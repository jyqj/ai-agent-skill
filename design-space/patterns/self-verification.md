# Self-Verification

> **Evidence Status** — synthesized. 从 Coding Agent、Workflow Agent、Browser Agent、Ops Agent 等多个场景中提炼的共性模式。

Agent 有一个普遍的倾向：过早宣布完成。代码改了但没测，API 调了但状态没变，浏览器点了按钮但页面未提交，研究报告有结论但证据不支持。Self-Verification 要求 Agent **在声称完成之前，像用户、像系统、像审计员一样验证自己的工作**。

## 四层验证

验证不是只跑一遍单元测试。按结果的性质不同，需要不同层次的验证：

**结构验证（Structural）** -- 输出的格式和 schema 是否正确。这是最基础的一层，成本最低，能拦截明显的格式错误。

**执行验证（Execution）** -- 工具是否真的被执行了。Agent 有时会"幻想"自己已经执行了某个操作。检查工具调用记录可以捕获这类问题。

**效果验证（Effect）** -- 外部世界是否真的发生了变化。这是最关键的一层：文件是否被写入、数据库记录是否更新、页面是否跳转。做法通常是回读（Read-back）——写入后立即读取，确认结果符合预期。对于最终一致性（Eventual Consistency）的系统，可能需要等待后再回读。

**断言验证（Claim）** -- 最终输出中的事实主张是否都有证据支撑。如果 Agent 说"这个 bug 的原因是 X"，那么对话历史中是否有支持 X 的证据？存在冲突来源时是否做了说明？

## 验证清单

在实践中，一个简洁的完成清单比复杂的验证框架更实用：

- 相关测试是否通过
- diff 中是否有与任务无关的改动
- 效果是否经过回读确认
- 剩余风险是否披露

Agent 声称完成时，逐项检查。任一项未通过则将失败原因注入上下文，让 Agent 继续工作。

## 权衡

验证的成本是额外的执行时间和工具调用。对于简单任务（如回答一个事实性问题），四层验证全开显然过度。合理的做法是根据任务类型选择验证层次：纯文本回答只做断言验证，代码修改做执行+效果验证，涉及外部系统的操作则四层全开。

另一个风险是验证循环：验证失败 -> 修复 -> 再验证 -> 又失败。需要设置最大重试次数，超过后向用户报告而非无限循环。
