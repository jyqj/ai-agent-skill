# Summary Forced Extraction

> **Evidence Status** — production-validated. 来自 GenericAgent 的 summary 强制抽取机制，在长对话 agent 中稳定运行，有效控制上下文膨胀。

## 问题

长对话 agent 的上下文窗口是有限资源。每一轮 tool 调用和模型回复都在消耗上下文空间，而多数中间步骤的细节在后续轮次中并不需要，真正重要的是"这一步做了什么、结果如何"的摘要。

如果依赖模型"自觉"输出摘要，实际表现极不稳定：模型可能输出冗长分析、跳过摘要、或将摘要淹没在大段文本中。上下文很快被无价值的历史细节塞满，导致关键信息被截断或丢失。

## 核心机制

**强制要求模型每轮输出结构化摘要标签，未输出则自动降级生成 + 注入惩罚提示。**

### 抽取流程

```
模型输出
  ├── 包含 <summary>...</summary> ──→ 提取，长度校验（≤30字）
  │     ├── 通过 ──→ 存入摘要队列
  │     └── 超长 ──→ 截断到 30 字 + 提示下次精简
  └── 不包含 <summary> ──→ 降级：工具名 + 参数自动摘要
        └── 注入惩罚提示："请在回复中包含 <summary> 标签"
```

### 摘要标签协议

在系统提示中约定：模型每轮回复必须包含一个 `<summary>` 标签，内容为不超过 30 字的本轮操作摘要。例如：

```
<summary>读取 config.yaml，发现端口配置错误</summary>
```

### 降级自动摘要

当模型未输出 `<summary>` 标签时，系统自动从该轮的工具调用中提取摘要：

- 有工具调用：`"调用 {tool_name}({关键参数})"` — 如 `"调用 read_file(config.yaml)"`
- 无工具调用：`"文本回复（无工具调用）"`

降级摘要虽然信息密度低于模型主动生成的摘要，但保证了摘要流不中断。

### 滑动窗口历史

摘要按时间顺序累积为 `<history>` 块，作为滑动窗口注入后续 turn 的上下文中：

```
<history>
[1] 读取 config.yaml，发现端口配置错误
[2] 修改端口为 8080，写入文件
[3] 运行测试，3 个用例通过
...
</history>
```

窗口大小可配置（如保留最近 20 轮摘要）。超出窗口的旧摘要被丢弃，但可选择性保留标记为"关键"的条目。

### Anchor Prompt 注入结构

每轮 prompt 的上下文组装顺序：

```
system prompt
  └── earlier_context    ← 被压缩/截断的早期原始对话
  └── <history>          ← 摘要滑动窗口
  └── <key_info>         ← 用户标记或系统识别的关键信息
  └── current turn       ← 当前轮的用户输入 + 工具结果
```

这个结构确保：即使 `earlier_context` 被大幅压缩，`<history>` 和 `<key_info>` 仍能提供足够的任务连续性。

## 项目对照

| 项目 | 实现方式 |
|---|---|
| GenericAgent | 完整实现：`<summary>` 标签抽取 + 降级自动摘要 + 惩罚提示 + `<history>` 滑动窗口 + anchor prompt 注入 |
| Claude Code | 通过 compaction 机制在上下文超限时做摘要，但非每轮强制抽取 |

## 与其他 plane/pattern 的关系

| 相关模式 | 关系 |
|---|---|
| `compaction.md` | compaction 是被动触发（上下文超限时压缩），本模式是主动预防（每轮强制抽取摘要减缓上下文增长） |
| `behavior-correction-layer.md` | 摘要未输出时的惩罚提示注入是 behavior correction 的具体应用 |
| `context-compression-safety.md` | 压缩上下文时必须保留 `<history>` 和 `<key_info>` 块，本模式为其提供结构化输入 |
| `token-budget-controller.md` | token budget 决定可用空间，本模式决定如何在有限空间内保留最大信息密度 |
| `layered-memory.md` | 摘要历史是 working memory 层的一种实现；关键摘要可晋升为 long-term memory |

## 适用条件

**适用：**
- 长对话 agent（预期 10+ 轮交互）
- 上下文窗口受限且任务跨度大的场景
- 需要任务连续性但无法保留完整历史的系统

**不适用：**
- 短对话 agent（3-5 轮内完成）——摘要开销大于收益
- 上下文窗口极大且任务简单——直接保留原始对话更简单
- 模型不支持结构化标签输出——降级路径可能不可靠

## 反模式

| 反模式 | 表现 | 修复 |
|---|---|---|
| 无降级路径 | 模型不输出摘要时历史断链，后续轮次丢失上下文 | 必须实现自动降级摘要 |
| 摘要过长 | 模型输出 100+ 字"摘要"，摘要本身消耗大量上下文 | 硬性长度上限 + 超长截断 |
| 仅存摘要丢原文 | 过早丢弃原始对话，关键细节无法回溯 | earlier_context 保留近 N 轮原文，摘要覆盖更早的部分 |
| 惩罚提示堆积 | 连续多轮未输出摘要，惩罚提示不断累积 | 惩罚提示使用固定槽位，不叠加 |
