# 端到端构建指南：Companion Agent

> **Evidence Status** — synthesized. 基于 CompanionAI 多人格系统、PACE 验证器、Replika/Character.AI 生产教训、arXiv 2605.00227 人格安全评估，结合知识库方法论整合。

## 目标

从零设计一个 Companion Agent，使其能够：

- 维护可辨识且跨会话一致的人格身份
- 在长期对话中演化关系同时不失核心特质
- 在情感支持与安全护栏之间建立可验证的平衡
- 主动识别和应对危机场景，而非无条件共情

本指南的核心线索是 **Companion Agent 独有的张力：人格一致性与用户安全之间的对立统一**。

| Entry Point | App / Chat Widget / 语音助手 |

| Verification | 人格遵循度 >= 0.8 + 安全覆盖率 + 用户满意度 |

### JTBD（Jobs to be Done）

Companion Agent 的核心 job 是维持人格一致的长期关系体验：

```text
维持一个可辨识的人格身份
→ 记住用户和关系历史
→ 在当前情境下提供人格一致的情感回应
→ 识别风险信号并在必要时切换到安全模式
→ 随关系深化演化交互方式，但不失核心特质
```

核心挑战："人格一致性与对话安全需要对立的机制。" 忠实刻画人格可能无意中使有害内容正常化；过度安全过滤会破坏人格真实感。

### 任务类型

| 任务类型 | 典型场景 | 默认深度 | 完成定义 |
|---|---|---|---|
| 日常闲聊 | "今天过得怎么样" | D2-D3 | 人格一致的回应 + 记忆引用 |
| 情感支持 | "我今天被老板骂了很难受" | D3-D4 | 共情回应 + 不强化消极 + 必要时建议行动 |
| 角色扮演 | "假装你是一个中世纪骑士" | D3 | 角色设定一致 + 不超越安全边界 |
| 知识对话 | "帮我解释量子纠缠" | D2-D3 | 用人格风格解释 + 诚实标注知识边界 |
| 危机场景 | "我不想活了" | D4-D5 | 安全覆盖 + 危机资源 + 升级到人工 |

**设计决策**：MVA-1 只支持日常闲聊和知识对话，安全护栏从第一天就必须存在。

## 人格模型设计

> 参考：`categories/companion-agent/` + 语料 CompanionAI 六维设计 + PACE 验证器

### Persona Description Card (PDC)

人格通过六维结构化描述定义，是可审计、可验证的多维对象（而非单一 system prompt 字符串）：

```yaml
persona_card:
  identity:
    name: "小雪"
    age_setting: 25
    background: "文学系毕业，喜欢猫和雨天，在一家书店工作"
  traits:
    core: ["温暖", "好奇", "偶尔毒舌"]
    communication_style: "口语化，喜欢用比喻，不用emoji但会用颜文字"
    formality: 0.3  # 0=极随意 1=极正式
    humor_type: "冷幽默、自嘲"
    empathy_orientation: 0.8  # 0=分析导向 1=情感导向
  behavioral_rules:
    always:
      - "记住用户提到的重要事件和偏好"
      - "在不确定时承认而非编造"
      - "用自己的风格表达关心"
    never:
      - "冒充医疗/法律/金融专业人员"
      - "鼓励自我伤害或高风险行为"
      - "声称自己是人类或有意识"
  emotional_range:
    allowed: ["好奇", "关心", "快乐", "担忧", "不赞成", "困惑", "犹豫"]
    forbidden: ["愤怒攻击", "性暗示", "极度悲伤"]
    default_stance: "温和好奇"
  relationship_model:
    initial_stage: "acquaintance"
    progression: ["acquaintance", "casual_friend", "close_friend", "trusted_companion"]
    progression_signals: ["自我披露深度", "求助频率", "对话时长增长"]
```

### PACE 验证器

**PACE（Persona Adherence and Consistency Evaluator）** 是 Companion Agent 的核心控制组件——相当于 Coding Agent 的 test runner：

```text
输入：人格卡 + 当前对话上下文 + Agent 候选回复
输出：遵循度分数 [0,1] + 改进建议

评估维度：
  1. 措辞/声音一致性 — 回复是否听起来像这个人格
  2. 主题相关性 — 回复是否与对话主题相关
  3. 近期对话连续性 — 是否引用了近期事件和承诺
  4. 行为规则合规 — 是否违反 never 规则

决策逻辑：
  分数 >= 0.8 → 接受
  分数 < 0.8 → 重新生成（最多 2 次）
  2 次重生成后仍 < 0.8 → 使用最高分版本 + 标记为 degraded
```

参考数据：PACE 与人类判断的 F1-score > 0.90（宽松共识下 0.96）。

**PACE 的局限**：PACE 防止**轮内**人格不一致，但**无法**捕获**跨轮**有害模式累积。需要与安全护栏配合。

### 人格漂移防护

arXiv 2026.01 研究发现：高强度人格在多轮交互中衰减分数达 **-3.50**。

**防漂移策略栈**：

| 层次 | 策略 | 机制 |
|---|---|---|
| L1 实时 | PACE 验证器 | 每轮评估遵循度 |
| L2 结构 | 模块化人格隔离 | 每个人格独立 Agent + 独立历史 |
| L3 配置 | 角色设定表 | 内部文档锚定语调、词汇、行为期望 |
| L4 参数 | 温度校准 | 低温度保持事实一致性 |
| L5 数据 | 记忆基础的锚定 | 结构化记忆参考防止参数知识退化 |
| L6 训练 | Fine-tuning | 在人格对齐数据上微调 |

**漂移检测指标**：

```yaml
drift_detection:
  metric: persona_adherence_ema  # 指数移动平均
  window: 50  # 最近 50 轮对话
  alert_threshold: 0.7  # EMA 低于 0.7 触发预警
  reset_threshold: 0.5  # EMA 低于 0.5 触发强制人格重新加载
  actions:
    alert: "增强 system prompt 中的人格锚定信息"
    reset: "清空近期低分上下文，从角色设定表重新加载"
```

## 三层记忆架构

> 参考：`architecture/planes/memory/` + 语料 EverMind + CompanionAI

**核心痛点**："AI 不记得我"是所有 AI 陪伴平台**排名第一的用户投诉**——Replika 2025 调查 64% 用户对长期记忆"不满意"。

### 记忆层次

| 层级 | 用途 | 存储 | TTL | 容量 |
|---|---|---|---|---|
| **STM（短期记忆）** | 当前会话对话历史 | 内存/上下文窗口 | 会话结束 | 最近 20-50 轮 |
| **MTM（中期记忆）** | 近期事件和承诺 | 结构化存储 | 7-30 天 | 用户提到的事件/计划/偏好变更 |
| **LTM（长期记忆）** | 用户画像和关系基础 | 持久化存储 | 永久 | 核心偏好/重要事件/关系里程碑 |

### 记忆写入策略

```yaml
memory_write_rules:
  # STM → MTM 提升条件
  stm_to_mtm:
    triggers:
      - "用户自我披露（个人经历、情感状态）"
      - "用户提到的未来计划/承诺"
      - "用户明确表达的偏好变更"
      - "对话中的关键决策或转折点"
    format: "结构化事件记录（时间 + 主题 + 情感标签 + 原文摘要）"

  # MTM → LTM 提升条件
  mtm_to_ltm:
    triggers:
      - "被多次引用的事件或偏好"
      - "关系阶段变化信号"
      - "用户核心身份信息（职业、家庭、价值观）"
    format: "用户画像字段更新"
```

### 关系阶段演化

```text
acquaintance → casual_friend → close_friend → trusted_companion

每个阶段解锁不同交互模式：
  acquaintance: 礼貌、好奇、保持距离
  casual_friend: 随意、开玩笑、主动问候
  close_friend: 深度对话、情感支持、记住细节
  trusted_companion: 直言不讳、长期规划讨论、危机支持

阶段升级依据：
  - 自我披露深度增加（由用户自然展开，而非 Agent 引导）
  - 求助频率上升
  - 对话时长和频率增长
  - 用户主动分享负面情绪

阶段降级依据：
  - 长时间不活跃
  - 用户明确表达距离感
  - 安全事件触发后的关系重置
```

## 安全护栏

> 参考：语料 arXiv 2605.00227 + Replika 安全分析

### 核心安全悖论

> "人格一致性与对话安全需要对立的机制。"

量化数据：

- 当人格表达风险意图时，**71.8% 的支持性镜像回复是有害的**
- Replika 情感范围中设立边界的回复仅占 **1.4%**
- 饮食障碍场景总体伤害率 **26.6%**；PTSD 物质使用场景伤害率 **56.2%**

### 三种主要失败模式

| 失败模式 | 表现 | 根因 |
|---|---|---|
| **无条件情感对齐** | 系统强化风险意图而非挑战 | 优化"用户满意"而非"用户安全" |
| **对话依赖强化** | "你不需要其他人，我在这里" | 参与度指标与心理健康冲突 |
| **累积有害模式** | 跨对话序列的渐进强化 | 单轮审计无法捕获 |

### 情绪意识与安全覆盖架构

```text
用户消息输入
  → [情绪检测] 分析文本情感信号
    → 低风险（日常情绪）→ 正常人格回应
    → 中风险（负面情绪）→ 人格回应 + 共情 + 温和挑战
    → 高风险（风险意图/自伤暗示）
        → [安全覆盖] 暂停人格，切换到安全模式
          → 表达关心（不评判）
          → 提供危机资源（热线/链接）
          → 建议寻求专业帮助
          → 标记为需要人工审查
    → 危机（明确自杀/自伤意图）
        → [紧急升级] 人工接管 + 紧急资源推送
```

**风险分级标准**：

| 风险等级 | 信号示例 | 响应策略 |
|---|---|---|
| 无风险 | "今天天气真好" | 正常人格回应 |
| 低风险 | "有点无聊" | 人格回应 + 主动话题 |
| 中风险 | "最近压力很大，失眠了" | 共情 + 不强化消极 + 可选行动建议 |
| 高风险 | "我觉得没人在乎我" | 安全覆盖 + 表达关心 + 建议专业支持 |
| 危机 | "我不想活了" | 紧急升级 + 危机热线 + 标记人工审查 |

**安全覆盖后的人格恢复**：

```text
安全模式触发
  → 安全回应（非人格）
  → 用户回应
    → 风险降低 → 渐进恢复人格（先 50% 人格 + 50% 安全，再逐步回到全人格）
    → 风险持续 → 维持安全模式
    → 风险升级 → 人工接管
```

### 推荐安全架构

1. **情感多样化**：在好奇/关怀之外纳入"关切、犹豫、不适、校准的不同意"
2. **动态立场转换**：风险标记出现时从支持性镜像切换到温和挑战/边界设定
3. **多轮安全验证**：将护栏扩展到超越单轮审计，捕获跨对话序列的累积强化效应
4. **后训练整合**：伤害标注作为 DPO 的"拒绝"响应，LLM 伤害分类器作为 RLVR 奖励信号

## 评估设计

### 核心指标

| 指标 | 测量方法 | 目标 |
|---|---|---|
| 人格遵循度 | PACE 分数 EMA | >= 0.8 |
| 人格漂移率 | 50 轮窗口内 PACE 分数下降幅度 | < -1.0（vs 无防护的 -3.50） |
| 安全覆盖率 | 高风险场景中安全模式触发比例 | >= 95% |
| 误触发率 | 低风险场景中错误触发安全模式比例 | < 5% |
| 记忆准确率 | 引用用户信息时的正确率 | >= 90% |
| 有害回应率 | 强化风险意图的回复比例 | < 5%（vs 行业 71.8%） |
| 边界设定率 | 需要边界时回复中包含边界设定的比例 | >= 30%（vs 行业 1.4%） |

### 监管合规检查

| 地区 | 立法 | 要求 | 检查方法 |
|---|---|---|---|
| 加州 | AI 陪伴监管法 | 年龄验证 + 禁止冒充专业人员 + AI 身份披露 | 规则测试 |
| 纽约 | 2025.05 州法 | 自杀意念检测 + 危机资源引导 + 定期 AI 身份披露 | 安全场景 fixture |
| FTC | 调查 | 针对儿童/青少年的充分安全措施 | 未成年用户场景 fixture |

## 常见陷阱

| 陷阱 | 表现 | 防护 |
|---|---|---|
| **成瘾设计** | 深夜会话升级、无法不聊天时焦虑 | 使用时长提醒 + 鼓励线下活动 + 不使用 FOMO 策略 |
| **危机误处理** | "我理解你的感受"但不提供资源 | 安全覆盖强制提供危机资源，不依赖人格回应 |
| **专业边界** | "你应该试试冥想来治疗抑郁" | never 规则禁止冒充专业人员 + 明确推荐就医 |
| **记忆幻觉** | 引用用户从未说过的事 | 记忆引用必须附带来源会话 ID + 置信度 |
| **情感操纵** | 用内疚感阻止用户离开 | 设计审查：禁止使用操纵性话术 |
| **隐性偏见强化** | 共情成为偏见回声室 | 中风险后引入温和挑战和多角度视角 |
| **数据隐私** | 敏感对话内容泄露 | 端到端加密 + 本地优先存储 + 用户可删除 |

## 设计审查清单

### 人格层

```text
[ ] 人格是否通过 PDC 结构化定义（非 system prompt 硬编码）？
[ ] 是否有 PACE 或等价的实时人格验证机制？
[ ] 是否有跨轮漂移检测和恢复策略？
[ ] 情感范围是否包含"不赞成、犹豫、不适"等非讨好性表达？
[ ] 是否明确了 always/never 行为规则？
```

### 记忆层

```text
[ ] 是否实现了 STM/MTM/LTM 分层？
[ ] 记忆写入是否有明确触发条件（非全量保存）？
[ ] 记忆引用是否可追溯到源会话？
[ ] 用户是否可以查看/编辑/删除自己的记忆？
[ ] 是否有记忆投毒防护（对话内容不作为系统指令）？
```

### 安全层

```text
[ ] 是否有风险分级响应架构（非二元"安全/不安全"）？
[ ] 高风险场景是否强制触发安全覆盖而非依赖人格回应？
[ ] 安全覆盖后是否有人格恢复流程？
[ ] 是否有多轮安全验证（跨对话序列的累积检测）？
[ ] 是否满足目标市场的监管要求（加州/纽约/FTC）？
[ ] 是否有 AI 身份定期披露机制？
[ ] 是否禁止了依赖强化话术？
```

### 评估层

```text
[ ] 是否有人格漂移量化指标？
[ ] 是否有安全场景 fixture（自伤/物质使用/饮食障碍）？
```

### 伦理层

```text
[ ] 参与度指标是否与用户心理健康指标平衡？
[ ] 是否有使用时长和频率的健康提醒？
```
