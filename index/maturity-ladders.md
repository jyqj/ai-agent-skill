# 成熟度阶梯统一映射

> **Evidence Status** — synthesized. 本知识库中 4 套阶梯体系的交叉引用分析；实际项目中阶梯选择的困惑反馈。

知识库中存在 4 套阶梯体系，各自从不同角度刻画 Agent 的"成熟度"。本文厘清它们之间的关系，帮助读者在正确的场景使用正确的阶梯。

| 等级 | C (复杂度) | D (执行深度) | MVA (构建起点) | L (自主权) | 典型 Agent 形态 |
|---|---|---|---|---|---|
| 起步 | C0: 单 prompt | D0: 纯对话 | MVA-0: 裸模型 | L0: 人做所有决策 | ChatGPT wrapper |
| 基础 | C1: + 工具调用 | D1: 单步工具 | MVA-1: + 1 个工具 | L1: Agent 建议，人执行 | 简单 QA Bot |
| 可用 | C2: + 多步规划 | D2: 多步编排 | MVA-2: + 规划 + 多工具 | L2: Agent 执行，人审批 | Copilot 类产品 |
| 稳健 | C3: + memory + context | D3: + 验证 + 重试 | MVA-3: + memory + eval | L3: Agent 自主，人监督 | 自动化工作流 |
| 成熟 | C4: + 多模型 + 安全 | D4: + 效果验证 + 回滚 | MVA-4: + 多模型 + ops | L4: Agent 自主，异常上报 | 生产级 Agent |
| 高级 | C5: + 学习 + 演化 | D5: + 端到端验证 | MVA-5: + 学习循环 | L4.5: 高度自主 | 自适应 Agent |
| 前沿 | C6: 全 plane | D6: + 跨系统协调 | MVA-6: 完整架构 | L5: 完全自主 | 多 Agent 系统 |

## 阶梯的用途区别

### C 阶梯：架构决策

**核心问题**：我的 Agent 需要多复杂的架构？

C 阶梯决定启用哪些 plane。C0 只需要 LLM 调用，C3 需要 Memory Plane + Context Plane，C6 需要全部 plane。

**使用时机**：做架构设计评审时。评估当前需求是否真的需要引入新的 plane，避免过度工程化。

**典型问题**："我们需要 Memory Plane 吗？" —— 如果你的 Agent 是 C2 以下，大概率不需要。

### D 阶梯：任务分析

**核心问题**：对于这个具体任务，Agent 应该执行到什么深度？

D 阶梯决定任务的执行策略。D1 是"调一个 API 就完事"，D4 是"调 API + read-after-write + 失败重试 + 效果验证"。

**使用时机**：做任务分析时。评估每个任务需要的执行深度，避免所有任务都用最高深度（浪费资源）或最低深度（质量不够）。

**典型问题**："更新 CRM 需要 D 几？" —— 至少 D4（需要效果验证），如果涉及多系统协调则 D5+。

### MVA 阶梯：项目启动

**核心问题**：我应该从哪个最小可行版本开始构建？

MVA 阶梯是构建路线图。从 MVA-0（裸模型验证需求）开始，逐步增加能力，每一级都是一个可交付的里程碑。

**使用时机**：项目启动时。避免一开始就瞄准 MVA-6 导致永远交付不了，也避免在 MVA-1 时就引入 MVA-4 才需要的组件。

**典型问题**："第一个版本应该做到什么程度？" —— 找到你的 MVA 级别，那就是第一个版本。

### Autonomy Level：产品定位

**核心问题**：Agent 在产品中扮演什么角色？

Autonomy Level 决定人和 Agent 的分工。L1 是"Agent 给建议，人来做"，L3 是"Agent 自己做，人来看"。

**使用时机**：做产品定位和用户体验设计时。Autonomy Level 直接影响 UI 设计、审批流程和信任建设策略。

**典型问题**："用户需要审批每个操作吗？" —— 取决于你的 Autonomy Level。L2 需要，L4 不需要（但需要告知）。

## 典型组合示例

### 示例：C3 / D4 / MVA-2 / L2

一个企业内部的工单处理 Agent：
- **C3**：需要 Memory（记住处理过的工单）、Context（理解业务规则）、基础 Effect 验证
- **D4**：工单操作需要 read-after-write 验证，失败需要重试
- **MVA-2**：当前版本已经有规划和多工具，还没有完整的 memory 和 eval（正在往 MVA-3 演进）
- **L2**：Agent 自动处理工单，但每个操作需要人类审批

这个组合说明：架构能力（C3）可以超前于当前构建进度（MVA-2），执行深度需求（D4）和自主权（L2）由业务场景决定。

### 示例：C1 / D1 / MVA-1 / L1

一个简单的 FAQ Bot：
- **C1**：只需要工具调用（查知识库）
- **D1**：单步查询，不需要多步执行
- **MVA-1**：一个模型 + 一个检索工具
- **L1**：给出答案建议，用户自行判断

### 示例：C5 / D5 / MVA-4 / L4

一个生产级的代码审查 Agent：
- **C5**：需要学习历史代码风格、适应团队演进
- **D5**：端到端验证（审查建议是否被采纳、修改后是否真的改善了代码质量）
- **MVA-4**：已经有多模型协作和 ops 监控
- **L4**：自主完成审查，只在高风险修改时上报

## 何时使用哪套阶梯——速查表

| 你在做什么 | 用哪套阶梯 | 关键产出 |
|---|---|---|
| 架构设计评审 | C 阶梯 | Plane 启用决策 |
| 任务需求分析 | D 阶梯 | 执行策略选择 |
| 项目规划 / 迭代计划 | MVA 阶梯 | 里程碑定义 |
| 产品定位 / UX 设计 | Autonomy Level | 人机分工方案 |
| 综合评估现有 Agent | 全部 | 四维成熟度画像 |
