# Evidence Status 升降级标准

> **Evidence Status** — theoretical. 标准本身是首次定义，尚未实际运行过完整的升降级流程。

## 1. 概述

Evidence Status 7 级体系（从高到低）：

```text
production-validated > prototype-validated > grounded > synthesized > mixed > theoretical > speculative
```

本文档定义每次升降级所需的**最低证据门槛**。证据来源包括：项目分析（`projects/`）、用户反馈（`meta/feedback/reports/`）、外部文献、代码审计。

## 1.1 使用态度：不要把生产验证当作唯一价值来源

本 skill 的定位是 Agent 开发工具包与架构参照物，不是只收录生产定论的标准库。因此：

- `theoretical` / `synthesized` 可以用于启发、草案、review checklist；
- `grounded` 可以作为项目观察和设计参考，但仍需检查上下文差异；
- fixture、starter-kit、synthetic trace 属于 **simulated-eval 证据**，能证明不变量和机制可表达，不证明真实 Agent 能力；
- `prototype-validated` / `production-validated` 是更强证据，但不是所有内容进入知识库的门槛。

写文档时应明确“这个内容能支持什么决策”，而不是简单把未生产验证的内容视为低价值。

## 2. 升级标准

### speculative → theoretical

| 维度 | 要求 |
|---|---|
| 逻辑一致性 | 内容有清晰的逻辑推导链，无自相矛盾 |
| 学术/行业支撑 | 至少引用 1 个相关的学术概念、行业实践报告或成熟领域的类比 |
| 边界说明 | 明确标注适用范围和已知局限 |
| 审核 | 维护者确认推导合理 |

### theoretical → synthesized

| 维度 | 要求 |
|---|---|
| 跨源抽象 | 从 2+ 个独立来源（项目、文献、框架）提炼出共性 |
| 结构化表达 | 内容已按知识库模板结构化，有明确的概念定义、对比和引用关系 |
| 内部一致性 | 与知识库其他相关文件无矛盾，或已标注差异点 |
| 审核 | 维护者确认抽象质量 |

### synthesized → grounded

| 维度 | 要求 |
|---|---|
| 代码/文档证据 | 至少 1 个参考项目（`projects/` 中有对应分析）的源码或文档直接支撑该 claim |
| 具体映射 | 能指出"哪个项目的哪个组件体现了这个模式/原则" |
| 反馈佐证（可选） | 1+ 份用户反馈确认在实践中观察到类似结构 |
| 审核 | 维护者确认证据链完整 |

### grounded → prototype-validated

| 维度 | 要求 |
|---|---|
| 原型实践 | 至少 1 个已运行的原型（prototype / MVP / 内部工具）按照该内容做了设计决策 |
| 效果数据 | 有可量化或可描述的效果数据（成功率、失败模式、用户反馈） |
| 反馈报告 | 至少 1 份结构化反馈报告（`meta/feedback/reports/`）确认内容在原型中被使用且有效 |
| 局限记录 | 记录了在原型中发现的局限或需要调整的部分 |
| 审核 | 维护者确认原型证据真实且相关 |

### prototype-validated → production-validated

| 维度 | 要求 |
|---|---|
| 生产实践 | 至少 2 个独立的生产系统（不同团队或不同项目）按照该内容做了设计决策 |
| 生产数据 | 有生产环境的效果数据：成功率、故障率、恢复时间、成本等 |
| 反馈报告 | 至少 2 份来自不同项目的结构化反馈报告确认内容在生产中有效 |
| 时间跨度 | 生产运行时间 >= 1 个月（排除短期偶然成功） |
| 边界更新 | 基于生产经验更新了适用范围和已知局限 |
| 审核 | 维护者确认生产证据真实、独立且时间充分 |

## 3. 降级标准

降级不需要等待季度分析，发现即可触发。

### 降级触发条件

| 触发条件 | 说明 | 降级幅度 |
|---|---|---|
| **误导反馈** | 2+ 份独立反馈报告指出内容在实践中产生误导 | 至少降一级，视严重程度可降多级 |
| **过时失效** | 依赖的外部工具/API/框架已发生重大变更，内容不再适用 | 降至 theoretical 或标注 `[outdated]` |
| **矛盾证据** | 新的项目分析或生产数据直接反驳核心 claim | 降至与剩余证据匹配的级别 |
| **证据撤回** | 原本支撑升级的项目被发现数据不准确或不可复现 | 回退到升级前的级别 |
| **长期无验证** | grounded 及以上的内容超过 12 个月无新的验证证据 | 标注 `[needs-revalidation]`，不自动降级 |

### 降级流程

```text
1. 发现降级触发条件
2. 维护者评估证据，确认需要降级
3. 更新目标文件的 Evidence Status 标注
4. 在文件中添加降级说明（为什么降级、什么时候可以重新升级）
5. 记录到 meta/feedback/changelog.md
6. 如果影响到导航或推荐，同步更新相关 index 文件
```

## 4. 特殊情况处理

### mixed 状态

当一个文件中部分内容有高级别证据、部分内容仅有低级别证据时：
- 文件级别标注为 `mixed`
- 在文件内部用行内标注区分不同部分的证据级别
- 格式：`<!-- evidence: grounded -->` 或 `> 以下部分为 theoretical，尚未经过实践验证。`

### 批量升降级

当一个反馈报告覆盖多个文件时：
- 每个文件独立评估，不做批量自动变更
- 但可以在 changelog 中统一记录为一次事件

### 争议处理

当不同反馈对同一文件给出矛盾评价时：
- 记录所有反馈，不偏采任何一方
- 分析差异原因（场景不同？品类不同？规模不同？）
- 如果无法判断，保持当前级别并在文件中标注争议点
