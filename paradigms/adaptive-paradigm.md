# 自适应范式——运行时参数动态调整

> **Evidence Status** — mixed. 部分来自 Adaptive Thinking（Claude Opus 4.6 内置）、RouteLLM（ICLR 2025）、MAGE 元强化学习框架和生产可观测性数据。

## 1. 为什么需要单独讨论

当前范式层假设"在启动时或切换点选择范式"（`paradigm-routing.md`），但生产系统需要在**范式内部**进行参数自适应：同一个 ReAct 循环中，根据任务进展动态调整 step limit、model tier 和 retry budget。

```text
paradigm-routing.md 讨论的是：
  ReAct → Plan-Execute → Reflection  （离散切换）

本文讨论的是：
  同一个 ReAct 循环内：
    step_limit: 10 → 15（因为进展慢于预期）
    model_tier: gpt-4o-mini → gpt-4o（因为子任务变复杂）
    retry_budget: 3 → 1（因为预算即将耗尽）
```

这种"范式内连续调参"在生产中普遍存在，但缺少系统化的设计框架。

## 2. 三层自适应

### 2.1 模型层自适应（Model-Level）

模型自身动态评估 prompt 复杂度并分配计算资源。

**Adaptive Thinking**：复杂问题花更多时间"思考"，简单问题快速回答。这是模型内置能力，不需要 Harness 干预。

```text
Harness 需要知道的：
  - 模型的思考时间是可变的 → 超时设置不能太紧
  - 模型的 token 消耗是可变的 → 成本预估需要留余量
  - 对于同一 prompt，不同次调用的成本可能不同

Harness 不需要做的：
  - 不需要控制模型"思考多久"
  - 不需要判断 prompt 复杂度（模型自己会判断）
```

### 2.2 路由层自适应（Routing-Level）

根据任务特征将请求路由到不同模型。

**RouteLLM**（ICLR 2025）：仅 26% 请求发到 GPT-4 即达 95% 性能，成本节省 85%。核心思路是大多数请求不需要最强模型。

**NVIDIA Orchestrator-8B**：8B 编排器在成本、延迟、准确率上全面超越前沿大模型做编排。小模型做路由决策，大模型做复杂推理。

**生产数据**：70%+ 组织使用 3 个以上模型。

路由策略按三维权衡：

```text
质量优先路由：
  所有请求 → 最强模型
  适用：高风险决策、安全审计、法律合规
  成本：最高

成本优先路由：
  所有请求 → 最便宜模型，失败时升级
  适用：大批量低风险任务
  质量：可能下降

平衡路由（推荐）：
  复杂度评估 → 高复杂度走强模型，低复杂度走轻模型
  风险评估 → 高风险走强模型，低风险走轻模型
  成本约束 → 预算不足时降级
```

### 2.3 策略层自适应（Strategy-Level）

同一范式内的参数根据运行时信号动态调整。

**与 paradigm-routing.md 的区别**：

| 维度 | paradigm-routing | 策略层自适应 |
|---|---|---|
| 调整对象 | 范式本身（ReAct → Plan-Execute） | 范式内参数（step_limit, temperature） |
| 调整幅度 | 离散切换 | 连续调整 |
| 触发频率 | 低（通常每个任务 0-2 次） | 高（可能每步都评估） |
| 状态迁移 | 需要迁移协议 | 无需状态迁移 |

**可调参数**：

| 参数 | 调整方向 | 触发信号 |
|---|---|---|
| step_limit | 增加/减少 | 进展速度 vs 预算剩余 |
| temperature | 增加/减少 | 方案多样性不足 vs 方案过于发散 |
| retry_budget | 增加/减少 | 错误类型（暂时性 vs 系统性） |
| verification_depth | 增加/减少 | 任务风险级别 + 已有置信度 |
| model_tier | 升级/降级 | 子任务复杂度 + 预算约束 |

**触发信号**：

```text
连续失败次数 → 升级 model_tier 或增加 step_limit
token 消耗率 → 降级 model_tier 或减少 verification_depth
用户反馈 → 调整 temperature 和 retry_budget
置信度变化 → 低置信度时增加 verification_depth
进度检测 → 进展停滞时触发参数重评估
```

**MAGE 元强化学习**：WebShop 100% 成功率（vs 基线 79.7%），通过多 Episode 训练 + 反思内循环实现策略自适应。关键：Agent 在每个 episode 后反思策略有效性并调整。

## 3. 自改进闭环

策略层自适应的进一步延伸：Agent 不只是调参，还能改进策略本身。

### 什么时候值得投入自改进

不是所有 Agent 都需要自改进。判断标准：

| 条件 | 值得投入 | 不值得 |
|---|---|---|
| 任务是否可验证 | 有客观信号（测试通过、API 成功、数值正确） | 主观判断（创意好坏、对话恰当性） |
| 任务是否重复 | 同类任务反复出现 | 每次都是全新场景 |
| 失败是否有模式 | 同类失败反复发生 | 随机性失败 |
| 策略库是否有维护能力 | 有 eval 回归和版本管理 | 没有验证手段 |

### 两种已验证的自改进模式

**模式 A：轨迹蒸馏**（EvolveR 等）。从历史成功轨迹中提炼策略原则，存入策略库；新任务时检索相关策略指导决策，新轨迹再回流到策略库。闭环条件：必须有客观验证信号筛选"什么是成功轨迹"。

**模式 B：技能固化**（SAGE 等）。从成功执行中提取可复用的工具链或 prompt 片段，持久化为"技能"。实测效果：目标完成率提升约 9pp，token 消耗降低约 60%。前提：技能的 precondition 和 scope 必须标注清楚，否则会在不适用场景中引入错误。

两种模式的共同约束：**自改进的可靠性完全依赖于验证信号的质量。** 没有客观验证信号的场景（创意、策略、对话），自改进容易固化偏差。

### 增长预期

在可验证场景中，自改进系统与静态系统的差距随时间拉大。METR 的测量数据显示任务自主能力约每 4-7 个月翻倍。但这个增速假设了持续的验证反馈。验证信号中断或质量下降时，增长会停滞或偏离。

## 4. 与经济学的关联

### 4.1 成本-质量-延迟三维权衡空间

```text
          质量
          ↑
          │    ★ 理想点（不可达）
          │
          │  ● 强模型（高质量，高成本，高延迟）
          │
          │      ● 路由（平衡点）
          │
          │          ● 弱模型（低质量，低成本，低延迟）
          │
          └──────────────────→ 成本
         ╱
        ╱
       延迟
```

### 4.2 关键成本数据

| 优化手段 | 成本节省 | 实现复杂度 |
|---|---|---|
| 模型路由 | 30-85% | 低-中 |
| Prompt 缓存 | 最高 90%（缓存命中价为标准价 10%） | 低 |
| 策略层自适应 | 10-30%（减少无效步骤） | 高 |

### 4.3 从预算反推

```text
给定：$X/月预算
  → 计算可用 token 总量（按模型价格）
  → 确定模型组合（强模型占比、弱模型占比）
  → 配置路由策略（确保不超预算）
  → 设置 step_limit 和 retry_budget 上限
```

## 5. 实践指南

| 自适应层 | 实现复杂度 | 收益 | 适用 MVA 阶段 |
|---|---|---|---|
| 模型层（Adaptive Thinking） | 零（模型内置） | 中 | MVA-0+ |
| 路由层（RouteLLM 等） | 低-中 | 高（30-85% 成本） | MVA-3+ |
| 策略层（元学习/规则） | 高 | 高（质量+效率） | MVA-5+ |

**推荐实施顺序**：

```text
Step 1（MVA-0）：使用支持 Adaptive Thinking 的模型，不做额外工作
Step 2（MVA-3）：引入两级模型路由（强+弱），按复杂度分流
Step 3（MVA-5）：添加策略层自适应，基于规则引擎调参
Step 4（MVA-7+）：引入自改进闭环，持续优化策略库
```

## 6. 设计检查清单

- [ ] 是否利用了模型内置的自适应思考？
- [ ] 是否有模型路由（至少两级：高质量模型 + 低成本模型）？
- [ ] 策略参数（step_limit, retry_budget）是否可根据运行时信号调整？
- [ ] 自适应策略是否有预算上限防止失控？
- [ ] 自改进闭环是否有客观验证信号？
