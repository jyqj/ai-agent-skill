# Context, Memory, State, and World State Boundaries

> **Evidence Status** — grounded. 从 Claude Code、Codex、MemPalace、Nocturne 等项目中观察到的共性混淆和分离方案。

## 为什么要把它们分开

很多系统只有一个"大缓存"来存放一切——当前对话、历史记忆、任务进度、外部对象状态全揉在一起。这看起来简单，但会造成三个后果：恢复困难（不知道哪些该恢复、哪些该刷新）、记忆污染（临时状态被写入长期存储）、错误停止（任务进度显示完成但外部世界没有达标）。

## 四者的区别

| 对象 | 生命周期 | 核心作用 | 主要风险 |
|---|---|---|---|
| **上下文（Context）** | 当前窗口 | 模型当前能看到什么 | 膨胀后注意力衰减（context rot） |
| **记忆（Memory）** | 跨会话 | 积累的事实、偏好、技能 | 过期信息污染后续决策 |
| **任务状态（Task State）** | 当前任务 | 做到哪一步、决策了什么 | 中断后进度丢失 |
| **世界状态（World State）** | 外部对象的当前快照 | 写动作前后的参考基准 | 过期或冲突导致无效操作 |

## 常见混淆及后果

**用 Memory 代替 World State**：Agent 记住了"数据库里有 3 条记录"，但实际上已经变成了 5 条。写动作基于过期记忆执行，结果不可预测。Memory 记录的是曾经的观察，World State 需要的是当前的事实。

**用 Context 代替 Task State**：任务进度只存在于对话历史中。一旦上下文被压缩或中断，Agent 不知道自己做到哪里、哪些方案已经排除、哪些风险未解决。

**用 Task State 代替 Effect Verification**：任务状态标记为"已执行"，但没有回读确认外部世界确实发生了变化。任务进度是主观记录，效果验证需要客观回读。

## 设计启发

分离它们不需要复杂架构——关键是承认它们是不同的东西，有不同的生命周期和刷新策略：

- Context 在窗口内管理，需要压缩和卸载策略
- Memory 跨会话持久化，需要验证和过期机制
- Task State 随任务生死，需要 checkpoint 和恢复策略
- World State 随外部世界变化，需要在写动作前刷新
