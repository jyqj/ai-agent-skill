# Pattern Migration

> **Evidence Status** — synthesized. 面向已有运行系统的渐进式 pattern 引入指南；避免全量改写。

## 30 秒判断

已有 Agent 引入新 pattern 的原则：**一次只加一个，先做最小可用版本，保留回滚开关。**

## 10 个高影响 Pattern 的迁移卡片

### 1. effect-ledger — 效果账本

| 维度 | 说明 |
|---|---|
| 前置条件 | 已有工具调用层；能获取工具返回值 |
| 最小实现 | 在工具调用后追加一次 readback/断言，记录 expected vs actual |
| 集成点 | 工具执行后、状态更新前 |
| 成本影响 | +1 次 API 调用/readback；延迟 +200-500ms |
| 回滚方案 | 移除 readback 步骤，恢复原流程 |

### 2. tool-registry — 工具注册

| 维度 | 说明 |
|---|---|
| 前置条件 | 至少 1 个外部工具；有 schema 描述 |
| 最小实现 | JSON schema 列表 + 按名称分发；不需要动态发现 |
| 集成点 | Prompt 组装阶段，注入可用工具列表 |
| 成本影响 | +token（schema 占上下文）；可按需裁剪 |
| 回滚方案 | 回退到硬编码工具描述 |

### 3. layered-memory — 分层记忆

| 维度 | 说明 |
|---|---|
| 前置条件 | 任务跨多轮；上下文接近窗口上限 |
| 最小实现 | 短期（当前对话）+ 长期（KV 摘要存储）两层；写入时打来源标签 |
| 集成点 | 每轮开始时注入摘要；每轮结束时写回 |
| 成本影响 | +摘要生成 token；+存储读写延迟 |
| 回滚方案 | 停止写入长期层，仅用当前对话上下文 |

### 4. checkpoint-hydration — 检查点恢复

| 维度 | 说明 |
|---|---|
| 前置条件 | 任务可序列化为 JSON/YAML 状态；有持久化存储 |
| 最小实现 | 每完成一个子任务写一次 state snapshot；恢复时加载最近 snapshot |
| 集成点 | 子任务边界处（milestone 之间） |
| 成本影响 | +序列化/写入开销；通常可忽略 |
| 回滚方案 | 停止写 checkpoint，退化为无状态执行 |

### 5. compaction — 上下文压缩

| 维度 | 说明 |
|---|---|
| 前置条件 | 多轮对话；上下文超过窗口 50% |
| 最小实现 | 对历史消息做摘要替换，保留最近 N 轮原文 |
| 集成点 | Prompt 组装前，检查 token 用量 |
| 成本影响 | +1 次摘要调用；节省后续所有轮次 token |
| 回滚方案 | 跳过摘要步骤，截断历史消息 |

### 6. loop-detection — 循环检测

| 维度 | 说明 |
|---|---|
| 前置条件 | Agent 有迭代执行逻辑 |
| 最小实现 | 记录最近 K 步 action hash；连续 N 次相同则中断 |
| 集成点 | 每步执行后、下一步规划前 |
| 成本影响 | 几乎为零（hash 比较） |
| 回滚方案 | 移除 hash 检查逻辑 |

### 7. guard-model — 守卫模型

| 维度 | 说明 |
|---|---|
| 前置条件 | 有高风险动作（写入/删除/付款） |
| 最小实现 | 高风险动作前用小模型做一次"该不该执行"判断 |
| 集成点 | 工具调用前的拦截层 |
| 成本影响 | +1 次小模型调用；延迟 +100-300ms |
| 回滚方案 | 旁路 guard 调用，直接执行 |

### 8. self-verification — 自我验证

| 维度 | 说明 |
|---|---|
| 前置条件 | 输出有可检查的正确性标准 |
| 最小实现 | 生成输出后追加一轮"检查自己的输出是否满足要求" |
| 集成点 | 最终输出前 |
| 成本影响 | +1 次 LLM 调用 |
| 回滚方案 | 跳过验证轮，直接输出 |

### 9. depth-budgeting — 深度预算

| 维度 | 说明 |
|---|---|
| 前置条件 | 任务可能触发深度递归或长链执行 |
| 最小实现 | 设定 max_steps / max_tokens 上限；每步递减计数器 |
| 集成点 | 循环/递归入口处 |
| 成本影响 | 无额外成本；限制成本上限 |
| 回滚方案 | 放宽或移除上限 |

### 10. hook-system — 钩子系统

| 维度 | 说明 |
|---|---|
| 前置条件 | 需要在执行流程中插入可扩展逻辑 |
| 最小实现 | 在关键节点（pre-tool / post-tool / pre-output）暴露回调接口 |
| 集成点 | 执行管线的固定锚点 |
| 成本影响 | 取决于 hook 实现；框架本身近零 |
| 回滚方案 | 注销所有 hook，管线直通 |

## 推荐迁移顺序

```text
低风险先行：loop-detection → depth-budgeting → compaction
核心能力：  tool-registry → effect-ledger → self-verification
高级能力：  layered-memory → checkpoint-hydration → guard-model → hook-system
```

先加成本最低、回滚最简单的 pattern；每加一个都验证后再加下一个。

## 反模式

| 反模式 | 后果 | 修正 |
|---|---|---|
| 一次引入所有 pattern | 集成故障无法定位 | 每次只加一个，验证后再加 |
| 无回滚方案就上线 | 出问题只能全量回滚 | 每个 pattern 必须有 feature flag |
| 跳过前置条件 | pattern 无法正常工作 | 对照前置条件逐项检查 |
